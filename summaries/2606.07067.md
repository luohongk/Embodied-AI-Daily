## 0. 概览（Concise Summary）

- **主题（Topic）**：自动驾驶（AD）中的功能卸载（Function Offloading）安全性与责任敏感安全（RSS）模型。
- **问题（Problem）**：当自动驾驶功能（如轨迹规划）被卸载到边缘服务器时，无线通信（V2X）会导致端到端响应时间增加且不稳定，这直接影响了车辆的反应时间，从而破坏了基于固定响应时间的传统RSS安全模型所提供的安全保障。
- **方法（Method）**：提出了一种扩展的RSS模型，该模型显式区分了本地执行和卸载执行的不同响应时间。基于此，设计了一个RSS感知的功能卸载协议，该协议引入了一个新的“警戒（guarded）”状态，用于在安全条件恶化时提前准备回退（fallback）机制，并采用“热备份（warm-standby）”策略以加速回退过程。
- **创新（Innovation）**：
    1.  将RSS模型从二态（安全/危险）扩展为三态（安全/警戒/危险），以处理卸载场景下的可变响应时间。
    2.  将扩展后的RSS模型集成到功能卸载决策中，只有当当前交通状况在卸载响应时间下仍保持安全时，才允许卸载。
    3.  提出了一种增强的回退策略，利用“警戒”状态提前激活本地服务的“热备份”，实现从卸载到本地执行的更快、更安全的切换。
- **意义（Significance）**：该工作首次将形式化的RSS安全模型与自动驾驶功能卸载决策相结合，在享受卸载带来的计算效率提升的同时，确保了安全性不妥协。它为解决分布式自动驾驶系统中的安全-效率权衡问题提供了一个实用且可验证的框架。
- **一句话总结**：本文通过扩展责任敏感安全（RSS）模型，引入“警戒”状态和热备份回退机制，实现了对卸载式自动驾驶服务的安全评估与决策，在保证安全的前提下最大化功能卸载的效率。

## 1. 研究动机（Motivation）

- **解决的问题**：自动驾驶车辆面临计算和能源资源有限的挑战。功能卸载（将计算任务卸载到边缘/云端）可以缓解这一问题，提升计算效率和降低能耗。然而，将安全关键（safety-critical）的自动驾驶功能（如轨迹规划）卸载后，通过无线V2X通信传输数据会引入显著且不稳定的端到端响应时间（response time）。
- **为什么重要**：传统的责任敏感安全（RSS）模型是自动驾驶领域公认的安全形式化模型，它依赖于一个固定的响应时间参数来计算安全距离。如果实际响应时间（由于卸载而变长）超过了RSS模型假设的响应时间，那么RSS计算出的安全距离就会不足，导致车辆在危险情况下无法及时反应，从而引发事故。
- **挑战在哪里**：
    1.  **响应时间动态变化**：卸载后的响应时间不仅包括服务执行时间，还包括不可预测的V2X通信延迟，这使得无法使用一个固定的响应时间进行安全评估。
    2.  **安全与效率的权衡**：完全禁止卸载可以保证安全，但会牺牲计算效率。需要一个智能的决策机制，在安全条件允许时允许卸载，在安全条件恶化时及时回退。
    3.  **回退的时效性**：当需要从卸载切换回本地执行时，切换过程本身也需要时间（`ρ_switch`）。如果切换太慢，车辆可能在切换完成前就陷入危险。因此，需要一种更快的回退策略。

## 2. 创新点（Innovation）

1.  **三态RSS模型（Three-State RSS Model）**：核心创新。将传统RSS的“安全（safe）”和“危险（dangerous）”二态分类扩展为“安全”、“警戒（guarded）”和“危险”三态。
    - **安全**：在本地响应时间（`ρ_local`）下，场景是安全的。
    - **警戒**：在卸载响应时间（`ρ_off`）下，场景是危险的，但在本地响应时间加上切换回退时间（`ρ_local + ρ_switch`）下，场景是安全的。这个新状态是卸载场景独有的，为提前准备回退提供了时间窗口。
    - **危险**：即使在最保守的估计（`ρ_local + ρ_switch`）下，场景也是危险的。

2.  **RSS感知的功能卸载协议（RSS-Aware Offloading Protocol）**：基于三态RSS模型，设计了一个决策协议（Algorithm 1）。该协议将RSS状态作为卸载决策的核心依据：
    - 只有在“安全”状态下才允许卸载。
    - 在“警戒”状态下，允许继续卸载，但必须立即开始准备回退（如激活本地服务的“热备份”）。
    - 在“危险”状态下，立即停止卸载，切换到本地执行并触发紧急响应。

3.  **热备份回退策略（Warm-Standby Fallback Strategy）**：为了加速从“警戒”状态到“危险”状态时的回退，提出了“热备份”机制。在“警戒”状态下，本地服务（如轨迹规划器）被提前激活，但以较低的更新频率运行（即“热备份”状态）。这样，当需要回退时，只需将更新频率调高，而无需从零开始启动服务，从而显著缩短了切换时间（`ρ_switch`）。

## 3. 主要内容（Main Content）【重点】

### 3.1 设计架构与方法

论文的核心架构是一个集成了RSS扩展的**服务导向功能卸载框架（SOFOF）**。主要流程如下：

1.  **状态评估**：车辆持续使用本地响应时间（`ρ_local`）评估RSS状态，确保基本安全。
2.  **卸载决策**：当考虑卸载时，使用卸载响应时间（`ρ_off`）评估RSS状态。
    - 如果`ρ_off`下为“安全”，则允许卸载。
    - 如果`ρ_off`下为“危险”，则进一步使用`ρ_local + ρ_switch`评估。
        - 如果此评估为“安全”，则进入“警戒”状态，允许卸载但需准备回退。
        - 如果此评估也为“危险”，则禁止卸载，保持本地执行。
3.  **回退机制**：在“警戒”状态下，系统启动“热备份”回退准备。一旦状态恶化到“危险”，立即激活本地服务，执行紧急响应。

### 3.2 关键算法与数学推导

**1. 传统RSS安全距离公式（来自[5]）**：
纵向安全距离（同向行驶，本车在后）：
$$d_{\text{lon}}^{\min} = \left[ v_e \rho + \frac{a_{\text{lon}}^{\max,a} \rho^2}{2} + \frac{(v_e + \rho a_{\text{lon}}^{\max,a})^2}{2a_{\min,b}} - \frac{v_o^2}{2a_{\text{lon}}^{\max,b}} \right]_+$$
其中，$v_e$和$v_o$是本车和前方车辆的速度，$\rho$是响应时间，$a_{\text{lon}}^{\max,a}$是最大加速度，$a_{\min,b}$是最小制动减速度，$a_{\text{lon}}^{\max,b}$是最大制动减速度，$[d]_+ = \max\{0, d\}$。

**2. 核心算法：RSS感知的功能卸载协议（Algorithm 1）**
该算法是论文方法的核心，其伪代码逻辑如下：

```
Algorithm 1 RSS-Aware Function Offloading Protocol
Input: ρ_local, ρ_off, ρ_switch, states_ego, states[other]
Output: Bool for offloading possibility

1: if offloading inactive then
2:   for s_o in [s_other] do
3:     state_RSS,local = RSS(ρ_local, s_ego, s_o)
4:     if state_RSS,local is dangerous then
5:       Trigger proper response
6:       return false
7: else // offloading is active
8:   for s_o in [s_other] do
9:     state_RSS, off = RSS(ρ_off, s_ego, s_o)
10:    if state_RSS, off is dangerous then
11:      state_RSS, switch = RSS(ρ_local + ρ_switch, s_ego, s_o)
12:      if state_RSS, switch is dangerous then
13:        Switch back to local execution
14:        Trigger proper response
15:        return false
16:      else
17:        Prepare fallback
18:        return true
```

**算法解读**：
- **输入**：三个关键响应时间（本地`ρ_local`、卸载`ρ_off`、切换`ρ_switch`），本车状态，其他车辆状态列表。
- **输出**：布尔值，表示是否允许卸载。
- **逻辑**：
    - **卸载未激活时**（第1-6行）：用`ρ_local`检查所有车辆。如果任何车辆导致危险，则触发紧急响应并禁止卸载。
    - **卸载激活时**（第7-18行）：用`ρ_off`检查所有车辆。
        - 如果所有车辆都安全（第10行条件不成立），则返回`true`，继续卸载。
        - 如果发现危险（第10行条件成立），则用更保守的`ρ_local + ρ_switch`再次检查（第11行）。
            - 如果仍然危险（第12行条件成立），则立即回退到本地执行，触发紧急响应，并返回`false`。
            - 如果变得安全（第12行条件不成立），则进入“警戒”状态，准备回退（第17行），但允许继续卸载（返回`true`）。

### 3.3 模型、训练与数据集

- **模型**：本文不涉及机器学习模型的训练。其核心模型是**扩展后的RSS数学模型**，以及基于此模型设计的**确定性决策协议**。该模型是规则驱动的，而非数据驱动的。
- **训练**：无。
- **数据集**：无传统意义上的训练/测试数据集。评估使用的是：
    - **仿真环境**：基于Lanelets地图和智能驾驶员模型（IDM）的软件在环（SIL）仿真。
    - **真实世界**：在配备5G通信的实验车辆上进行的实车测试。

### 3.4 实验设置与结果

- **实验设置**：
    - **仿真**：在AMD Ryzen Threadripper 3990X CPU上运行SIL仿真。地图以Lanelets表示，其他车辆由IDM模型控制。ROS 2节点模拟AD栈和MEC服务器。
    - **实车**：一辆实验用CAV，配备与仿真相同的AD栈，通过专用5G网络与MEC服务器通信。
    - **响应时间测量**：通过离线测量“从生成虚拟障碍物到车辆做出反应”的时间差，并取0.99分位数作为保守估计。测得`ρ_local = 0.319s`，`ρ_off = 0.450s`。实车测试中，为更容易触发临界场景，人为增加了0.3s的偏移量。
- **实验结果**：
    1.  **回退时间改进**：通过“热备份”机制，从“警戒”状态回退的平均时间（`ρ_g_switch = 53.69ms`）比从“危险”状态直接回退（`ρ_n_switch = 95.76ms`）快了近一半（图4相关数据）。
    2.  **安全性评估**：将提出的RSS方法与SOFOF（简单位置决策）和MUFASA（先前工作）进行了比较。结果如图5所示。
        - **关键指标**：图中每个柱状图的上端是总卸载时间，下端是RSS安全状态下的卸载时间。两者之间的连线长度代表了“在RSS危险状态下进行卸载”的时间。
        - **结论**：RSS方法的连线长度最短，表明其在卸载期间发生危险情况的时间最少，安全性最高。尽管其总卸载时间（效率）略低于MUFASA，但安全性得到了显著提升，体现了安全-效率的权衡。
    3.  **实车验证**：在8次实车测试中（图6、7），RSS方法成功识别了多种状态转换。
        - **“警戒”状态的价值**：在多次场景中，状态从“安全”变为“警戒”后又回到“安全”（图7(b)中~6s, 33s, 38s处），避免了不必要的回退，保持了卸载效率。总计7次避免了回退。
        - **成功回退**：6次成功从“警戒”状态回退到本地执行，避免了潜在危险。
        - **直接危险**：仅有4次直接从“安全”跳转到“危险”，表明模型对突发情况的响应是有效的。

### 3.5 技术细节

- **实现**：所有模块（包括RSS节点、SOFOF节点、轨迹规划器等）均作为ROS 2生命周期节点实现。RSS节点订阅车辆状态和跟踪列表（tracks）等ROS 2话题。车辆到车道的映射通过Frenet变换实现。
- **响应时间选择**：采用0.99分位数作为保守估计，以应对尾延迟（tail latency）问题。作者论证了在10Hz运行频率下，这个选择是合理的。
- **热备份定义**：对于轨迹规划服务，“热备份”定义为节点已激活但以较低更新频率运行。对于其他服务（如跟踪），可以采用类似思路。

## 5. 澄清与简化（Clarifications and Simplifications）

- **RSS类比**：可以把RSS想象成考驾照时的“安全跟车距离”规则。传统RSS假设你的反应时间是固定的（比如1秒）。但本文的情况是，你的一部分驾驶任务（比如规划路线）不是由你自己（本地电脑）完成，而是打电话（V2X）让一个远程助手（边缘服务器）帮你做。这个电话有延迟，所以你的“反应时间”变长了（比如2秒）。如果还用1秒的跟车距离，就太近了，容易追尾。
- **“警戒”状态类比**：本文的创新就是引入了“警戒”状态。想象你正在打电话让助手帮你规划路线。你发现前面的车离你越来越近，虽然还没到危险距离，但已经比用2秒反应时间算出的安全距离近了。这时，你进入“警戒”状态。你不会立刻挂电话，但你会开始做两手准备：比如，把脚从油门上挪开，放到刹车踏板上方（这就是“热备份”）。如果前车继续靠近，你就能立刻踩刹车（切换到本地执行），反应极快。如果前车加速走了，你就能继续安心打电话（保持卸载）。
- **“热备份”类比**：就像电脑上的“睡眠”模式。它不是完全关机（冷备份），也不是满负荷运行（热备份）。它处于一种低功耗的待机状态，可以非常快速地唤醒。在本文中，本地轨迹规划器在“警戒”状态下就处于这种“睡眠”模式，一旦需要，可以瞬间“唤醒”接管控制权。
