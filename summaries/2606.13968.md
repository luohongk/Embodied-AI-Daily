## 0. 概览（Concise Summary）

- **主题（Topic）**：大型语言模型（LLM）推理的中间件系统，旨在统一本地、高性能计算（HPC）和云端三种推理环境。
- **问题（Problem）**：研究人员在使用LLM时面临碎片化选择：本地模型免费但受限于硬件；HPC中心资源强大但交互性差、有防火墙限制；商业云API成本高且存在数据隐私风险。现有系统无法统一这三者，导致工作流割裂和资源浪费。
- **方法（Method）**：提出了一个名为STREAM（Smart Tiered Routing Engine for AI Models）的多层中间件。它包含：(1) 一个基于复杂度的三层路由架构；(2) 一个创新的“双通道HPC流式传输”架构，分离控制平面（Globus Compute）和数据平面（WebSocket中继），以穿透防火墙实现亚秒级首Token延迟；(3) 层级感知的上下文摘要，防止长对话导致简单查询被路由到昂贵层级；(4) 一个HPC-as-API代理模式，将HPC推理暴露为OpenAI兼容的API。
- **创新（Innovation）**：核心创新在于“双通道HPC流式传输”设计，它巧妙地利用Globus Compute进行任务调度和认证（控制平面），同时通过一个独立的WebSocket中继（数据平面）实时流式传输Token。由于客户端和HPC节点都只向外连接中继，从而无需修改防火墙规则或VPN即可实现实时流式传输。此外，将复杂度路由、层级感知摘要和HPC-as-API模式整合成一个统一系统也是重要贡献。
- **意义（Significance）**：STREAM首次实现了本地、HPC和云端LLM推理的无缝统一，将HPC推理的首Token延迟（TTFT）从批处理模式的11.40秒降低到流式模式的0.54秒（提升21.1倍），使得HPC资源对交互式应用变得实用。它通过将简单查询保留在免费层级，减少了付费云API的使用，同时为复杂查询保留了访问前沿模型的能力，有望显著降低研究成本并提高HPC资源的利用率。
- **一句话总结**：STREAM通过一个创新的双通道流式架构和智能路由中间件，成功地将本地、HPC和云端LLM推理整合为一个统一、高效且防火墙友好的交互式服务。

## 1. 研究动机（Motivation）

- **解决的关键问题**：研究人员和从业者在选择LLM推理环境时面临一个“三难困境”：
    1.  **本地推理**：免费、私密，但受限于个人硬件，无法运行大模型或处理长上下文。
    2.  **机构HPC中心**：提供强大的GPU资源且无边际成本，数据安全，但通常位于防火墙后，设计用于批处理作业，缺乏交互式流式传输能力，导致用户体验差（等待整个响应生成完毕）。
    3.  **商业云API**：提供顶尖模型，按需使用，但成本高昂，且数据保留政策不适合敏感研究数据。
- **为什么重要**：这种碎片化现状迫使研究人员维护多个独立的工作流，导致不必要的云服务开销，同时机构内强大的HPC GPU资源却未被充分利用。一个能够统一这三者、智能路由请求并解决HPC交互性问题的系统，对于降低研究成本、提高资源利用率和加速研究进程至关重要。
- **挑战在哪里**：核心挑战在于如何让HPC集群支持**实时流式传输**。HPC调度器（如SLURM）和作业执行模型是批处理式的，没有内置机制将部分输出实时推送给远程客户端。此外，HPC集群的防火墙策略阻止所有入站连接，使得传统的服务器推送流式传输方案（如WebSocket服务器监听）无法直接部署。STREAM需要在不修改防火墙规则或依赖VPN的情况下，实现从HPC到用户端的亚秒级首Token延迟。

## 2. 创新点（Innovation）

论文提出了四项关键创新：

1.  **三层路由架构**：首次将本地设备、机构HPC和商业云API整合成一个单一、自动化的服务。通过一个本地的LLM复杂度判断器（Complexity Judge）对用户查询进行分类（LOW, MEDIUM, HIGH），并智能路由到最合适的层级（本地→HPC→云），在保证响应质量的同时最大化免费资源的使用。
2.  **双通道HPC流式传输（核心创新）**：这是解决HPC防火墙和流式传输问题的关键技术。它将一个HPC推理请求的处理路径分离为两个独立的通道：
    - **控制平面**：使用Globus Compute负责用户认证、任务调度和作业提交。它不传输实际的Token数据。
    - **数据平面**：使用一个公网可访问的WebSocket中继（Relay）负责实时传输Token。HPC节点和客户端都作为客户端**向外连接**到这个中继，而不是等待入站连接。
    - **创新之处**：这种设计巧妙地规避了HPC防火墙禁止入站连接的限制，因为所有连接都是出站的。同时，它实现了控制与数据的解耦，使得流式传输不受批处理任务调度的影响，从而将TTFT从11.40秒降低到0.54秒。
3.  **层级感知的上下文摘要**：解决了长对话导致简单查询被“强制升级”到昂贵层级的问题。当对话历史接近当前层级（如本地层32K）的上下文窗口限制时，系统会使用免费的本地模型对早期历史进行压缩摘要，保留最近的几轮对话。这确保了即使是“2+2等于几？”这样的简单问题，在长对话后仍能被路由到免费的本地层级，而不是因为上下文过长而被升级到HPC或云端。
4.  **HPC-as-API代理模式**：将HPC推理能力封装成一个标准的OpenAI兼容API端点。外部应用（如云服务、桌面工具）无需了解Globus Compute、WebSocket协议或HPC细节，只需一个Bearer Token和Base URL即可调用HPC推理。这个模式之所以可行，完全得益于创新点2实现的亚秒级TTFT，否则批处理模式下的11秒延迟将使该模式对交互式应用毫无价值。

## 3. 主要内容（Main Content）【重点】

### 3.1 设计架构与方法

STREAM作为一个中间件层，位于用户界面和三个推理后端之间。其核心架构如图1所示。

- **工作流程**：
    1.  用户查询进入系统。
    2.  **复杂度判断器（Complexity Judge）**：一个本地运行的Llama 3.2 3B模型，将查询分类为LOW、MEDIUM或HIGH。
    3.  **层级路由器（Tier Router）**：根据复杂度判断结果，将查询路由到对应的层级（LOW→本地，MEDIUM→HPC，HIGH→云）。用户可手动覆盖。
    4.  **流式处理器（Streaming Handler）**：负责后续处理。如果对话历史接近目标层级的上下文窗口限制，则调用**上下文摘要器（Context Summarizer）** 进行压缩。
    5.  处理后的查询通过统一的**LiteLLM网关**转发到选定的后端。
    6.  后端生成的Token通过**服务器发送事件（SSE）** 实时流式返回给用户。
    7.  系统记录Token使用量和成本到数据库（不存储消息内容），并在失败时自动回退到下一层级。

- **三层推理后端**：
    - **本地层**：运行Ollama模型（如Llama 3.2 3B），32K上下文，免费。
    - **HPC层**：运行vLLM服务（如Qwen 2.5-VL-72B-AWQ），64K上下文，免费。vLLM运行在Apptainer容器中，通过Globus Compute的CPU Worker与集群交互。
    - **云层**：通过OpenRouter访问300+模型，64K-1M上下文，按量付费。

- **部署模式**：
    - **服务器模式**：5个Docker容器，适合共享部署。
    - **桌面模式**：单进程PyWebView应用，无外部依赖。
    - **HPC-as-API模式**：独立部署，将HPC层暴露为公共API端点。

### 3.2 关键算法与数学推导

论文的核心算法在于**双通道HPC流式传输**的协议设计，而非复杂的数学公式。其关键逻辑如下：

1.  **通道创建**：HPC代理为每个查询生成一个唯一的UUID作为通道ID（$channel\_id$）。
2.  **任务提交**：代理通过Globus Compute（控制平面）提交一个任务，任务参数中包含 $channel\_id$ 和vLLM/中继的URL。**关键安全设计**：中继共享密钥（$relay\_secret$）和AES-256-GCM加密密钥（$encryption\_key$）被预加载到HPC节点的环境变量中，不作为任务参数传递，避免泄露。
3.  **双向外连**：
    - HPC代理（消费者）立即向WebSocket中继发起一个携带 $channel\_id$ 和 $relay\_secret$ 的出站连接。
    - 当Globus Compute任务在HPC节点上开始执行时，它（生产者）也向同一个WebSocket中继发起一个携带相同 $channel\_id$ 和 $relay\_secret$ 的出站连接。
4.  **数据流**：生产者从vLLM读取SSE流，将每个Token（可选地使用AES-256-GCM加密）通过WebSocket发送到中继。中继根据 $channel\_id$ 将数据转发给已连接的消费者。中继会缓冲最多1000个Token，以防消费者连接稍慢。
5.  **通道销毁**：查询完成后，双方断开连接，通道立即被移除。若一方未连接，通道在300秒超时后回收。

**AES-256-GCM加密**：
- 加密过程：生产者生成一个12字节随机数（nonce），使用预共享的256位密钥对Token载荷进行加密，生成密文和16字节认证标签。
- 传输格式：将 `nonce || ciphertext || tag` 进行Base64编码后放入JSON信封发送。
- 解密过程：消费者使用相同的密钥和nonce进行解密，并验证认证标签，确保数据在传输过程中未被篡改。

### 3.3 模型、训练与数据集

- **模型**：
    - **复杂度判断器**：`Llama 3.2 3B`（本地运行，无需额外训练）。
    - **本地推理**：`Llama 3.2 3B` / `Gemma 3 4B-VL`。
    - **HPC推理**：`Qwen 2.5-VL-72B-AWQ`（AWQ量化版），运行在H100 NVL GPU上。
    - **云推理**：通过OpenRouter访问，实验中使用了`Claude Sonnet 4.6`。
- **训练**：论文未涉及模型训练。复杂度判断器是直接使用的预训练模型，未进行微调。
- **数据集**：
    - **路由评估**：一个包含1200个查询的平衡测试集（每个复杂度等级400个），来自StackExchange、MMLU、MMLU-Pro、PubMedQA等公开数据集，涵盖10个领域。**真实标签**由`Claude Sonnet 4.6`标注。
    - **延迟基准**：50个单轮请求，绕过复杂度判断器，直接测试各层级性能。
    - **上下文摘要评估**：5个合成的40轮对话（每轮约1050 tokens），用于测试摘要效果。

### 3.4 实验设置与结果

- **实验设置**：
    - **本地**：Apple M4 Pro (24 GB统一内存)。
    - **HPC**：专用单用户Globus Compute端点（无队列竞争），H100 NVL GPU (94 GB)，vLLM 0.15.1（从源码编译以匹配CUDA 12.4驱动）。
    - **云**：Claude Sonnet 4.6 via OpenRouter。
    - **指标**：首Token延迟（TTFT，从请求提交到第一个SSE事件的时间）、吞吐量（tok/s）、路由准确率、付费层级泄漏率。

- **主要结果**：

    **表1：路由混淆矩阵（Llama 3.2 3B作为判断器）**
    - **总体准确率**：49.0%（随机基线为33.3%）。
    - **免费层级保留率**：85.1%（即只有14.9%的查询被错误地路由到付费的云层）。
    - **主要弱点**：LOW查询的召回率极低（8.2%），意味着几乎所有简单查询都被过度路由到MEDIUM（HPC）层级。这是一个保守的错误，因为HPC也是免费的，但导致简单查询使用了比必要更大的模型。
    - **判断器延迟**：中位数164 ms，p95为230 ms。

    **表2：响应延迟（中位数，50次运行）**
    | 层级 | TTFT (s) | 吞吐量 (tok/s) |
    | :--- | :--- | :--- |
    | 本地 (Llama 3.2 3B) | 0.26 ± 0.07 | 85.5 |
    | HPC (中继流式) | **0.54 ± 0.34** | 26.9 |
    | HPC (批处理回退) | 11.40 ± 3.27 | 26.9 |
    | 云 (Claude Sonnet 4.6) | 1.68 ± 0.52 | 41.8 |
    - **核心发现**：HPC中继流式传输将TTFT从批处理模式的11.40秒降低到0.54秒，**提升了21.1倍**。两种模式吞吐量相同，证明中继没有增加逐Token开销。

    **表3：上下文摘要结果**
    - 在没有摘要的情况下，对话在30轮后（约31.6K tokens）超过本地层32K上下文限制，导致简单问题被升级到HPC层。
    - 启用摘要后，系统在上下文达到80%限制时进行压缩（约90%的压缩率），使得简单问题在40轮后仍能留在本地层处理。

### 3.5 技术细节

- **防火墙穿透**：双通道架构的核心。HPC节点和客户端都只发起**出站**WebSocket连接（wss://，端口443），这是防火墙普遍允许的。中继服务器是唯一需要公网IP和入站端口的组件。
- **序列化问题**：Globus Compute使用`dill`序列化函数。在桌面模式（PyInstaller打包）下，`dill`无法解析内部导入系统。解决方案是将函数作为源代码字符串传递，并使用`exec()`执行。
- **远程包依赖**：STREAM未安装在HPC端点上，因此其AES-256-GCM加密辅助函数无法在远程函数中导入。解决方案是将该辅助函数的代码直接复制到远程函数体内。
- **安全设计**：将中继视为不可信中介。三层防护：TLS（传输安全）、访问控制（中继共享密钥在WebSocket握手后通过JSON消息传递，避免出现在URL日志中）、端到端AES-256-GCM加密（中继操作员无法读取Token载荷）。

## 4. 意义与影响（Significance and Impact）

- **解决的问题与贡献**：
    - **解决了HPC LLM推理的交互性问题**：通过双通道流式传输，将HPC推理的TTFT从不可用的11秒降低到可交互的0.5秒，这是该领域的一个重大突破。
    - **实现了真正的多层级统一**：首次将本地、HPC和云三种环境整合到一个智能路由系统中，为研究人员提供了一个“一站式”的LLM推理解决方案。
    - **降低了研究成本**：通过智能路由，85.1%的查询被保留在免费层级，显著减少了对昂贵云API的依赖。
    - **提高了HPC资源利用率**：通过HPC-as-API模式，降低了HPC的使用门槛，使得更多研究人员能够利用机构内的强大计算资源。

- **对未来研究/应用的影响**：
    - **HPC中间件设计范式**：双通道架构为其他需要从HPC进行实时数据传输的应用（如科学模拟、实时数据分析）提供了新的设计思路。
    - **LLM服务化**：HPC-as-API模式使得HPC可以像云服务一样被集成到各种应用中，有望推动更多基于HPC的AI服务出现。
    - **智能路由研究**：论文指出当前基于LLM的判断器准确率（49%）仍有很大提升空间，这为未来研究更高效、更准确的查询复杂度分类器（如微调的小型分类器）指明了方向。

- **局限与开放问题**：
    - **路由准确率**：49%的准确率是主要局限，特别是对简单查询的低召回率。作者明确表示，用微调的分类器替换LLM判断器是最重要的下一步。
    - **评估局限性**：路由评估的标签来自单一LLM（Claude Sonnet 4.6），可能存在偏差；延迟测试在专用单用户端点上进行，未考虑多用户并发场景；摘要评估基于合成对话。
    - **未来工作**：包括替换复杂度判断器、收集真实用户路由轨迹、与Globus Streams等替代方案进行直接比较、以及实现多用户SLURM作业归属。

## 5. 澄清与简化（Clarifications and Simplifications）

- **双通道流式传输的类比**：想象一下，你想让一个在戒备森严的实验室（HPC集群）里的机器人（vLLM）为你实时画画（生成Token）。实验室规定，任何人不能从外面打电话进去（禁止入站连接）。
    - **传统方式（批处理）**：你写一封信（提交作业），让门卫（SLURM）送进去。机器人画完一整幅画后，门卫再把画带出来给你。整个过程需要很长时间（11秒）。
    - **STREAM的双通道方式**：
        - **控制平面**：你通过一个专门的快递员（Globus Compute）给门卫送了一张纸条，上面
