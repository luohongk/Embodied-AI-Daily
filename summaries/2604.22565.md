## 0. 概览（Concise Summary）

- **主题（Topic）**：大型语言模型（LLM）的长上下文推理、证据选择、提示优化。
- **问题（Problem）**：LLM在处理长且充满噪声的上下文时，容易“迷失在中间”，忽略关键证据，导致推理性能下降。现有方法如压缩或重写输入会丢失或扭曲信息，而手动提示工程无法适应每个实例。
- **方法（Method）**：提出**HiLight**框架，将证据选择与推理解耦。训练一个轻量级的“强调执行器”（Emphasis Actor），通过强化学习（RL）在原始上下文中插入最小化的高亮标记，以引导一个冻结的“求解器”（Solver）LLM进行推理。
- **创新（Innovation）**：
    1.  **解耦架构**：首次将证据选择与推理过程明确解耦，通过输入端的非破坏性标记进行干预。
    2.  **弱监督学习**：将高亮问题建模为弱监督的决策问题，仅使用下游任务奖励（无需证据标签）优化执行器，且不访问求解器的梯度或内部状态。
    3.  **零样本迁移**：训练好的执行器可以零样本迁移到不同系列和规模的求解器（包括API模型），表明其学到了通用的证据结构。
- **意义（Significance）**：提供了一种轻量级、可解释、非破坏性的方法来提升LLM在长上下文场景下的性能，尤其适用于证据稀疏、干扰项多的任务。它兼容黑盒API模型，并揭示了证据选择与推理能力在一定程度上是可分离的。
- **一句话总结**：HiLight通过一个轻量级的强化学习执行器，在不解构原始文本的前提下，为冻结的LLM动态插入高亮标记，显著提升了其在长噪声上下文中的证据利用和推理能力。

## 1. 研究动机（Motivation）

- **核心问题**：大型语言模型（LLM）在长上下文推理中，当关键证据被大量无关、冗余或过时信息淹没时，会表现不佳。这种现象被称为“迷失在中间”（Lost in the Middle）效应。
- **为什么重要**：LLM越来越多地被部署在需要处理长文本的场景中，如多文档问答、智能体工作流、个性化推荐等。如果不能有效利用上下文，提供更多信息反而会损害性能，成为应用瓶颈。
- **挑战在哪里**：
    1.  **耦合问题**：传统的做法将“证据选择”（从噪声中找出关键信息）和“推理”（基于证据进行多步推断）这两个子问题耦合在LLM的单次前向传播中。LLM的注意力机制在低信噪比环境下无法可靠地隔离稀疏证据。
    2.  **缺乏监督**：在大多数任务中，我们只有输入和最终答案标签，没有“哪些是证据”的细粒度标注，这使得直接训练一个证据选择器变得困难。
    3.  **兼容性**：许多方法需要访问模型内部状态（如梯度、注意力权重），这对于API模型或专有模型不可行。同时，压缩或重写输入的方法会丢失或扭曲信息，是不可逆的。

## 2. 创新点（Innovation）

- **解耦框架（HiLight）**：核心创新在于提出了一个**证据强调（Evidence Emphasis）** 框架，将证据选择与推理明确解耦。引入一个轻量级的“强调执行器”（Actor），其唯一职责是识别并标记证据，而让一个强大的、冻结的“求解器”（Solver）LLM专注于推理。
- **弱监督强化学习**：将证据高亮问题建模为一个弱监督的决策问题。执行器通过强化学习（REINFORCE类算法）进行优化，其奖励信号完全来自求解器在下游任务上的表现（如准确率、F1分数）。这**不需要任何证据标签**，也**不需要访问或修改求解器**（不依赖梯度或中间激活），使其兼容黑盒模型。
- **非破坏性输入干预**：与压缩、重写或剪枝不同，HiLight**保留原始文本**，仅通过插入最小化的、人类可读的结构化标记（如 `<start_important>`）来强调关键片段。这保证了输入保真度，避免了信息丢失。
- **零样本跨模型迁移**：实验证明，在一个求解器上训练的执行器，可以零样本地迁移到其他不同系列（如Qwen、Gemma、Llama）、不同规模（从4B到70B）甚至API模型（GPT-5 mini）上，依然能带来性能提升。这表明执行器学到了任务和输入中的**通用证据结构**，而非过拟合于特定模型。

## 3. 主要内容（Main Content）【重点】

### 3.1 设计架构与方法

HiLight框架由两个核心组件构成（如图1所示）：

1.  **强调执行器（Emphasis Actor）**：一个轻量级的预训练语言模型（如Qwen3-0.6B到4B）。它接收查询 `Q` 和上下文 `X`，输出一个二值掩码 `M`，指示哪些token需要被高亮。
2.  **冻结求解器（Frozen Solver）**：一个强大的、被冻结的LLM（如Qwen3-14B）。它接收被高亮后的上下文 `ˆX` 并执行下游推理任务。

**工作流程**：
- **推理**：给定 `Q` 和 `X`，Actor在预算 `γ` 的限制下选择关键span，插入高亮标记，形成 `ˆX`。然后将 `ˆX` 和 `Q` 一起输入给冻结的Solver，Solver生成最终输出 `y`。
- **训练**：Actor通过强化学习进行训练。对于每个训练实例，Actor采样多个候选掩码，每个掩码生成一个被高亮的上下文，Solver据此生成输出并计算任务奖励。Actor的目标是最大化期望奖励。

### 3.2 关键算法与数学推导

**问题形式化**：
- 输入：`(Q, X, y*)`，其中 `X = (x_1, ..., x_L)` 是长度为 `L` 的token序列。
- 冻结求解器：`M(Q, X) -> y`
- 任务奖励：`R(y, y*)`
- 目标：学习一个策略 `π_θ(M|Q, X)` 来最大化期望奖励：
  $$J(\theta) = \mathbb{E}_{(Q,X,y^*) \sim D, M \sim \pi_\theta(\cdot|Q,X), y \sim M(\cdot|Q, g(X,M))} [R(y, y^*)]$$
  其中 `g(X, M)` 是插入高亮标记的非破坏性操作。

**强调执行器（Actor）**：
- Actor是一个预训练LM，编码 `[Q; X]` 得到隐藏状态 `H = (h_1, ..., h_L)`。
- 每个token的重要性概率通过一个线性层和sigmoid函数计算：
  $$p_i = \sigma \left( \frac{w_p^\top \text{LayerNorm}(h_i)}{\tau} \right)$$
  其中 `w_p` 是可学习的权重向量，`τ` 是温度参数。

**预算约束与高亮构建**：
- **预算**：高亮的token数不能超过总token数的 `γ` 比例：$\sum_{i=1}^L M_i \leq \gamma L$。
- **训练时采样**：先按伯努利分布采样 `\tilde{m}_i \sim \text{Bernoulli}(p_i)`，然后通过投影操作 `Proj_k` 保留概率最高的 `k = ⌊γL⌋` 个token，得到最终掩码 `M`。
- **推理时选择**：直接选择 `p_i` 值最大的 `k` 个token。
- **Span合并与标记注入**：将相邻的被选中的token合并成span，并插入边界标记（如 `<start_important>` 和 `<end_important>`）。

**Actor优化（分组策略梯度）**：
- 由于离散决策和非可微的奖励，使用策略梯度。
- **分组策略梯度**：对每个实例，采样 `G` 个候选掩码 `{M^j}`，得到 `G` 个奖励 `{r_j}`。
- 计算归一化优势：
  $$\hat{A}_j = \frac{r_j - \mu_r}{\sigma_r + \epsilon}$$
- 策略梯度损失：
  $$L_{PG} = -\frac{1}{G} \sum_{j=1}^G \hat{A}_j \log \pi_\theta(\tilde{M}^j | Q, X)$$
- **熵奖励**：鼓励探索，防止策略过早坍缩：$L_{ENT} = -H(\pi_\theta(\cdot | Q, X))$
- **目标长度正则化**：鼓励平均选择概率接近预算率 `γ`：
  $$L_{LEN} = \left( \frac{1}{|\Omega|} \sum_{i \in \Omega} p_i - \gamma \right)^2$$
- **最终损失**：
  $$L(\theta) = L_{PG} + \lambda_{len} L_{LEN} + \beta_{ent} L_{ENT}$$

### 3.3 模型、训练与数据集

- **模型**：
    - **求解器（Solver）**：默认使用 **Qwen3-14B**，并在迁移实验中测试了 Qwen3-4B/8B/32B, Gemma-3-27B, Llama-3-70B 和 GPT-5 mini。
    - **执行器（Actor）**：使用 **Qwen3-0.6B**（SQuAD 2.0, PubMedQA），**Qwen3-1.7B**（Amazon Beauty），**Qwen3-4B**（HotpotQA）。
- **数据集**：
    - **Amazon-Beauty**：序列推荐任务，用户历史长且噪声大，证据稀疏。
    - **HotpotQA**：多跳问答，需要连接多个文档中的证据。
    - **SQuAD 2.0**：阅读理解，包含不可回答的问题。
    - **PubMedQA**：生物医学问答分类（Yes/No/Maybe）。
- **训练细节**：
    - 优化器：Adam，学习率 `1e-4`，权重衰减 `1e-2`。
    - 超参数：`β_ent = 1.0`, `λ_len = 0.01`, 组大小 `G=4`。
    - 预算 `γ`：Amazon-Beauty 为 0.25，其他任务为 0.15。

### 3.4 实验设置与结果

- **主要结果（表2）**：HiLight在所有四个任务、八个指标上均优于所有基线（包括手动指令MI、PRL、BFRS、OPRO、DSPy、APE）。在Amazon-Beauty上提升最大（HR@10提升27.53% vs MI），在PubMedQA上提升最小（1.08%）。
- **消融实验（表3）**：
    - 对比了随机高亮、少样本高亮和**剪枝（Pruned）**（只给Solver高亮的span）。
    - 在Amazon-Beauty上，剪枝效果不错，但HiLight更好，说明保留上下文有助于消歧。
    - 在HotpotQA上，剪枝严重损害性能，而HiLight是唯一提升的变体，说明保留连接性上下文对多步推理至关重要。
- **效率分析（表6, 7）**：HiLight的推理开销极小（<1.01x token开销），训练所需的Solver查询次数远少于搜索式基线方法（如PRL, APE）。Actor的延迟仅为Solver延迟的1.3%-2.9%。
- **敏感性分析**：
    - **预算（图3）**：性能在 `γ ∈ [0.10, 0.30]` 范围内稳定，有一个温和的最优值。
    - **标记格式（表8）**：对不同语法（Markdown, HTML等）鲁棒。
    - **Actor容量（表9）**：更大的Actor在更难的证据选择任务（HotpotQA）上帮助更大。
    - **上下文长度（表10）**：在扩展到32K token的上下文中，HiLight依然保持优势。
- **迁移性（表11, 12）**：在Qwen3-14B上训练的Actor，零样本迁移到其他模型（包括GPT-5 mini），均带来一致提升，且优于让目标模型自我高亮（Self-Mark）的基线。
- **可解释性（图4, 5, 表13）**：
    - 案例研究展示了Actor如何高亮用户对“grips”的偏好，从而将目标商品排名从第14提升到第5。
    - 可视化显示Actor的分数集中在少数与真实证据重叠的区域。
    - 定量分析表明，Actor的高亮与人工标注的证据有高达0.78的F1重叠，尽管从未见过标签。

### 3.5 技术细节

- **Span合并**：将相邻的被选中token合并为span，并允许合并短间隙（gap ≤ δ tokens）以保持语义完整性。
- **提示模板**：Solver使用一个固定的提示模板，鼓励其关注高亮区域，同时允许使用周围上下文（如图2所示）。
- **训练奖励**：为每个任务设计了复合奖励（如HR@10 + NDCG@10）以提供更密集的梯度信号（表1）。
- **失败分析（表14）**：分析了HiLight导致Solver从正确变为错误的案例，发现主要原因是误导性高亮（16.5%）或Solver自身推理失败（33.8%），但成功案例数量远超失败案例。

## 4. 意义与影响（Significance and Impact）

- **解决了什么问题**：解决了LLM在长噪声上下文中“迷失在中间”的核心问题，通过一种轻量级、非侵入式的方法显著提升了证据利用效率。
- **贡献是什么**：
    1.  提出了一个新颖的**解耦框架**，将证据选择与推理分离。
    2.  开发了一种**弱监督强化学习方法**，无需证据标签即可训练证据选择器。
    3.  证明了**零样本跨模型迁移**的可能性，揭示了证据结构的通用性。
    4.  提供了**可解释性**，高亮标记为模型行为提供了人类可读的归因。
- **对未来研究/应用的影响**：
    - **应用**：可广泛应用于需要处理长文本的领域，如个性化推荐、法律文档分析、医疗报告解读、智能客服等。
    - **研究**：启发了“输入侧控制”这一研究方向，即通过修改输入而非模型本身来引导LLM行为。为研究LLM的“证据选择”与“推理”能力是否可分离提供了实证支持。
- **局限与开放问题**：
    - 当前研究聚焦于单轮查询。未来工作可探索**多轮对话**中的缓存感知服务。
    - 与**重写式**的查询自适应方法进行系统比较。
    - 更深入地研究何时**非破坏性强调**优于**删除式选择**。

## 5. 澄清与简化（Clarifications and Simplifications）

- **类比：一个高效的阅读助手**
    - **问题**：想象你（Solver LLM）是一个聪明的侦探，但给你的案件卷宗（长上下文）里99%都是无关的闲聊。你需要从中找出那1%的关键线索（证据）来破案。你很容易被海量信息淹没，找不到重点。
    - **HiLight方案**：HiLight就像给你配了一个**专属的、训练有素的图书管理员（Emphasis Actor）**。这个管理员不负责破案，他的唯一工作就是快速浏览卷宗，然后用**荧光笔**（高亮标记）把最有可能的线索划出来。然后，他把划好重点的卷宗交给你。你只需要专注于分析这些被划出来的重点，当然也可以随时参考上下文，因为原始卷宗没有被撕毁或重写。
    - **训练过程**：这个管理员是怎么学会划重点的呢？他不是靠看标准答案（证据标签），而是通过**试错**。他先试着划一些地方，然后看你（侦探）能不能根据这些线索成功破案。如果你破案了（奖励高），他就知道这次划得好；如果你失败了（奖励低），他就知道下次要调整。通过无数次这样的尝试，他学会了如何划出最有助于你破案的线索。
    - **为什么能迁移**：这个管理员学到的不是某个特定侦探的偏好，而是**案件本身的结构**——比如，在推荐任务中，用户明确表达“喜欢防滑”的评论就是关键线索。这个规律对任何侦探（任何LLM）都适用，所以他能很好地为其他侦探服务。

- **技术概念简化**：
    - **策略梯度（Policy Gradient）**：可以理解为“试错学习”。Actor做出一个动作（选择高亮哪些词），然后根据结果（Solver的得分）的好坏来调整自己下次做类似动作的概率。动作好就增加概率，动作差就降低概率。
    - **非破坏性（Non-destructive）**：意味着不删除、不压缩、不重写原文。就像用荧光笔在书上划线，而不是把书页剪下来。这保留了信息的完整性，让Solver在必要时仍能看到完整的上下文。
    - **零样本迁移（Zero-shot Transfer）**：就像在驾校用教练车学会了开车，然后直接开你朋友的不同品牌的家用车，也能开得很好。因为学会的是“开车”这个通用技能，而不是某个特定车型的操作细节。

## 6. 补充说明（Additional Notes）

- **相关工作关联**：
    - **与硬选择（Hard Selection）的区别**：传统方法如检索（Retrieval）或剪枝（Pruning）是“硬选择”，会丢弃大部分上下文。HiLight是“软强调”，保留所有信息，只添加标记。
    - **与自动提示优化（APO）的区别**：APO（如OPRO, DSPy）优化的是**任务级别**的静态系统提示，对所有输入都一样。HiLight进行的是**实例级别**的动态优化，为每个输入
