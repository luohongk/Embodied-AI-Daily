## 0. 概览（Concise Summary）

- **主题（Topic）**：大型语言模型（LLM）推理服务中的GPU资源管理与成本优化。
- **问题（Problem）**：在服务大量、多样且流量模式动态变化的LLM时，现有的空间共享（Space Sharing）和时间共享（Time Sharing）策略无法有效适应真实生产环境中“突发-分组”（bursty-group）的流量模式，导致资源利用率低、服务等级协议（SLO）达标率差。
- **方法（Method）**：提出一种名为Prism的、以内存为中心的LLM协同服务框架。其核心创新是**GPU内存气球（Memory Ballooning）** 技术，通过一个名为`kvcached`的“气球驱动”程序，在GPU运行时层面统一管理模型权重和KV缓存，实现跨模型的、弹性且透明的内存分配与回收，从而统一了空间共享和时间共享。
- **创新（Innovation）**：
    1.  **概念创新**：将虚拟化中的“内存气球”概念引入LLM服务，将GPU内存视为可弹性伸缩的统一资源，从根本上统一了空间和时间共享。
    2.  **机制创新**：设计了`kvcached`气球驱动，通过解耦虚拟和物理GPU内存、弹性张量（eTensor）抽象、快速模型加载等机制，实现了毫秒级、细粒度（2MB）的跨模型内存重分配，且无需修改现有推理引擎的注意力内核。
    3.  **策略创新**：提出了一个以内存为中心的控制平面，包含负载感知的模型放置算法（最小化KV压力比KVPR）和松弛感知的请求仲裁算法（基于Moore-Hodgson算法），协同优化全局和局部的内存分配。
- **意义（Significance）**：Prism显著提升了多LLM服务的资源利用率和SLO达标率。实验表明，在相同GPU数量下，TTFT SLO达标率提升高达3.3倍，TPOT SLO达标率提升2倍。其核心组件`kvcached`已在超过10,000块GPU的生产环境中部署，证明了其实际价值和影响力。
- **一句话总结**：Prism通过引入GPU内存气球技术，将GPU内存视为弹性资源，统一了空间和时间共享策略，从而高效、低成本地服务动态变化的多LLM工作负载。

## 1. 研究动机（Motivation）

- **核心问题**：LLM推理服务提供商（如Hugging Face、阿里云）需要托管成千上万个基础模型和微调模型，其中许多模型请求量低但必须保持可用。为了满足严格的SLO，业界通常为每个模型实例独占GPU资源，导致GPU利用率极低（平均低于30%），造成巨大的成本浪费。
- **现有方案的不足**：
    - **空间共享**（如MuxServe）：将多个模型同时放在同一GPU上，适合持续有流量的低流量模型。但在面对长时间空闲的模型时，会造成内存碎片化，浪费资源。
    - **时间共享**（如QLM、Aegaeon）：通过将模型换入/换出GPU内存来服务请求，适合处理零星请求。但在流量波动剧烈、多个模型请求交织时，频繁的模型切换（thrashing）会导致严重的延迟SLO违规。
- **关键挑战**：真实生产环境的流量模式呈现出**突发-分组（bursty-group）** 特征，即活跃的模型集合随时间动态变化，且请求到达极其波动。没有任何单一的共享策略能完美适应这种复杂模式。将两种策略简单混合会因资源竞争（如时间共享需要抢占资源，空间共享需要持久资源）而失效。
- **重要性**：解决这个问题对于降低LLM服务成本、提高资源效率、并保证用户体验至关重要，尤其是在token价格不断下降、模型数量激增的背景下。

## 2. 创新点（Innovation）

- **核心创新：GPU内存气球（GPU Memory Ballooning）**。这是本文最根本的创新。作者观察到，无论是空间共享还是时间共享，其核心瓶颈都是GPU内存。时间共享关注模型权重的换入换出，空间共享关注KV缓存容量的分配。Prism借鉴虚拟化技术，将GPU内存视为一个可弹性伸缩的“气球”，通过一个“气球驱动”`kvcached`，可以动态地从空闲模型回收内存（支持时间共享），或从低请求率模型回收KV缓存（支持空间共享），并统一分配给需要资源的模型。
- **与现有方案的不同**：
    - 现有系统（如vLLM、SGLang）在应用层（如PagedAttention）管理内存，内存分配是静态的、模型内孤立的，无法跨模型共享。
    - Prism将内存管理下沉到GPU运行时层，通过`kvcached`实现跨模型的、透明的、细粒度的内存重分配，这是现有系统所不具备的能力。
- **为什么能奏效**：
    - **解耦**：`kvcached`解耦了虚拟地址空间和物理内存页。每个推理引擎拥有一个大的虚拟地址空间，但物理内存按需分配和映射。这使得内存的“膨胀”和“收缩”操作非常轻量（毫秒级），且不影响正在执行的模型。
    - **统一**：通过统一管理模型权重和KV缓存，`kvcached`打破了两种共享策略之间的壁垒。系统可以灵活地在空间共享（模型常驻）和时间共享（模型换入换出）模式间切换，甚至对同一GPU上的不同模型同时应用两种模式。
    - **智能策略**：Prism的控制平面（模型放置和请求调度）以“最小化内存争用”为核心目标，确保弹性内存分配机制能被有效利用，避免颠簸和性能下降。

## 3. 主要内容（Main Content）【重点】

### 3.1 设计架构与方法

Prism的系统架构（图3）包含两个核心部分：

1.  **气球驱动（Balloon Driver）- `kvcached`**：这是Prism的基石，位于推理引擎和GPU物理内存之间。它负责管理所有GPU物理内存，并为每个推理引擎提供一个大的、连续的虚拟地址空间。`kvcached`将应用层的语义（权重、KV缓存）映射到底层的物理内存页，实现透明、弹性的内存分配。
2.  **以内存为中心的控制平面（Memory-Centric Control Plane）**：
    - **全局模型放置（Global Model Placement）**：一个负载感知的算法，旨在将模型分配到GPU上，以最大化每个GPU的内存“余量”（headroom），从而为内存气球提供空间。它使用**KV压力比（KVPR）** 作为启发式指标。
    - **局部请求仲裁（Local Request Arbitration）**：在每个GPU内部，使用一个共享请求队列和松弛感知的调度算法，根据请求的SLO紧迫程度来分配内存，优先服务最可能违反SLO的请求。

### 3.2 关键算法与数学推导

**1. KV压力比（KV Pressure Ratio, KVPR）**

KVPR是衡量单个GPU上内存争用程度的指标，其公式为：

$$KVPR = \frac{w\_token\_rate}{shared\_kv}$$

其中：
- $w\_token\_rate = \frac{token\_rate \times token\_size}{SLO}$，表示模型在单位时间内的SLO加权内存需求速率。$token\_rate$是模型处理输入和解码token的速率，$token\_size$是每个token占用的KV缓存大小，SLO是TPOT SLO。
- $shared\_kv$是GPU上可用于KV缓存的剩余内存容量。

高KVPR意味着该GPU内存压力大，不利于内存气球发挥作用。

**2. 负载感知模型放置算法（Algorithm 1）**

该算法是一个贪心算法，核心思想是将内存需求最高的模型优先放置到KVPR最低的GPU上，以实现“需求互补”（demand-complementary）。

- **输入**：N个GPU，M个模型，每个模型有token率 $t_j$，权重 $w_j$，token大小 $tz_j$，当前所在GPU $g_j$，SLO $s_j$。
- **步骤**：
    1.  将所有模型按 $w\_token\_rate = \frac{t_j \times tz_j}{s_j}$ 降序排列。
    2.  初始化每个GPU的 $shared\_kv$ 为总容量C，$w\_token\_rate$ 为0。
    3.  遍历排序后的模型：
        - 找到使KVPR最小的GPU `best_idx`。
        - 如果当前GPU的KVPR与最佳GPU的KVPR之差大于阈值 $\tau$，则迁移模型到最佳GPU；否则留在原GPU。
        - 更新目标GPU的 $w\_token\_rate$ 和 $shared\_kv$。

**3. 松弛感知请求仲裁算法（Algorithm 2）**

该算法将请求调度问题转化为一个经典的“最小化截止时间错过数”问题，并采用Moore-Hodgson算法求解。

- **输入**：一组请求R，每个请求有提示长度 $p_i$，块预填充速度 $c_i$，TTFT SLO $s_i$，到达时间 $a_i$。
- **步骤**：
    1.  将所有请求按截止时间 $d_i = a_i + s_i$ 升序排列。
    2.  初始化一个空集合S（已接受的请求），当前时间 `current_time`。
    3.  遍历排序后的请求：
        - 将请求r加入S，并更新 `current_time` 加上其执行时间 $e_r = p_r / c_r$。
        - 如果 `current_time` 超过了r的截止时间 $a_r + s_r$，则从S中移除执行时间最长的请求 $r_{max}$（即 $p_{r_{max}} / c_{r_{max}}$ 最大的请求），并回退 `current_time`。
    4.  返回S，S中的请求即为能按时完成的请求子集。

### 3.3 模型、训练与数据集

- **模型**：论文评估使用了58个具有不同架构和大小的代表性LLM，包括Llama、Qwen、DeepSeek等系列模型。
- **训练**：Prism是一个推理服务系统，不涉及模型训练。
- **数据集**：使用了四个真实生产环境的流量追踪（Traces）：
    - **Hyperbolic**：来自Hyperbolic推理提供商，包含24个模型，4个月数据。
    - **Novita**：来自Novita AI推理提供商，包含16个模型，1个月数据。
    - **Arena-Battle**：来自Chatbot Arena，包含129个模型，16个月数据。
    - **Arena-Chat**：来自Chatbot Arena，包含84个模型，11天数据。

### 3.4 实验设置与结果

- **实验设置**：
    - **基线系统**：与SGLang（作为空间共享基线）、QLM（作为时间共享基线）、MuxServe（作为混合共享基线）进行比较。
    - **硬件**：使用多达32块H100 GPU的集群。
    - **指标**：TTFT SLO达标率、TPOT SLO达标率、成本降低、请求处理量。
- **主要结果**：
    - **SLO达标率**：在相同GPU数量下，Prism的TTFT SLO达标率最高可提升**3.3倍**，TPOT SLO达标率最高可提升**2倍**。
    - **成本效率**：在达到相同SLO达标率的情况下，Prism相比最先进方案可降低**超过2倍**的成本，或处理**3.5倍**更多的请求。
    - **生产部署**：`kvcached`已在超过10,000块GPU的生产环境中部署，证明了其稳定性和有效性。

### 3.5 技术细节

- **`kvcached`设计细节**：
    - **统一管理**：将模型权重和KV缓存都视为内存页，实现统一分配和回收。
    - **自动Token块映射**：内部KV缓存管理器将不同模型、不同层数的token块映射到统一的虚拟和物理页上，解决了模型架构差异问题。
    - **优化**：
        - **批量分配**：将同一token的所有层的K和V向量在虚拟空间中连续存储，将2L次页分配减少为1次，实现2L倍加速。
        - **预分配线程**：异步预分配一小批GPU页，减少分配延迟。
        - **2MB大页**：使用2MB内存页，减少碎片化和管理开销。
    - **弹性张量（eTensor）**：通过PyTorch扩展接口实现，对上层推理引擎透明，无需修改注意力内核，兼容CUDA Graph优化。
- **快速模型加载**：
    - **可复用引擎池**：解耦引擎和模型生命周期，模型被换出后引擎不销毁，而是放回池中复用，避免引擎初始化开销。
    - **并行权重加载**：将模型权重分块，通过多GPU并行加载，然后通过NVLink聚合到目标GPU，充分利用CPU-GPU带宽。

## 4. 意义与影响（Significance and Impact）

- **解决的问题与贡献**：
    - 首次系统性地分析了多LLM服务中“突发-分组”的流量模式，并指出现有空间/时间共享策略的局限性。
    - 提出了**GPU内存气球**这一创新概念，为统一空间和时间共享提供了优雅且高效的解决方案。
    - 设计并实现了Prism系统，通过`kvcached`和智能控制平面，在真实生产环境中验证了其显著的成本效益和性能提升。
    - 开源了核心组件`kvcached`，为学术界和工业界提供了强大的工具。
- **对未来研究/应用的影响**：
    - 为LLM推理服务的资源管理开辟了新方向，即从“计算为中心”转向“内存为中心”。
    - 内存气球技术可以启发其他GPU资源管理问题的研究，如模型训练中的显存优化。
    - Prism的设计理念可以应用于更广泛的AI服务场景，如多模态模型、Agent系统等。
- **局限与开放问题**：
    - 论文未明确讨论模型放置算法的收敛性，以及在大规模集群（数百GPU）下的扩展性。
    - 模型迁移的阈值 $\tau$ 如何自适应调整，可能是一个值得研究的问题。
    - 对于需要跨节点Tensor Parallelism的超大模型（如超过单节点显存容量），Prism的模型放置策略可能需要更复杂的考虑。

## 5. 澄清与简化（Clarifications and Simplifications）

- **内存气球（Memory Ballooning）类比**：想象一个房间里有很多个气球（代表不同的LLM模型）。传统方法要么给每个气球一个固定的空间（空间共享），要么只允许一个气球在房间里，用完再换下一个（时间共享）。Prism的做法是，把房间里的空气（GPU内存）看作一个整体。当某个气球（模型）需要更多空气时（处理请求），它可以从其他暂时不需要空气的气球那里“吸”过来。这个“吸”空气的机制就是`kvcached`气球驱动。它让所有气球共享一个空气池，根据需要动态调整大小，非常灵活高效。
- **KV压力比（KVPR）类比**：KVPR就像是一个“拥挤度”指标。如果一个GPU上有很多模型都在“大口吸气”（高token率），而且它们的SLO要求很严格（需要快速吸气），那么这个GPU的“拥挤度”（KVPR）就很高。Prism的模型放置算法就像一个聪明的管理员，它会尽量把“大口吸气”的模型和“小口吸气”的模型放在一起，避免所有“大胃王”挤在一个房间里，从而保证每个房间都有充足的“空气”。
- **松弛感知调度（Slack-Aware Arbitration）类比**：这就像在银行柜台排队。每个客户（请求）都有一个最后期限（SLO）。传统方法是每个窗口（模型）排各的队，效率低下。Prism的做法是，所有客户在一个大厅里排队，只有一个总服务台（共享GPU队列）。服务台会计算每个客户的时间紧迫程度（松弛度），优先服务那些再不办就来不及的客户。如果某个客户（请求）处理时间太长，导致后面更紧急的客户可能错过期限，服务台会果断放弃这个“慢客户”，先服务更紧急的。这个算法（Moore-Hodgson）保证了尽可能多的客户能在截止时间前被服务。
