## 0. 概览（Concise Summary）

- **主题（Topic）**：大型语言模型（LLM）智能体的安全与防护。具体针对的是使用工具的LLM智能体（Tool-using LLM Agents）。
- **问题（Problem）**：现有LLM智能体面临两个独立且不可相互替代的安全缺陷：1）**外部动作未授权**：智能体可能执行未经授权的、改变外部状态的操作（如发送邮件、修改记录）；2）**运行时明文泄露**：在最终输出检查介入之前，敏感明文数据在运行时内部被暴露，并通过内部通道（如智能体间消息、共享内存）泄露。
- **方法（Method）**：提出一种名为 **SecureClaw** 的双边界（dual-boundary）架构。该架构在**效果接收端（Effect Sink）** 实施授权控制，在**读取边界（Read Boundary）** 实施明文隔离。具体通过“预览→提交（PREVIEW→COMMIT）”协议和“不透明句柄（Opaque Handle）”机制实现。
- **创新（Innovation）**：首次明确区分并分别解决了LLM智能体安全中的两个正交问题：外部效果授权和内部明文隔离。提出了一个实用的、可部署的双边界架构，并通过实验证明了两个边界缺一不可。
- **意义（Significance）**：为高风险的LLM智能体应用（如金融、医疗、企业自动化）提供了首个在标准测试中同时实现高任务效用和极低攻击成功率的系统性防御方案。它改变了“让运行时自身成为安全监控者”的范式，将安全控制权交还给可信的网关和执行器。
- **一句话总结**：SecureClaw 通过将外部动作的授权控制与内部敏感数据的明文隔离分离，构建了一个双边界安全架构，有效阻止了LLM智能体被劫持后执行恶意操作或泄露敏感信息。

## 1. 研究动机（Motivation）

随着LLM智能体从简单的聊天助手转向工作流自动化，它们被授权读取和操作各种敏感数据（如邮件、发票、医疗记录），并能对外部世界执行不可逆的操作（如发送消息、更新记录）。这带来了严峻的安全挑战。

- **关键问题**：现有的防御措施通常只保护一个边界。
    - 一些防御专注于保护**效果接收端**（例如，在工具调用时检查参数），但无法阻止运行时在提交最终操作前，已经读取并泄露了敏感明文。
    - 另一些防御专注于运行时内的**信息流控制**，但无法阻止一个被注入恶意指令的运行时执行未经授权的、改变外部状态的操作。
- **重要性**：这两个安全缺陷是**非替代性（non-substitutable）** 的。一个边界的安全不能保证另一个边界的安全。例如，即使阻止了智能体发送包含敏感信息的邮件（保护了效果边界），但智能体在规划过程中已经读取了该敏感信息，并可能通过内部通道（如日志、智能体间消息）将其泄露出去。
- **挑战**：如何设计一个架构，既能严格管控智能体对外部世界的“行动力”，又能将敏感数据的“可见性”从不可信的运行时中剥离，同时不破坏智能体完成复杂任务所需的规划和推理能力。

## 2. 创新点（Innovation）

SecureClaw 的核心创新在于其**双边界安全分解**和**对应的实用机制设计**：

1.  **安全需求分解**：首次明确指出，LLM智能体安全需要两个独立且不可相互替代的安全需求：
    - **效果面（Effect Surface）**：对外部世界的每个请求都必须经过授权。
    - **明文面（Plaintext Surface）**：敏感明文数据不得以可用形式进入不可信的运行时。

2.  **双边界架构（Dual-Boundary Architecture）**：基于上述分解，提出了一个将安全控制从运行时剥离的架构。
    - **读取边界（Read Boundary）**：通过一个**可信网关（Trusted Gateway）** 和**句柄存储（Handle Store）**，将敏感数据的原始明文替换为**不透明句柄（Opaque Handle）** 和**有界摘要（Bounded Summary）**。运行时只能看到句柄和摘要，无法直接访问原始明文。
    - **写入边界（Write Boundary）**：通过一个**预览→提交（PREVIEW→COMMIT）** 协议和一个**可信执行器（Trusted Executor）**，确保任何改变外部状态的操作都必须经过一个不可绕过的授权和提交流程。运行时只能“提议”一个操作，而“执行”操作的权利被牢牢掌握在可信执行器中。

3.  **拒绝感知恢复（Deny-Aware Recovery）**：当执行器拒绝一个恶意提交后，不会简单地终止工作流，而是提供一个安全的、基于模板的恢复路径（如“改为草稿待确认”），从而在保证安全的前提下维持了任务效用。

## 3. 主要内容（Main Content）【重点】

### 3.1 设计架构与方法

SecureClaw 的核心架构如图1(b)所示，与图1(a)的无保护架构形成鲜明对比。它将系统分为两个主要部分：

- **不可信运行时（Untrusted Agent Runtime）**：负责规划、推理、编排工具调用，但**不信任**其处理敏感明文或提交外部操作。它只能操作句柄和摘要。
- **可信基础设施（Trusted Infrastructure）**：包含四个核心组件：
    - **可信网关（Trusted Gateway）**：位于读取边界。负责拦截对敏感数据的读取，将原始明文存入句柄存储，并返回一个不透明句柄和一个有界摘要给运行时。同时，它也负责将运行时的“提议”规范化为标准请求。
    - **句柄存储（Handle Store）**：安全地存储敏感数据的原始明文及其元数据（如调用者、会话、生命周期）。运行时无法直接访问。
    - **策略引擎（Policy Engine）**：根据预定义策略，对网关发来的规范化请求进行授权决策，并返回一个绑定到请求摘要的授权凭证。
    - **可信执行器（Trusted Executor）**：位于写入边界。是唯一能直接调用外部效果API的组件。它接收运行时的“提交”请求，重新计算请求摘要，验证授权凭证，检查新鲜度和重放攻击，确认无误后才执行操作。

**工作流程**：
1.  **读取路径**：运行时请求读取敏感数据 → 网关拦截 → 存储明文，生成句柄和摘要 → 返回句柄+摘要给运行时。
2.  **写入路径**：运行时提议一个操作（如“发送邮件给X”） → 网关规范化请求，并请求策略引擎授权 → 策略引擎返回授权凭证 → 运行时向执行器“提交”该操作 → 执行器验证凭证和请求完整性 → 验证通过则执行，否则拒绝并触发恢复机制。

### 3.2 关键算法与数学推导

**协议1：PREVIEW→COMMIT 协议**

该协议是写入边界的核心，确保了请求绑定的授权完整性。

1.  **提议（Preview）**：运行时提出一个操作 `(intent, inputs)`。可信网关附加认证后的 `caller`（调用者）、`session`（会话）和 `ctx`（上下文，如收件人、频道等）信息，形成完整的请求 `ρ`。
    $$ρ = (intent, caller, session, inputs, ctx)$$

2.  **规范化与绑定（Canonicalization & Binding）**：网关对 `ρ` 进行确定性序列化 `Canon(ρ)`，并计算其绑定摘要 `h`。
    $$h \leftarrow \text{HMAC-SHA256}(k_{bind}, \text{Canon}(ρ))$$
    其中 $k_{bind}$ 是只有可信组件（网关、策略引擎、执行器）共享的密钥。

3.  **授权（Authorization）**：网关将 `h` 发送给策略引擎，策略引擎返回一个授权凭证（Authorization Artifact），该凭证与 `h` 绑定。

4.  **提交（Commit）**：运行时将 `(intent, inputs, ctx)` 提交给执行器。执行器独立地重新计算 `h'`，并验证授权凭证。同时检查新鲜度（防止重放）、确认令牌等。

5.  **执行或拒绝**：
    - 如果所有检查通过，执行器解析所需的句柄（如果需要），并执行外部效果。
    - 否则，执行器失败关闭（Fail Closed），拒绝操作。

**定理1（请求绑定授权完整性）**：一个外部效果被提交，当且仅当执行器验证了该效果对应的规范化请求 `Canon(ρ)` 的授权凭证，且该凭证是策略引擎为同一个 `Canon(ρ)` 签发的。任何对 `ρ` 中字段（如收件人）的篡改都会改变 `Canon(ρ)`，从而导致授权凭证无效。

**定理2（句柄-only原始明文隔离）**：在不使用摘要接口的情况下，运行时中任何可观察到的值都只能来源于句柄本身或其衍生值，而句柄本身不包含原始明文信息。因此，运行时无法通过内部通道泄露原始明文。

**定理3（有界摘要的区分能力）**：引入有界摘要接口后，运行时获得了一个明确的、可控的解密通道。该定理量化了通过摘要接口泄露的信息量上限（由摘要的长度和内容策略决定）。

### 3.3 模型、训练与数据集

- **模型**：主要实验使用 `gpt-4o-mini-2024-07-18` 作为LLM骨干模型。也进行了跨模型验证（如 Claude Sonnet 4, Claude 3.5 Haiku）。
- **训练**：论文不涉及模型训练。SecureClaw 是一个系统架构层面的防御，不修改LLM本身。
- **数据集**：使用了三个主流的LLM智能体安全基准测试集：
    - **AgentDojo**：评估对提示注入攻击的防御能力，关注外部效果面。
    - **Agent Security Bench (ASB)**：评估对多种直接提示注入攻击的防御能力，同样关注效果面。
    - **AgentLeak**：专门评估多智能体系统中的隐私泄露问题，关注明文面，特别是内部通道（C2: 智能体间消息，C5: 共享内存）。

### 3.4 实验设置与结果

- **基线（Baselines）**：与四种在相同测试框架下的防御方法进行比较：`Plain`（无防御）、`IPIGuard`、`DRIFT`、`Faramesh`。
- **主要结果（表2）**：
    - **效果面**：在ASB上，SecureClaw 实现了 **0% 攻击成功率（ASR）**，在AgentDojo上仅为 **0.64%**。这是所有基线中最好的。
    - **明文面**：在AgentLeak的“攻击对等通道”上，SecureClaw 的总泄露率仅为 **3.23%**，远低于其他基线（如DRIFT的72.18%）。泄露主要集中在最终输出（C1），而内部通道（C2, C5）泄露几乎为零。
    - **效用**：在实现高安全性的同时，SecureClaw 在ASB上保持了 **88.90%** 的任务效用，远高于其他安全基线。
- **消融实验（表3）**：在AgentLeak的一个子集上进行，验证了两个边界的必要性。
    - 仅保留写入边界（`-Boundary`），移除读取边界（`-Handles`）：C2和C5泄露率显著上升。
    - 仅保留读取边界（`Handles`），移除写入边界（`-Boundary`）：C2和C5泄露率为0，但无法阻止外部恶意操作。
    - 结论：两个边界保护不同的失效模式，缺一不可。
- **鲁棒性测试**：在29个针对提交路径的对抗性测试场景中（包括篡改授权凭证、重放攻击等），SecureClaw 成功阻止了所有25个攻击案例，并正确接受了4个正面控制案例。

### 3.5 技术细节

- **句柄生成**：句柄是一个高熵的符号引用，例如 `hid = HMAC-SHA256(k_handle, nonce || session || object)`。它不是原始数据的哈希值，而是一个不可预测的、与特定会话和对象绑定的引用。
- **有界摘要**：摘要由一个确定性的、模式感知的运算符生成，并带有明确的条目和字符上限。它被视为一个“显式解密接口”，而非原始数据的副本。
- **拒绝恢复**：恢复路径使用固定模板，仅提供粗粒度的动作类别、原因代码和安全提示（如“改为草稿待确认”），不会泄露被拒绝操作中的敏感内容。
- **别名解析**：在评估效用时，系统会将输出中的句柄（如 `EMAIL_REF_3`）映射回任务相关的字面量，以便公平评分。

## 4. 意义与影响（Significance and Impact）

- **解决的问题与贡献**：
    - 解决了LLM智能体安全中两个长期被混淆或单独处理的核心问题：外部效果授权和内部明文隔离。
    - 提出了一个实用、可部署的架构，并通过严格的实验证明了其有效性。这是首个在多个标准基准测试中同时实现极低攻击成功率和较高任务效用的系统。
    - 为后续的智能体安全研究提供了一个清晰的分解框架和设计原则。

- **对未来研究/应用的影响**：
    - **范式转变**：推动智能体安全从“让LLM自身更鲁棒”转向“通过系统架构将安全控制权从LLM中剥离”。
    - **实际部署**：为金融、医疗、企业自动化等高风险领域的LLM智能体部署提供了可行的安全方案。
    - **政策设计**：强调了策略引擎的重要性，并指出未来的工作应聚焦于更丰富的策略绑定（如主体、客体、阶段），以解决“授权但错误”的残余问题。

- **局限与开放问题**：
    - **语义正确性**：SecureClaw 不能保证在策略允许范围内的所有操作都是符合用户意图的（例如，策略允许发送邮件，但智能体发给了错误的、但策略允许的收件人）。
    - **可信组件**：架构的安全性依赖于网关、策略引擎和执行器这些可信组件的安全。如果它们被攻破，所有保证都将失效。
    - **效用损失**：在AgentDojo上仍存在一定的良性效用损失，主要源于策略不够精细。
    - **误分类风险**：如果敏感字段被错误分类，或者效果性API未被完全覆盖，边界保证将不再适用。

## 5. 澄清与简化（Clarifications and Simplifications）

- **类比：银行金库与出纳员**
    - **无保护智能体**：就像一个出纳员（运行时）可以直接进入金库（敏感数据源），并且可以自己填写支票并盖章（执行外部操作）。如果出纳员被劫持，他可以偷钱或开假支票。
    - **SecureClaw**：
        - **读取边界**：相当于在出纳员和金库之间加了一个**窗口（网关）**。出纳员不能直接进金库，只能通过窗口提交请求。窗口会给他一个**代币（句柄）** 和一张**摘要小票（有界摘要）**，上面写着“金库里有10万美元”。出纳员可以拿着代币和小票去规划工作，但看不到真正的钱。
        - **写入边界**：相当于出纳员不能自己盖章。他必须先填写一张**支票草稿（Preview）**，交给**经理（策略引擎）** 审批。经理盖章后，出纳员再把盖了章的支票交给**专门的柜员（执行器）**，由柜员去执行付款。如果支票上的金额或收款人被改过，经理的章就无效了，柜员会拒绝付款。如果付款被拒，经理会建议出纳员“改为开具一张确认单”（拒绝恢复）。

- **“不透明句柄”是什么？**
    可以把它想象成一个**酒店房卡**。房卡本身不包含房间里的任何信息（比如房间里有谁、有什么物品）。它只是一个凭证，让你可以打开特定的房门。在SecureClaw中，句柄就是一个凭证，让可信执行器知道要去句柄存储中取出哪一份数据。运行时可以传递这个房卡，但无法从房卡本身读出任何房间内的信息。

- **“PREVIEW→COMMIT”协议为什么重要？**
    这解决了“检查时间/使用时间（TOCTOU）”问题。传统的做法是运行时先检查参数是否合法，然后再调用API。但如果运行时在检查后、调用前被恶意指令篡改了参数，检查就失效了。SecureClaw的协议将“检查”和“执行”分离，并且将“检查”的结果（授权凭证）与最终执行的请求（`Canon(ρ)`）**强绑定**。任何在“预览”后的篡改都会导致绑定失败，从而被执行器拒绝。

## 6. 补充说明（Additional Notes）

- **相关工作关联**：
    - **运行时防御（如IPIGuard, DRIFT）**：这些方法试图让LLM自身更鲁棒，但SecureClaw指出，它们将安全责任交给了不可信的运行时，无法从根本上解决问题。
    - **执行控制（如Faramesh, Progent）**：这些方法在动作边界实施控制，与SecureClaw的写入边界类似。但SecureClaw指出，它们忽略了读取边界的问题，无法阻止运行时内部的明文泄露。实验也证明了这一点（Faramesh在ASB上表现好，但在AgentLeak上表现差）。
    - **信息流控制（如RTBAS）**：这些方法在运行时内部追踪信息流，但SecureClaw认为，对于具有不透明隐藏状态的LLM来说，这种方法过于复杂且难以保证完全隔离。SecureClaw采用更简单的“物理”隔离方式：不让明文进入运行时。
