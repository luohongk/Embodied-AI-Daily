## 0. 概览（Concise Summary）

- **主题（Topic）**：3D 重建与编辑，视觉-语言模型（VLM）代码生成。
- **问题（Problem）**：现有的 3D 表示方法（如 NeRF、点云、网格）虽然能实现高保真渲染，但本质上是低层次的，难以进行程序化控制和精确编辑。论文旨在探索一种新的、可解释、可编辑的 3D 重建范式。
- **方法（Method）**：提出 **3D 代码合成（3D-CoS）** 范式，利用视觉-语言模型（VLM）从单张图像生成可执行的 Blender Python 代码，将 3D 资产构建为程序化代码。系统评估了多种代码合成工作流（如蓝图规划、检索增强生成、少样本示例、组件级智能体）。
- **创新（Innovation）**：首次系统性地将“代码”作为 3D 重建的核心表示，并建立了一套统一的评估协议。通过对比多种 VLM 和提示范式，证明了代码表示在 3D 编辑任务中相比传统点云基线具有显著优势（更好的指令遵循和未编辑区域保持）。
- **意义（Significance）**：开辟了 3D 重建的新方向，将 3D 资产创建从低层次几何恢复提升到高层次程序化生成，极大地增强了 3D 模型的可编辑性、可解释性和可控性。为未来可编辑 3D 内容创作提供了新思路。
- **一句话总结**：本文提出并系统评估了 3D 代码合成（3D-CoS）范式，通过让 VLM 从单张图像生成可执行的 Blender 代码，实现了高保真、强可控、易编辑的 3D 重建，尤其在编辑任务中展现出超越传统方法的巨大优势。

## 1. 研究动机（Motivation）

- **解决的问题**：当前主流的 3D 重建方法（如 NeRF、3D Gaussian Splatting、点云、网格）虽然渲染质量高，但存在一个根本性缺陷：**它们都是低层次的几何表示，难以进行程序化控制和精确编辑**。例如，要修改一个椅子的腿，在点云或网格上操作非常复杂且容易破坏整体结构。
- **为什么重要**：在游戏、影视、工业设计等领域，3D 资产的**可编辑性**和**可控性**至关重要。艺术家和设计师需要能够快速、精确地修改模型的特定部分，而不是重新生成整个模型。代码作为一种高层次、参数化的表示，天然具备这些优势。
- **挑战**：
    1.  **表示鸿沟**：如何让 VLM 理解图像中的 3D 结构，并将其转化为精确、可执行的程序代码（如 Blender Python API）？
    2.  **评估缺失**：缺乏一个统一的、标准化的基准来系统评估“图像→代码→3D”这一新范式的性能。
    3.  **能力边界**：当前 VLM 在代码生成方面的能力边界在哪里？它们能否胜任复杂的 3D 几何建模任务？

## 2. 创新点（Innovation）

- **提出新范式**：提出了 **3D 代码合成 (3D-CoS)** 这一全新的 3D 重建范式，将 3D 资产表示为可执行的 Blender 代码，而非传统的低层次几何数据。这是核心创新。
- **系统评估框架**：建立了一套统一的评估协议和基准，用于系统评估不同 VLM（开源和闭源）在图像条件化代码重建任务上的表现。这填补了该领域系统评估的空白。
- **多种结构化工作流**：设计并比较了四种直接提示范式（单次调用、蓝图规划、检索增强生成、少样本示例）和一个组件级智能体工作流，深入分析了不同策略对重建质量的影响。
- **编辑优势验证**：通过定量和定性实验，首次系统地证明了基于代码的 3D 表示在**局部编辑**任务上，相比基于点云的编辑基线（BlendedPC）具有显著优势，包括更好的指令遵循度和对未编辑区域的保持能力。

## 3. 主要内容（Main Content）【重点】

### 3.1 设计架构与方法

论文的核心架构是 **图像 → 代码 → 3D 资产** 的流程。具体包括：

1.  **问题设定**：给定一张单视角图像 $I$，目标是生成一个可执行的 Blender Python 脚本，其渲染出的 3D 对象与目标对象尽可能匹配。
2.  **四种直接提示范式**：
    - **单次调用 (Single-call)**：直接让 VLM 一次性生成完整的 Blender 脚本。速度最快，但容易遗漏细节。
    - **蓝图规划 (Planning)**：将任务分为两步。第一步，VLM 从图像中推断出一个定量的“蓝图”（Blueprint）$B$，描述各个部件的相对尺寸和位置。第二步，基于蓝图 $B$ 和图像 $I$ 生成代码。这提高了可解释性。
    - **检索增强生成 (RAG)**：在生成代码时，从 Blender API 文档数据库中检索相关的函数和参数，作为外部知识提供给 VLM，以减少 API 幻觉。流程是：先推断蓝图 $B$，然后为每个部件生成 API 查询，检索 Top-k 文档，最后基于 $(I, B, K)$ 生成代码。
    - **少样本示例 (Few-shot)**：在提示中提供几个手动精选的“图像-代码”示例对，引导 VLM 学习正确的几何推理和代码编写模式。
3.  **智能体范式 (Agent Paradigm)**：
    - 将重建任务分解为部件级生成。智能体首先制定一个结构化的对象计划 $P = \{p_i\}_{i=1}^N$，描述每个部件。
    - 然后，为每个部件 $p_i$ 独立生成代码，并执行验证。如果某个部件代码执行失败，智能体可以修改并重试。
    - 最后，将所有部件代码组装成完整的重建脚本。这种方法特别有利于提高局部几何保真度和部件覆盖率。

### 3.2 关键算法与数学推导

论文的核心不在于复杂的数学公式，而在于流程设计和评估指标。关键的数学部分在于**评估指标**和**配准协议**。

**配准协议 (Registration Protocol)**：
在进行指标计算前，需要对齐生成的网格和真实网格，以消除全局尺度、平移和偏航角（yaw）的差异。这是一个优化问题：
1.  **目标**：找到一个相似变换（尺度 $s$，平移 $t$，旋转 $R$），该变换保持上方向（pitch和roll固定），仅优化全局尺度、平移和偏航角。
2.  **方法**：
    - 枚举候选偏航角，并使用鲁棒的对称表面距离目标函数进行评分。
    - 对最佳候选，使用迭代最近点（ICP）算法在偏航角约束下进一步优化。
    - 将最终得到的变换应用于生成的网格，再进行后续的指标计算。

**评估指标 (Metrics)**：
- **3D 指标**：
    - **倒角距离 (CD)**：
        $$CD = \frac{1}{|S_1|}\sum_{x \in S_1} \min_{y \in S_2} ||x-y||_2 + \frac{1}{|S_2|}\sum_{y \in S_2} \min_{x \in S_1} ||x-y||_2$$
        其中 $S_1$ 和 $S_2$ 分别是生成网格和真实网格上的采样点集。值越低越好。
    - **F-score at 5% (F@5%)**：以真实网格尺度的 5% 为阈值，计算精确率和召回率的调和平均数。
    - **空间平衡召回率 (SBR@2%)**：在 2% 阈值下，将真实表面划分为空间单元，计算每个单元内的召回率并取平均，以衡量空间分布的恢复质量，避免只恢复大面积平坦区域。
    - **法向一致性 (NC@5%)**：在 5% 距离阈值内的匹配点对之间，计算法向量点积的绝对值平均值，衡量局部表面方向的一致性。
- **2D 指标**（从输入视角渲染）：
    - **轮廓 IoU (Silhouette IoU)**：前景掩码的交并比。
    - **深度归一化均方根误差 (Depth NRMSE)**：前景重叠区域的深度误差。
    - **法向平均绝对误差 (Normal MAE)**：前景重叠区域的法向量角度误差。

### 3.3 模型、训练与数据集

- **模型**：
    - **开源 VLM**：InternVL3.5 38B, LLaVA-OneVision Qwen2 72B, Qwen2.5 VL 72B Instruct。
    - **闭源 VLM**：Claude Sonnet 4, o3, Gemini 3 Pro。
    - **外部基线**：Unique3D, InstantMesh（传统网格重建），MeshCoder（点云到代码基线）。
- **训练**：论文中的 VLM 模型**没有经过微调**。所有实验均基于预训练模型的零样本或少样本能力。这是该工作的一个重要特点，旨在评估现有 VLM 的固有能力。
- **数据集**：
    - **ModelNet10**：用于 3D 重建评估。从中采样 100 个物体，每个物体从 8 个视角渲染，由人工选择信息量最丰富的视角作为输入。并分为 easy/hard 子集。
    - **ModelNet10-V**：从 ModelNet10 衍生，为每个图像配对一个由 GPT-4o 生成的编辑指令，用于定性评估文本条件重建。
    - **BlendNet-E**：用于 3D 编辑评估。基于 BlendNet 数据集，选择了 55 个代表性物体，并为每个物体手动编写了高质量的编辑指令。

### 3.4 实验设置与结果

**3.4.1 3D 重建评估**

- **设置**：在 ModelNet10 上，对所有 VLM 模型评估四种直接提示范式，并对闭源模型额外评估智能体范式。结果与 Unique3D、InstantMesh 和 MeshCoder 对比。
- **主要结果（表 1）**：
    - **代码重建具有竞争力**：Gemini 3 Pro 在 Few-shot 范式下，在所有指标上均超越了 Unique3D 和 MeshCoder。例如，CD 从 0.0442 (MeshCoder) 降至 0.0249。
    - **闭源 VLM 是前沿**：在所有共享范式中，闭源模型（尤其是 Gemini 3 Pro）的性能显著优于开源模型。例如，在 Few-shot 下，Gemini 3 Pro 的 F@5% 为 0.8532，而最佳开源模型 Qwen2.5-VL 为 0.6702。
    - **结构化工作流有效**：不同的提示范式对不同模型效果不同。Gemini 3 Pro 在 RAG 和 Few-shot 下表现最佳；o3 在 Planning 和 Few-shot 下最佳。这表明结构化信息（蓝图、API、示例）能有效提升性能。
    - **智能体范式提升部件覆盖率**：智能体范式在 SBR@2% 指标上表现突出，表明其能更好地恢复物体的各个部件，但可能因部件间协调问题影响全局指标。
    - **与 InstantMesh 的差距**：最佳代码重建结果（Gemini 3 Pro Agent）在 CD (0.0248 vs 0.0191) 和 SBR@2% (0.6263 vs 0.6443) 上已接近 InstantMesh，但仍存在差距。

**3.4.2 3D 代码编辑评估**

- **设置**：在 BlendNet-E 数据集上，评估基于代码的编辑。与点云编辑基线 BlendedPC 在 lamp/table 子集上对比。
- **主要结果（表 2, 表 3, 图 4）**：
    - **代码编辑显著优于点云基线**：在 lamp/table 子集上，代码编辑的 CLIP 相似度 (0.0578) 是 BlendedPC (0.0142) 的 4.1 倍，CLIP 方向性 (0.2499) 也高出 23.9%。
    - **用户研究验证优势**：用户研究显示，代码编辑在“指令遵循”和“未编辑区域保持”两个维度上均获得远高于 BlendedPC 的评分（4.37 vs 1.90, 4.30 vs 2.45）。
    - **定性结果**：图 4 直观展示了代码编辑能精确地执行“将底座改为单腿”、“添加抽屉”等操作，同时完美保留其他部分，而点云基线则产生了明显的伪影和全局变形。

### 3.5 技术细节

- **RAG 数据库构建**：爬取 Blender 4.4 文档，提取 `bpy`, `bmesh`, `mathutils` 等模块的函数、签名和参数语义，标准化为 JSON 格式。数据库包含 1,683 个页面和 21,102 个函数，使用 Qdrant 向量数据库进行混合检索。
- **失败处理**：对于开源模型，如果代码执行失败，有 5 次重试机会，最终失败则不计入指标。对于闭源模型，使用自动重试/自纠正协议，直到获得可执行脚本。
- **配准细节**：配准过程保持俯仰角（pitch）和翻滚角（roll）固定，仅优化尺度、平移和偏航角（yaw），这符合大多数 3D 物体的直立特性。

## 4. 意义与影响（Significance and Impact）

- **解决的问题与贡献**：
    1.  提出了一个全新的、有前景的 3D 重建范式，将 3D 资产表示为可编辑的代码，从根本上解决了传统表示难以控制的问题。
    2.  提供了一个系统性的评估框架和基准，为未来该领域的研究奠定了基础。
    3.  通过大量实验，揭示了当前 VLM 在程序化 3D 建模方面的能力边界，并指出了提升方向。
    4.  令人信服地证明了代码表示在 3D 编辑任务中的巨大优势，为可编辑 3D 内容创作开辟了新道路。

- **对未来研究/应用的影响**：
    - **研究方向**：将激励更多研究关注“代码即 3D 表示”这一方向，推动 VLM 在 3D 推理和代码生成能力上的提升。特别是，如何结合自验证机制、更精细的部件分解和更鲁棒的代码合成。
    - **应用前景**：在游戏开发、影视特效、工业设计、虚拟现实等领域，设计师可以直接通过自然语言指令编辑 3D 模型的代码，实现快速迭代和精确控制。例如，输入“把桌腿加高 10 厘米”，系统就能自动修改代码中的参数。

- **局限与开放问题**：
    - **性能差距**：代码重建在整体几何精度上仍落后于最先进的专用网格重建模型（如 InstantMesh）。
    - **复杂装配问题**：智能体范式在生成复杂物体时，部件间的尺度、位置和一致性协调仍存在问题，可能导致代码执行失败或几何不连贯。
    - **VLM 能力瓶颈**：当前重建质量的上限受限于 VLM 本身的能力，而非代码表示的潜力。提升 VLM 的 3D 理解和代码生成能力是关键。

## 5. 澄清与简化（Clarifications and Simplifications）

- **“代码作为 3D 表示”是什么意思？**
    - **通俗解释**：想象一下，传统的 3D 模型（如点云、网格）就像一张用无数小点或三角形拼成的照片。你想修改照片里椅子的腿，只能一点一点地去涂抹、重画，非常麻烦。而代码表示就像一份制作椅子的说明书（菜谱），上面写着“用圆柱体做腿，高 0.5 米，半径 0.05 米”。如果你想加高椅子，只需要把说明书上的“0.5”改成“0.6”就行了，然后让电脑（Blender）按照新说明书重新做一把椅子。这就是程序化控制的威力。
- **“蓝图规划”和“智能体”有什么区别？**
    - **蓝图规划**：就像你画了一张设计图，然后让一个高级工程师（VLM）一次性根据这张图造出整个模型。工程师很厉害，但可能忽略一些细节。
    - **智能体**：就像一个项目经理，他先把模型拆分成“椅腿”、“椅面”、“椅背”等独立部件，然后分别派给不同的工人（VLM 的多次调用）去制造。每个工人只专注于自己的部件，做完了再组装起来。这样每个部件都可能做得更精细，但组装时可能出现尺寸不匹配的问题。
- **为什么代码编辑比点云编辑好？**
    - **类比**：点云编辑就像用橡皮泥捏一个雕塑，你想修改鼻子，但很可能把整个脸都捏变形了。代码编辑就像用乐高积木搭一个模型，你想换掉鼻子，只需要把那个特定形状的积木块（对应代码中的一段）替换掉就行，其他部分纹丝不动。这就是“局部性”和“可控性”的优势。
