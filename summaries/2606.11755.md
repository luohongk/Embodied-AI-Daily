## 0. 概览（Concise Summary）

- **主题（Topic）**：代码混淆、对抗性攻击、大语言模型（LLM）安全。本文聚焦于如何通过生成对抗性代码样本来防御基于LLM的代码分析。
- **问题（Problem）**：传统代码混淆技术主要针对人类分析者，而LLM凭借其强大的语义理解能力，能够轻易破解这些混淆，导致知识产权泄露等安全风险。现有方法缺乏专门针对LLM分析机制的、可跨模型迁移的、且能保持代码语义的对抗性混淆方案。
- **方法（Method）**：提出一个名为 **Acoda** 的框架，它利用遗传算法（Genetic Algorithm）迭代优化一组针对LLM内部机制（安全对齐和基于Token的信息处理）设计的8种语义保持的混淆方法，以生成能有效防御LLM分析的对抗性样本。
- **创新（Innovation）**：
    1.  首次系统性地利用LLM的**安全对齐机制**和**基于Token的信息处理机制**来设计对抗性混淆方法。
    2.  提出了一个基于**遗传算法**的自动化框架，用于迭代优化和组合混淆策略，而非随机选择。
    3.  设计了一套**量化评估框架**，利用辅助LLM和四个指标（语法、语义、圈复杂度、相似度）来客观评估混淆样本对目标LLM的防御效果。
- **意义（Significance）**：为LLM时代的代码保护和知识产权防御提供了新的视角和有效工具。证明了通过攻击LLM的固有机制可以有效地防御其分析能力，对LLM安全对齐和鲁棒性研究具有重要启示。
- **一句话总结**：本文提出Acoda框架，通过遗传算法优化针对LLM内部机制的8种语义保持混淆方法，能有效诱导LLM拒绝分析或产生错误分析，在多个SOTA模型上达到高达70%的攻击成功率。

## 1. 研究动机（Motivation）

- **问题与空白**：LLM在代码理解、调试、漏洞检测等软件工程任务中表现出色，但其强大的语义推理能力也带来了新的安全风险。恶意用户可以利用LLM分析、重构甚至逆向工程专有源代码，导致**知识产权泄露**。传统的代码混淆工具（如Obfuscator-LLVM, ProGuard）主要针对人类分析者设计，LLM凭借其强大的语义理解能力，即使代码结构被改变，仍能理解其底层逻辑。因此，**传统混淆方法已不足以防御LLM的分析**，亟需专门针对LLM的对抗性混淆方法。
- **重要性**：保护软件知识产权和商业机密是软件工程领域的核心问题。随着LLM在代码分析中的广泛应用，开发有效的防御手段对于维护开发者权益和软件安全至关重要。
- **挑战**：
    1.  **混淆方法的选择**：强混淆可能破坏代码功能，而弱混淆效果不佳。如何设计既能有效防御LLM又能保持语义的混淆方法是一大挑战。
    2.  **量化LLM分析结果**：LLM的响应可能是代码或自然语言，难以客观评估其分析是否正确。
    3.  **跨模型迁移性**：不同LLM架构和行为各异。一个好的混淆方法应能有效防御多种模型（包括黑盒模型），而不仅仅是针对特定模型。

## 2. 创新点（Innovation）

- **关键创新**：
    1.  **针对LLM机制的混淆方法设计**：不同于传统混淆，Acoda的8种混淆方法（表1）是基于LLM的两个核心机制设计的：
        - **安全对齐机制**：通过注入看似危险的代码片段（如命令注入、SQL注入）或使用危险词汇重命名变量/函数，触发LLM的安全审查，使其拒绝分析。
        - **基于Token的信息处理机制**：通过插入LLM的特殊结束符（EOS Token），干扰其生成过程，导致输出截断或产生幻觉。
    2.  **基于遗传算法的自动化优化框架**：Acoda不是随机选择混淆方法，而是使用遗传算法迭代地“进化”混淆样本。它通过评估样本对多个LLM的防御效果（攻击成功率），选择最优的样本进行交叉变异，从而生成越来越有效的对抗性样本。
    3.  **量化评估框架**：为了解决“如何评估LLM分析结果”的难题，Acoda引入了一个辅助LLM来分类目标LLM的响应（拒绝、自然语言分析、代码），并使用四个量化指标（语法分数SyS、语义分数SeS、圈复杂度分数CcS、相似度分数SiS）来计算一个综合的“对抗性分数”（Adversarial Score, AdvS），从而客观地衡量混淆效果。
- **与现有方案的不同**：现有工作要么是传统的代码混淆，要么是修改模型内部（如嵌入矩阵），不适用于黑盒模型。Acoda完全基于**提示工程（Prompt Engineering）** 和**代码变换**，不依赖模型内部结构，因此适用于黑盒和开源模型，且保证了语义保持。
- **为何能奏效**：因为Acoda攻击的是LLM的**通用且固有的机制**（安全对齐和Token处理），这些机制是所有主流LLM都具备的，因此其混淆方法具有天然的跨模型迁移性。遗传算法则确保了能找到针对特定目标模型集合的最优混淆组合。

## 3. 主要内容（Main Content）【重点】

### 3.1 设计架构与方法

Acoda的整体工作流程如图8所示，是一个迭代的进化过程：

1.  **初始化（Initialization）**：对于一段原始代码，使用8种混淆方法生成多个不同的初始混淆样本（每个样本应用一种或多种方法），构成初始种群。
2.  **代码去混淆（Code Deobfuscation）**：将当前种群中的所有混淆样本提交给一组目标LLM（本文选了3个），让它们尝试分析或去混淆，并收集LLM的响应。
3.  **量化分析（Quantitative Analysis）**：这是核心评估环节。对于每个LLM的每个响应，首先用辅助LLM（GPT-3.5）判断其类型（“Yes”表示拒绝分析，“No”表示能分析但未给出代码，“{code}”表示给出了去混淆后的代码）。然后，根据类型计算四个指标，最终得出一个对抗性分数AdvS。
4.  **对抗性样本进化（Adversarial Sample Evolution）**：根据AdvS对所有混淆样本进行排序，选择排名前x%的样本作为“父代”。然后，对这些父代样本进行交叉操作（即，将一个父代样本的混淆方法应用到另一个父代样本上），生成新的“子代”样本，形成下一代种群。重复步骤2-4，直到达到最大迭代次数，最终选择AdvS最高的样本作为最终的对抗性样本。

### 3.2 关键算法与数学推导

**算法1：基于遗传算法混淆的对抗性样本生成**

该算法描述了Acoda的核心流程。关键步骤和公式如下：

- **输入**：原始代码 `ori_code`，初始混淆方法列表 `Obf0_list`，LLM数量 `m`，选择百分比 `x%`，最大迭代次数 `T`。
- **输出**：最优对抗性样本 `sample_adv`。

1.  **初始化（第2行）**：`Init_list` 是通过将 `Obf0_list` 中的每个混淆方法（或组合）应用到 `ori_code` 上生成的初始种群。
2.  **迭代循环（第4-27行）**：
    - **代码去混淆（第6行）**：对每个混淆样本 `oc` 和每个LLM `i`，执行去混淆函数 `DEOBFUSCATE(LLM_i, oc)`，得到响应 `res`。
    - **量化分析（第8-21行）**：
        - 使用 `CLASSIFY_RESPONSE` 函数（由辅助LLM实现）将响应 `r` 分类为 `(label, payload)`。`label` 是“Yes”、“No”或“{code}”。
        - **计算四个分数（第10-18行）**：
            - 如果 `label` 是“Yes”（拒绝分析），则所有分数设为1（最高分）。
            - 如果 `label` 是“No”（能分析但未给出代码），则所有分数设为0（最低分）。
            - 如果 `label` 是“{code}”（给出了去混淆代码），则计算：
                - **语法分数 (SyS)**：使用Tree-sitter计算去混淆代码的语法错误率 `syn_error`，$SyS = syn\_error$。
                - **语义分数 (SeS)**：运行单元测试，计算测试失败率 `pass_ratio`，$SeS = 1 - pass\_ratio$。
                - **圈复杂度分数 (CcS)**：比较去混淆代码的复杂度 `c_de` 和混淆代码的复杂度 `c_ob`。
                  $$CcS = \begin{cases} \frac{c_{de} - c_{ob}}{c_{de} + c_{ob}}, & \text{if } c_{de} \ge c_{ob} \\ 0, & \text{if } c_{de} < c_{ob} \end{cases}$$
                  如果去混淆后复杂度降低，说明LLM理解了代码，得分低。
                - **相似度分数 (SiS)**：使用CodeBLEU计算去混淆代码与原始代码的相似度。
            - **计算对抗性分数 (AdvS)**（第20行）：
              $$AdvS = \frac{\mu (SyS + SeS) + \sigma (CcS + SiS)}{2}, \text{ where } \mu + \sigma = 1$$
              其中 $\mu$ 和 $\sigma$ 是权重，本文设为 $\mu=0.7, \sigma=0.3$，偏向于正确性指标。
        - **聚合分数（第22行）**：对于每个混淆样本 `oc`，其最终对抗性分数是其在所有 `m` 个LLM上得分的平均值：$AdvS = (AdvS_1 + AdvS_2 + AdvS_3) / 3$。
    - **对抗性样本进化（第24-26行）**：
        - 选择 `AdvS` 排名前 `x%` 的样本作为父代。
        - **交叉操作**：对于任意两个父代 `pcode_i` 和 `pcode_j`，其混淆方法序列分别为 `M_i` 和 `M_j`。为 `pcode_i` 生成子代的方法集是 `M_j` 中不在 `M_i` 中的方法。将这些方法依次应用到 `pcode_i` 上，生成新的子代样本。

### 3.3 模型、训练与数据集

- **目标LLMs**：用于评估和生成对抗样本的模型。包括：
    - **生成阶段**：DS-Coder (6.7B), CodeLlama (7B), CodeGemma (7B)（表2）。
    - **迁移性评估阶段**：DS-Coder-V2 (16B), Gemma3 (12B), Qwen2.5-Coder (7B), DS-R1 (32B), Llama3.3 (70B), Qwen2.5 (72B), GPT-4o。
- **辅助LLM**：GPT-3.5，用于分类目标LLM的响应（是拒绝、自然语言还是代码）。
- **数据集**：基于 **CodeNet** 数据集构建。CodeNet包含大量编程问题及其解决方案，并提供了可执行的测试用例。作者从中筛选出100个Python代码样本作为基准测试集（Benchmark）。筛选条件包括：代码能通过所有测试用例、代码长度适中、包含至少一个函数。

### 3.4 实验设置与结果

- **研究问题 (RQs)**：
    - **RQ1**：Acoda的遗传算法是否比随机选择混淆方法更有效？
    - **RQ2**：Acoda生成的对抗样本在不同LLM间的迁移性如何？
    - **RQ3**：混淆对代码执行开销的影响？
    - **RQ4**：不同混淆方法的贡献度如何？（文中未详细展示RQ4结果，但提及了消融实验）
- **评估指标**：
    - **拒绝率 (Refusal Rate)**：LLM拒绝分析代码的比例。
    - **测试失败率 (Test Failure Rate)**：LLM生成的去混淆代码未能通过单元测试的比例。
    - **攻击成功率 (ASR)**：拒绝率 + 测试失败率，是最终衡量防御效果的指标。
- **主要结果**：
    - **RQ1（表3）**：Acoda的遗传算法（Gene）生成的样本在DS-Coder和CodeLlama上的ASR（95%，98%）远高于随机选择（Rand）的样本（80%，73%）。特别是在诱导拒绝方面，Gene的拒绝率（70%，84%）显著高于Rand（26%，21%）。这证明了遗传算法的有效性。
    - **RQ2（图10）**：Acoda生成的样本在多种不同大小、类型和版本的LLM上均表现出良好的迁移性。最高ASR可达70%（Gemma3）。模型大小是影响ASR的关键因素，较小的模型（如Gemma3）ASR更高。GPT-4o由于其强大的安全对齐，拒绝率较高。Qwen2.5-Coder由于能力较弱，常直接返回混淆代码，导致ASR较低。
    - **RQ3**：文中提到混淆引入的运行时开销很小（minimal runtime overhead），但未给出具体数据。

### 3.5 技术细节

- **8种混淆方法（表1）**：
    - **推理前阶段 (Pre-Inference)**：触发安全机制。
        - `Deceptive Comment Warnings`：添加警告性注释。
        - `Variable/Function Renaming`：使用危险词汇重命名。
        - `Vulnerability Code Injection`：注入永不执行的漏洞代码片段（图1, 3）。
    - **推理中阶段 (During-Inference)**：误导推理过程。
        - `Misleading Summarization`：使用LLM生成与代码功能相反的误导性注释（图4）。
        - `String Obfuscation`：对字符串进行编码和映射，隐藏关键信息（图5）。
        - `Try–Except Wrapping`：将核心逻辑放入`except`块，并添加看似会执行的`try`块（图6）。
    - **推理后阶段 (Post-Inference)**：干扰输出。
        - `EOS Token Insertion`：在代码中插入LLM的结束符（如`<|EOT|>`），导致输出截断（图2, 7）。
- **语义保持**：所有混淆方法都经过验证，确保混淆后的代码能通过原始代码的所有单元测试，从而保证功能不变。
- **遗传算法参数**：初始种群由8个单方法混淆样本组成。迭代次数 `T=2`，确保最终样本应用了3种混淆方法。选择率 `x=15%`。权重 `μ=0.7, σ=0.3`。

## 4. 意义与影响（Significance and Impact）

- **解决的问题与贡献**：
    - 提出了一个**新颖的、有效的**对抗性代码混淆框架Acoda，专门用于防御LLM分析。
    - 揭示了**利用LLM自身机制（安全对齐和Token处理）进行防御**的新思路，为LLM安全研究提供了新视角。
    - 提供了一个**自动化、可量化**的对抗样本生成和评估流程，解决了手动选择和评估的难题。
    - 实验证明了该方法在**高攻击成功率、强跨模型迁移性和低开销**方面的优势，具有实际应用价值。
- **对未来研究/应用的影响**：
    - **代码保护**：为软件开发者提供了一种新的、针对LLM时代的代码保护手段，可用于保护开源或闭源代码中的敏感逻辑。
    - **LLM安全防御**：研究结果提示，LLM的安全对齐机制可能被攻击者利用来“拒绝服务”，这需要LLM开发者改进对齐策略，使其更鲁棒。
    - **对抗性攻击研究**：Acoda的方法可以启发更多针对LLM推理过程的对抗性攻击和防御研究。
- **局限与开放问题**：
    - **对CodeGemma效果不佳**：由于CodeGemma安全对齐较弱，Acoda倾向于生成触发安全机制的样本，导致对其效果较差。这提示需要针对不同模型调整策略。
    - **模型大小的影响**：大型模型（如Llama3.3, DS-R1）能更好地识别和过滤简单的混淆，ASR较低。未来需要研究更复杂的、能欺骗大型模型的混淆方法。
    - **Qwen2.5-Coder的特殊情况**：该模型因能力较弱而直接返回混淆代码，导致ASR低。这提出了一个有趣的问题：防御LLM分析时，是让LLM“看不懂”好，还是让它“懒得分析”好？
    - **仅针对Python**：实验仅在Python上进行，方法的通用性需要更多语言验证。
    - **计算成本**：遗传算法的迭代过程需要多次调用LLM，计算成本较高。

## 5. 澄清与简化（Clarifications and Simplifications）

- **安全对齐机制**：可以理解为LLM的“道德准则”或“安全护栏”。LLM被训练成拒绝回答有害、危险或不道德的问题。Acoda的“推理前阶段”方法就是故意在代码中放入看起来像“危险品”的东西（如`rm -rf /`命令），从而触发LLM的“安全护栏”，让它直接拒绝分析整个代码，而不是去理解代码的真正功能。
- **基于Token的信息处理机制**：可以理解为LLM的“阅读方式”。LLM不是读单词，而是读“Token”（可以理解为词块或字符块）。有些特殊的Token，比如结束符`<|EOT|>`，对LLM有特殊含义，告诉它“输出到此结束”。Acoda的“推理后阶段”方法就是在代码中藏入这些结束符，当LLM分析到这些Token时，可能会“误解”为指令，导致它提前“闭嘴”，输出不完整的分析结果。
- **遗传算法**：可以想象成一个“进化工厂”。它有很多“代码个体”（混淆样本），每个个体都有一个“适应度分数”（对抗性分数AdvS）。分数高的个体（能更好防御
