## 0. 概览（Concise Summary）

- **主题（Topic）**：机器人学中的任务与运动规划（Task and Motion Planning, TAMP），特别是针对双臂/多臂机器人的并行调度问题。
- **问题（Problem）**：现有的TAMP算法通常只能生成串行计划（plan），即一次只移动一个机械臂，无法生成允许机械臂并行运动的调度（schedule），从而限制了多臂机器人的执行效率（最小化完成时间）。
- **方法（Method）**：提出一个名为 **ScheduleStream** 的通用框架。该框架定义了一种新的规划语言，支持**持续动作（durative actions）** 和**采样器（samplers/streams）**。基于此，论文设计了领域无关的算法（Eager Stream Scheduling 和 Lazy Stream Scheduling），通过交替进行调度和采样来求解问题，并利用GPU加速采样过程。
- **创新（Innovation）**：
    1.  **首个通用框架**：ScheduleStream 是第一个支持采样操作的、领域无关的规划与调度框架。
    2.  **混合持续动作**：模型使用混合持续动作，这些动作可以异步开始，其持续时间是参数的函数，更贴近真实机器人操作。
    3.  **惰性流调度**：提出了一种惰性流调度算法，通过先使用占位符进行规划，再回溯调用昂贵的采样器（如逆运动学求解），显著提高了规划效率。
    4.  **GPU加速**：将GPU加速应用于采样器（碰撞检测、运动规划等），特别是处理多臂并行运动时所需的成对碰撞检查，大幅提升了计算速度。
- **意义（Significance）**：这项工作弥合了TAMP与调度之间的鸿沟，使得机器人规划系统能够自动生成高效的并行执行方案。这对于充分发挥双臂/人形机器人的潜力、提高任务执行效率具有重要意义，并为更复杂的多机器人协同操作提供了理论基础和实用工具。
- **一句话总结**：本文提出了ScheduleStream，一个通过定义支持持续动作和采样器的通用规划语言，并设计惰性调度算法与GPU加速技术，从而首次实现了领域无关的、高效的多臂任务与运动规划与调度（TAMPAS）框架。

## 1. 研究动机（Motivation）

- **解决的问题**：双臂/人形机器人具有并行操作多个手臂以高效完成任务的潜力，但控制多臂协同运动在计算上极具挑战性。传统的任务与运动规划（TAMP）算法虽然能处理混合离散-连续动作空间，但通常只能生成**串行计划（plan）**，即一次只规划一个手臂的运动。这无法利用多臂并行性来最小化任务完成时间（makespan）。
- **为什么重要**：在工业自动化和家庭服务场景中，效率是关键。能够并行使用双臂可以显著缩短任务执行时间，提升机器人的生产力和实用性。例如，一个机器人可以同时用两只手分别抓取和放置不同的物体。
- **挑战在哪里**：
    1.  **混合空间增长**：多臂协同使得混合离散-连续动作空间呈指数级增长，规划问题更加复杂。
    2.  **并行调度**：需要规划出动作的**开始时间**和**结束时间**，并确保并行执行的动作之间不发生冲突（如碰撞），这超出了传统TAMP的能力范围。
    3.  **计算复杂性**：检查并行运动是否安全（如双臂碰撞检测）需要计算所有运动对之间的碰撞，计算量是二次方增长的。
    4.  **领域无关性**：现有的一些能处理并行操作的工作通常是针对特定应用（如装配）或对问题有严格限制（如同步运动），缺乏一个通用的、领域无关的解决方案。

## 2. 创新点（Innovation）

1.  **ScheduleStream 语言**：提出了第一个领域无关的、支持**持续动作（durative actions）** 和**采样器（streams）** 的规划与调度语言。该语言基于Python构建，允许灵活地调用外部过程（如GPU加速的碰撞检测器），并支持函数式表达，超越了传统的基于谓词（如PDDL）的规划语言。
2.  **领域无关的调度算法**：
    - **Eager Stream Scheduling**：一种直接但完备的算法，通过迭代地实例化所有可能的采样器并求解调度子问题来找到解。
    - **Lazy Stream Scheduling**：一种更高效的算法，它**先使用占位符（placeholder）** 进行规划，识别出哪些采样器是真正需要的，然后再去调用这些昂贵的采样器。这避免了不必要的计算，是本文的核心算法创新。
3.  **调度子问题的求解方法**：将每个持续动作编译为**开始（start）** 和**结束（end）** 两个瞬时动作，从而将调度问题转化为一个经典的、可以求解动作顺序的**序列规划问题**。然后，从序列规划的解中提取出时间调度表。
4.  **GPU加速的采样器**：针对多臂规划中计算量巨大的碰撞检测（特别是双臂运动之间的碰撞），利用GPU进行批量化处理，显著加速了逆运动学求解、运动规划和碰撞检查等采样操作。

## 3. 主要内容（Main Content）【重点】

### 3.1 设计架构与方法

论文的核心是 **ScheduleStream** 框架，其架构包含三个主要部分：

1.  **语言定义**：定义了问题描述语言，包括：
    - **常量**：离散常量（如 `arm1`, `obj1`）和连续常量（如配置 `q`，位姿 `p`）。
    - **函数与谓词**：用于描述状态。函数（如 `At(arm)` 返回配置）和谓词（如 `Arm(arm)` 为真）。
    - **瞬时动作**：瞬间改变状态的动作（如 `pick`, `place`）。
    - **持续动作**：核心创新。它有开始条件、开始效果、持续条件、结束条件、结束效果和持续时间。例如 `move` 动作，其持续时间由轨迹 `t` 决定。
    - **采样器（Streams）**：用于生成连续参数的生成器。例如，`grasps` 流生成抓取姿态，`ik_qs` 流求解逆运动学，`motions` 流规划运动轨迹。它们将输入常量映射到输出常量。

2.  **调度算法**：提出了两种算法：
    - **Eager Stream Scheduling**：一个迭代过程。每次迭代，它先尝试用当前已知的常量求解一个有限调度问题。如果失败，则实例化所有可能的采样器，生成新的常量，然后再次尝试。这种方法虽然完备，但效率低。
    - **Lazy Stream Scheduling**：更高效的方法。它首先用占位符（如 `@q1`）代替采样器的真实输出，然后进行规划。规划成功后，它会回溯哪些占位符被使用，然后只调用生成这些占位符的采样器。如果采样成功，则得到最终解；如果失败，则记录这个失败的“调度骨架”，并在下一轮迭代中避免使用相同的占位符组合。

3.  **调度子程序（Schedule Subroutine）**：这是算法的核心模块，用于求解给定常量集下的有限调度问题。它将每个持续动作 `a` 编译为两个瞬时动作 `a.start` 和 `a.end`，并使用启发式搜索（如A*）来找到最优的动作序列，然后从序列中提取出时间表。

### 3.2 关键算法与数学推导

**1. 持续动作的编译（用于调度子程序）**

为了将调度问题转化为序列规划问题，每个持续动作 `a` 被编译为两个瞬时动作：

- **开始动作 `a.start`**：
  - **条件**：`a` 的**开始条件** + `a` 当前**未在进行中** (`~Ongoing(a.params)`) + 所有其他正在进行的动作的**持续条件**依然满足 (`OverCondition`)。
  - **效果**：`a` 的**开始效果** + 设置 `a` 为**正在进行中** (`Ongoing(a.params)`) + 设置 `a` 的**剩余时间**为其持续时间 (`Remaining(a.params) <= a.duration`)。

- **结束动作 `a.end`**：
  - **条件**：`a` 的**结束条件** + `a` 当前**正在进行中** (`Ongoing(a.params)`) + 所有其他正在进行的动作的**持续条件**依然满足 (`OverCondition`)。
  - **效果**：`a` 的**结束效果** + 设置 `a` 为**未在进行中** (`~Ongoing(a.params)`) + 更新所有其他正在进行的动作的**剩余时间** (`ElapseEffect`) + 设置 `a` 的剩余时间为0。
  - **成本**：`a` 的**剩余时间** (`Remaining(a.params)`)，这代表执行这个结束动作所“消耗”的时间。

**2. 调度子程序（Schedule Subroutine）算法**

```
Algorithm 2: Finite Planning & Scheduling
1: procedure SCHEDULE(I, G, A)
2:   A' ← {ba(x) | a(x) ∈ INSTANTIATE(I, A), ba ∈ [a.start, a.end]}
3:   π ← SEARCH(I, G, A')          # 用A*等算法搜索最优序列
4:   if π = None then return None
5:   π ← SEARCH(I, G, π)           # 对找到的序列进行局部优化
6:   τ ← [ ], t* ← 0; T ← { }      # 初始化调度表、当前时间、开始时间记录
7:   for ba(x) ∈ π do
8:     if ba = a.start then
9:       T[a(x)] ← t*              # 记录动作的开始时间
10:    else                         # ba = a.end
11:      Δt ← a(x).duration − (t* − T[a(x)]) # 计算还需等待的时间
12:      t* ← t* + max(0, Δt)      # 推进当前时间
13:      τ ← τ + [⟨T[a(x)], a(x), t*⟩] # 将动作加入调度表
14:   return τ
```

**解释**：
- 第2行：实例化所有可能的开始和结束动作。
- 第3行：使用搜索算法（如加权A*）找到最小化总成本的序列 `π`。这里的成本是结束动作的 `Remaining` 之和，即总时间。
- 第5行：用找到的序列中的动作实例，再次进行搜索（权重设为1），以局部优化动作顺序。
- 第6-13行：将序列 `π` 转换为时间调度表 `τ`。遍历序列，遇到开始动作就记录当前时间；遇到结束动作，就计算需要经过多少时间（`Δt`）才能结束该动作，并推进当前时间 `t*`。

### 3.3 模型、训练与数据集

- **模型**：本文提出的不是一个需要训练的机器学习模型，而是一个**规划与调度框架**。该框架由规划语言、算法和采样器组成。采样器（如逆运动学求解器、运动规划器）是**程序化的**，而非学习得到的。
- **训练**：无。本文不涉及任何模型训练。
- **数据集**：无传统意义上的训练/测试数据集。实验是通过在**模拟环境**中随机生成100个问题实例（每个任务）来进行的。这些实例定义了不同的机器人、物体、初始状态和目标状态。

### 3.4 实验设置与结果

- **实验设置**：
    - **任务**：5个模拟任务，每个任务有一个参数 `N` 控制复杂度（如物体数量、机器人数量）。
        1.  `Franka Hold Assigned N`：N个Franka机器人抓取指定物体。
        2.  `Franka Hold Any N`：N个Franka机器人抓取任意物体（需要匹配）。
        3.  `SO100 Hold Any N`：N个SO100机器人在同一平台上抓取任意物体（更拥挤）。
        4.  `Franka Pack N`：2个Franka机器人将N个物体打包到小区域。
        5.  `Franka Stack N`：2个Franka机器人将N个物体堆叠起来。
    - **对比算法**：
        1.  **Sequential**：串行规划，代表传统TAMP。
        2.  **Hierarchical**：先调度后运动规划，无回溯，代表一些多臂方法。
        3.  **Ours**：本文提出的Lazy Stream Scheduling。
        4.  **Ours + GPU**：Ours + GPU加速采样器。
    - **评估指标**：成功率（%）、首次解发现时间（Time）、首次解完成时间（t1*）、60秒内最优解完成时间（t∞*）。

- **实验结果**：
    - **表I（成功率和时间）**：
        - **Sequential** 成功率最高（99%），但**Hierarchical** 成功率很低（64%），因为它无法处理“不可向下细化”的问题。
        - **Ours** 成功率（91%）低于Sequential，但**Ours+GPU** 成功率（99%）与Sequential持平，且**首次解发现时间（1.9秒）远快于其他所有方法**。
        - GPU加速显著提升了成功率和速度。
    - **表II（完成时间）**：
        - **Sequential** 的完成时间最长（平均3.4秒），因为它只能串行执行。
        - **Ours** 和 **Ours+GPU** 的完成时间（平均约2.0秒）约为Sequential的一半，证明了并行调度的优势。
        - **Hierarchical** 虽然平均完成时间略低，但这是因为它只成功解决了较简单的问题。

### 3.5 技术细节

- **GPU加速**：使用cuRobo库的CUDA内核进行加速。
    - **碰撞检测**：将机器人模型表示为膨胀的球体集合。对于移动物体碰撞，将手臂轨迹的球体扫掠体积变换到物体坐标系进行检测。对于移动手臂碰撞，将其视为一个复合双臂机器人的球体-球体自碰撞检测。所有这些都可以在GPU上批量并行计算。
- **惰性流调度（Lazy Stream Scheduling）**：
    - 核心思想是**先规划，后采样**。
    - 算法维护一个“调度骨架”集合 `P`。每个骨架是一个用占位符表示的调度方案。
    - 在每次迭代中，算法会避免使用已经失败的骨架中的占位符组合，从而探索新的可能性。
    - 当找到一个新骨架后，算法会尝试用真实的采样器输出替换占位符。如果所有替换都成功，则得到最终解。

    - 设计了**惰性调度算法**，有效解决了在规划过程中调用昂贵采样器的效率问题。
    - 展示了**GPU加速**在解决多臂规划中计算瓶颈的巨大潜力。
    - 通过模拟和真实实验，证明了该方法在成功率和效率上优于现有方法。

    - **机器人规划**：为开发更高效、更智能的多臂/人形机器人规划系统提供了新范式。

    - **人形机器人**：为人形机器人完成复杂的双手操作任务（如开门、搬运大件物品）提供了可行的规划方案。

    - 论文未明确讨论，但可以推断的局限包括：

## 5. 澄清与简化（Clarifications and Simplifications）

- **Plan vs. Schedule**：
    - **Plan（计划）**：就像一个菜谱，告诉你**按什么顺序**做菜（先切菜，再炒菜）。动作是串行的。
    - **Schedule（调度）**：就像一个项目时间表，告诉你**每个任务在什么时间开始和结束**，并且允许多个任务同时进行（比如一边烧水，一边切菜）。目标是让整个项目（晚餐）尽快完成。

- **Durative Action（持续动作）**：可以想象成一个“定时任务”。比如“移动手臂到A点”这个动作，它需要一段时间才能完成。在这段时间里，手臂不能做别的事，而且必须保证它移动的过程中不撞到任何东西。ScheduleStream 把这个动作建模为：开始（手臂离开起点）、持续（手臂在空中移动）、结束（手臂到达终点）。

- **Stream（采样器）**：可以理解为一个“问题解答器”。规划器需要知道“手臂在哪个位置才能抓住杯子？”，它不会自己算，而是调用一个叫 `ik_qs` 的采样器。这个采样器输入“手臂、杯子、抓取姿态”，然后输出一个可行的“手臂关节角度”。这个计算可能很慢（比如解一个复杂的方程），所以规划器希望尽可能少调用它。

- **Lazy Stream Scheduling（惰性流调度）**：就像一个聪明的项目经理。他先假设所有“问题解答器”都能完美工作，并制定一个初步的项目计划（用“占位符”代替具体答案）。然后，他只去问那些在计划中真正被用到的“问题解答器”要答案。如果某个答案要不到（比如那个关节角度会导致碰撞），他就知道这个计划行不通，然后换一个计划。这比一开始就去问所有可能的问题要高效得多。
