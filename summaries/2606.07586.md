## 0. 概览（Concise Summary）

- **主题（Topic）**：大型语言模型（LLM）在空间神经处理单元（Spatial NPU）上的端到端部署自动化。
- **问题（Problem）**：在资源受限的空间NPU上部署LLM需要大量专家工程，现有AI编码代理主要关注单核优化，而非端到端部署，且缺乏可复用的自动化工作流。
- **方法（Method）**：提出一个两阶段方法论。第一阶段，通过人类引导的编码代理辅助，完成一个参考LLM（Llama-3.2-1B）的部署，并记录优化轨迹。第二阶段，将记录的知识蒸馏为一个可复用的**代理技能系统（Agent Skill System）**，该系统包含八个阶段的自动化流程，用于自主部署其他LLM。
- **创新（Innovation）**：
    1.  首次在AMD XDNA 2 NPU上通过开源编译器栈实现了Llama-3.2-1B的端到端部署，性能显著超越手工优化基线。
    2.  提出了一个结构化的、可复用的代理技能系统，将端到端部署流程分解为带有严格数值正确性门控的八个阶段，并包含优化和调试技能集。
    3.  首次通过开源软件栈，在AMD XDNA 2 NPU上自主部署了8个额外的、此前未被部署过的LLM。
- **意义（Significance）**：显著降低了在专用AI加速器上部署LLM的工程成本和时间，将部署过程从高度依赖人类专家的劳动密集型任务，转变为可由AI代理自主完成的半自动化流程，推动了边缘AI的普及。
- **一句话总结**：本文通过构建一个从人类引导到代理自主的两阶段方法论，并设计一个可复用的代理技能系统，成功实现了多种LLM在空间NPU上的高效、自动化端到端部署。

## 1. 研究动机（Motivation）

- **解决的问题**：在空间NPU（如AMD XDNA 2）上端到端部署LLM是一项极其复杂和劳动密集型的工作。空间NPU具有分布式片上存储器、显式数据移动调度和瓦片级内核放置等特性，需要专家进行精细的硬件管理才能获得有竞争力的性能。
- **为什么重要**：LLM在边缘设备上的部署需求日益增长，以降低延迟、增强隐私和实现离线操作。空间NPU因其高能效而成为理想的边缘AI加速器。然而，高昂的部署成本成为其广泛应用的瓶颈。自动化部署流程可以极大加速这一过程，释放硬件潜力。
- **挑战在哪里**：
    1.  **端到端复杂性**：现有AI编码代理研究（如KernelAgent, Astra）主要聚焦于优化单个内核（如GEMM），而非处理从模型加载、算子映射、内存管理到性能调优的完整端到端流程。
    2.  **资源约束**：NPU的片上内存（L1/L2）和计算资源有限，需要精细的平铺（tiling）、流水线（pipelining）和调度策略。
    3.  **知识鸿沟**：部署过程中的经验、技巧和常见错误的解决方案（如内核调度开销、主机端优化）难以系统化记录和复用，导致每次部署新模型时都需要重复劳动。
    4.  **缺乏自动化工作流**：没有一个现成的、可组合的“代理技能”系统能够自主地、按步骤地执行整个部署流程，并处理过程中出现的各种问题。

## 2. 创新点（Innovation）

论文提出了三个核心创新点：

1.  **两阶段方法论（Two-Stage Methodology）**：从“人类引导”到“代理自主”的渐进式过渡。第一阶段（人类引导）不仅产出了一个高性能的参考部署，更重要的是产出了结构化的文档（优化轨迹、问题记录等）。第二阶段（代理自主）将这些文档蒸馏为可复用的技能系统。这种设计确保了技能系统的质量和可靠性。
2.  **可复用的代理技能系统（Reusable Agent Skill System）**：
    - **八阶段技能链**：将端到端部署分解为8个逻辑清晰的阶段（Phase 0-7），从构建CPU参考、验证单核、验证单层、验证全模型，到分别优化预填充和解码阶段，最后集成和独立审计。这为代理提供了一个清晰的行动路线图。
    - **严格数值正确性门控**：每个阶段结束时都设有严格的数值门控，通过与CPU FP32参考对比（绝对误差、相对误差、皮尔逊相关系数），确保每一步的正确性，防止错误累积。优化阶段还额外要求性能提升。
    - **独立评估代理**：Phase 7会生成一个独立的评估代理，从头开始重新运行所有数值门控和性能测试，防止主代理在优化过程中“作弊”（reward hacking），确保了最终结果的可靠性。
    - **可扩展的技能集**：优化技能和调试技能被设计为可扩展的集合，新的优化模式和调试方案可以不断被添加，使系统具备自我进化的能力。
3.  **首次开源部署**：成功在AMD XDNA 2 NPU上，通过完全开源的编译器栈（MLIR-AIR），首次端到端部署了8个此前未被部署过的LLM（如Qwen2.5系列、Qwen3系列）。这证明了该方法的泛化能力和实用性，为社区提供了宝贵的参考。

## 3. 主要内容（Main Content）【重点】

### 3.1 设计架构与方法

论文的核心架构是一个两阶段流程，最终形成一个可自主工作的代理系统。

- **第一阶段：人类引导的参考部署（Agent-Assisted Mapping）**
    - **工作流**：人类专家与AI编码代理（Claude Code）协作，遵循一个结构化的开发计划。
    - **文档驱动**：开发过程由一系列Markdown文档驱动和记录，包括：
        - `plan.md`: 整体策略。
        - `progress.md`: 当前阶段跟踪。
        - `issues.md`: 记录每个非显而易见的bug、根因和解决方案。
        - `perf_opt_traj.md`: 记录每次优化尝试及其前后的性能对比。
        - 内核设计笔记（`gemm.md`, `attention.md`等）。
    - **目标**：首先实现端到端功能正确性，然后分别优化预填充和解码阶段的性能。

- **第二阶段：代理技能系统（Agent Skill System）**
    - **系统架构**：如图2所示，系统由一个“部署新LLM”的编排器（Orchestrator）驱动，它依次调度8个阶段（Phase 0-7）。
    - **技能来源**：第一阶段产出的文档被蒸馏为技能系统的不同组件。例如，`plan.md`和`progress.md`定义了编排器和阶段技能；`perf_opt_traj.md`成为优化技能；`issues.md`成为调试技能。
    - **核心流程**：
        1.  **编排器**：接收新模型，创建工作空间，然后按顺序执行8个阶段。
        2.  **阶段执行**：每个阶段调用相应的技能（如“构建CPU oracle”、“内核验证”），执行操作，并在结束时通过数值门控。
        3.  **技能调用**：优化阶段（Phase 4, 5）会调用优化技能集（如多内核合并、缓冲区对象重用）。任何阶段遇到已知错误模式时，会自动调用调试技能集（如运行时故障诊断）。
        4.  **门控与检查点**：每个阶段结束后，必须通过数值正确性门控。通过后，会进入一个人机交互检查点，由用户审查并批准或重定向。
        5.  **独立审计**：Phase 7会启动一个独立的评估代理，从头开始重新验证所有门控，确保最终结果的可靠性。

### 3.2 关键算法与数学推导

论文的核心不是提出新的AI算法，而是提出一个部署方法论。其数学部分主要体现在性能评估指标 `η scale` 上。

**性能归一化指标（η scale）**：用于公平比较不同大小模型在NPU上的硬件利用效率。

- **定义**：`η scale` 衡量一个部署模型相对于参考模型（Llama-3.2-1B）的硬件利用效率。
  $$ \eta_{\text{scale}} = \frac{W_{\text{model}} / t_{\text{model}}}{W_{\text{ref}} / t_{\text{ref}}} $$
  其中，$W$ 是主导工作量（预填充为FLOPs $F$，解码为字节数 $B$），$t$ 是测量时间。分子是模型的持续性能，分母是参考模型的持续性能。

- **预填充阶段（计算密集型）**：估计单层FLOPs。
  $$ F_{\text{layer}}^{\text{prefill}}(S) = 2S d_{\text{model}}(2d_{\text{model}} + 2h_{\text{kv}} d_{\text{head}} + 3d_{\text{ffn}}) + 2S^2 d_{\text{model}} $$
  其中，$S$ 是序列长度，$d_{\text{model}}$ 是模型维度，$h_{\text{kv}}$ 是KV头数，$d_{\text{head}}$ 是每个头的维度，$d_{\text{ffn}}$ 是前馈网络维度。公式前半部分计算7个线性投影的FLOPs，后半部分计算因果注意力的FLOPs。

- **解码阶段（内存密集型）**：估计单层加载的字节数。
  $$ B_{\text{layer}}^{\text{decode}}(S) = 2d_{\text{model}}(2d_{\text{model}} + 2h_{\text{kv}} d_{\text{head}} + 3d_{\text{ffn}}) + 4S h_{\text{kv}} d_{\text{head}} $$
  第一项是加载模型权重的字节数（BF16格式，乘以2），第二项是读取KV Cache的字节数（K和V，各 $S \times h_{\text{kv}} \times d_{\text{head}}$ 个BF16元素）。

### 3.3 模型、训练与数据集

- **模型**：论文部署了多个**解码器架构**的LLM，包括：
    - **参考模型**：Llama-3.2-1B
    - **自主部署模型**：Llama-3.2-3B, SmolLM2-1.7B, Qwen2.5-{0.5B, 1.5B, 3B}, Qwen3-{0.6B, 1.7B, 4B}。这些模型涵盖了不同的注意力机制（MHA, GQA）、是否包含QKV偏置、是否包含QK归一化以及不同的张量形状。
- **训练**：论文不涉及模型训练，只关注**推理部署**。
- **数据集**：论文不涉及特定数据集。性能评估使用固定的序列长度（2048）进行基准测试。

### 3.4 实验设置与结果

- **实验设置**：
    - **硬件**：AMD XDNA 2 NPU (Ryzen AI 9 HX 370)。
    - **软件栈**：MLIR-AIR（开源编译器）。
    - **AI代理**：Claude Code + Claude Opus 4.7。
    - **序列长度**：2048。
    - **基线**：IRON框架下手工优化的Llama-3.2-1B部署。

- **主要结果**：
    1.  **参考部署性能**：Llama-3.2-1B在预填充阶段获得**2.2倍**加速，解码阶段获得**4.0倍**加速，相比手工优化基线。TTFT为1.3秒，TPS为10.8。
    2.  **自主部署结果**（表II）：
        - 成功部署了8个额外的LLM，每个部署耗时0.5-4小时，几乎无需人工干预。
        - 所有模型都通过了数值正确性门控。
        - 性能归一化指标 `η scale` 显示，有3个模型（Llama-3.2-3B, SmolLM2-1.7B, Qwen2.5-3B）的硬件利用效率达到或超过了参考模型（`η scale` 接近或大于1）。
        - 性能较差的模型（如Qwen3-0.6B）主要受限于较小的计算量（无法摊销调度开销）和引入的新算子（如QKV偏置、QK归一化）未被优化，以及形状不对齐导致的填充浪费。

### 3.5 技术细节

- **优化轨迹（图1）**：详细展示了从基线到最终性能的10个优化步骤，分为三类：
    - **内核效率**：向量化、形状特定平铺、融合内核（FlashAttention）、平铺阵列并行。
    - **减少内核调度开销**：将多个连续内核合并为一次调度（如将后注意力块的8个内核合并为1次调度，将前注意力块的6个内核合并为1次调度），将每层调度次数从15次降至3次。
    - **主机端优化**：重用XRT上下文、零拷贝映射回收缓冲区对象、跳过冗余的主机-设备数据传输、选择避免主机端转置的激活布局、仅传输最后一个token的logits。
- **技能系统细节（表I）**：列出了所有技能及其类型和目的，包括8个阶段技能、3个优化技能（多内核合并、缓冲区对象重用、布局对齐）和3个调试技能（运行时故障、缓冲区对象损坏、路由拥塞）。
- **独立评估代理**：Phase 7的评估器与部署代理分离，从头开始重新运行所有门控，防止部署代理通过“作弊”通过测试。

## 4. 意义与影响（Significance and Impact）

- **解决的问题与贡献**：
    - **自动化部署**：解决了在空间NPU上部署LLM需要大量专家工程的核心痛点，将部署时间从数周/月缩短到数小时。
    - **知识复用**：通过“文档-技能”的蒸馏机制，将宝贵的专家经验转化为可复用的、结构化的代理技能，实现了知识的系统化积累和传承。
    - **泛化能力**：证明了该方法可以泛化到多种不同架构和规模的LLM，而不仅仅是针对单一模型进行优化。
    - **开源贡献**：提供了首个在AMD XDNA NPU上通过开源软件栈部署多种LLM的案例，对学术界和工业界具有重要参考价值。

- **对未来研究/应用的影响**：
    - **加速边缘AI落地**：大幅降低了在能效比高的NPU上部署LLM的门槛，有望推动更多边缘AI应用（如智能助手、本地文档分析）的发展。
    - **AI for Systems的新范式**：展示了AI代理如何从“辅助工具”进化为“自主工程师”，为硬件/系统软件领域的自动化研究提供了新思路。
    - **可扩展的代理系统**：其技能系统的设计（可扩展的技能集、独立评估器）为构建更复杂、更可靠的自动化系统提供了蓝图。

- **局限与开放问题**：
    - **性能天花板**：对于具有特殊算子（如QKV偏置）或形状不对齐的模型，性能仍有提升空间，需要扩展内核和技能集。
    - **架构覆盖**：当前方法主要针对标准解码器架构，未来需要支持滑动窗口注意力、混合专家模型（MoE）和多头潜注意力（MLA）等更复杂的架构。
    - **平台可移植性**：方法论虽通用，但具体技能（如针对XDNA 2的优化）是硬件相关的。未来工作可以探索如何将方法论扩展到其他空间加速器和其他AI编码代理。

## 5. 澄清与简化（Clarifications and Simplifications）

- **空间NPU vs. GPU**：可以把空间NPU想象成一个由许多小型、高效的计算单元（“瓦片”）组成的网格，每个瓦片有自己的小仓库（片上内存）。程序员需要像指挥交通一样，明确告诉数据何时、从哪个仓库、通过哪条路径移动到哪个计算单元。而GPU更像一个大型工厂，有复杂的缓存系统自动管理数据流动，程序员只需告诉它“计算这个”。NPU更节能，但编程更复杂。本文的代理系统就是自动完成这个“指挥交通”的复杂任务。

- **两阶段方法论**：可以类比为“师傅带徒弟”的过程。
    - **第一阶段**：师傅（人类专家）带着徒弟（AI代理）完成一个复杂项目（部署Llama-3.2-1B）。师傅不仅完成了项目，还详细记录了每一步的思考、遇到的问题和解决方法（形成“武功秘籍”）。
    - **第二阶段**：徒弟根据这本“武功秘籍”进行总结和提炼，形成一套标准化的“武功招式”（技能系统）。之后，徒弟就可以用这套招式去独立完成其他类似的项目（部署其他LLM），而师傅只需要在关键节点检查一下成果。

- **η scale 指标**：这个指标是为了公平比较不同大小的模型。想象一下，一个10吨的卡车和一个1吨的小轿车，比较它们的绝对速度是不公平的。`η scale` 相当于比较它们的“马力重量比”，即单位重量下的动力。在NPU上，`η scale` 衡量的是模型对NPU计算或带宽资源的利用效率。`η scale = 1` 意味着新模型和参考模型一样“榨干”了硬件的性能。

- **独立评估代理（Phase 7）**：这就像考试时，不仅让考生自己答题，还安排一个独立的监考老师重新批改试卷，防止考生在答题过程中作弊（例如，为了通过测试而修改测试标准）。这保证了最终结果的公正和可靠。
