## 0. 概览（Concise Summary）

- **主题（Topic）**：自动驾驶系统（ADS）测试场景的创建与复用。
- **问题（Problem）**：为自动驾驶系统创建复杂、逼真的测试场景既耗时又费力，且现有方法缺乏模块化和可复用性，导致代码冗余、开发效率低下。
- **方法（Method）**：提出并实现了一个名为 **Modular2Simple** 的工具。该工具基于 OpenSCENARIO 格式，允许开发者通过组合多个简单的或已有的模块化场景来构建复杂的模块化场景，并最终将其转换为单个标准场景文件。
- **创新（Innovation）**：引入了一种**模块化场景构建范式**，通过自定义的 `.mosc` 包格式和 `ScenarioReference` / `parameterAttributes` 标签，实现了场景的灵活组合、参数化定制与高效复用，显著降低了代码复杂度和开发工作量。
- **意义（Significance）**：为ADS测试场景的创建提供了一种更高效、更灵活、更易于维护的方法。它促进了场景的复用和标准化，有助于加速自动驾驶系统的开发与验证流程，并降低了因代码冗余引入错误的风险。
- **一句话总结**：Modular2Simple 通过模块化组合和参数化复用的方式，简化了基于 OpenSCENARIO 格式的复杂自动驾驶测试场景的创建过程，显著减少了代码量并提升了开发效率。

## 1. 研究动机（Motivation）

- **解决的问题**：在自动驾驶系统（ADS）的开发和测试中，创建复杂、多样且逼真的交通场景是验证系统安全性和可靠性的关键。然而，传统的场景创建方法通常需要为每个测试用例编写冗长、独立的 OpenSCENARIO 文件。这种方法存在以下问题：
    1.  **开发效率低**：创建复杂场景需要大量手动编码，耗时耗力。
    2.  **代码冗余严重**：许多场景包含相似的行为模式（如车辆变道、行人过马路），但每次都需要重复编写，导致代码库庞大且难以维护。
    3.  **缺乏复用性**：场景之间难以共享和复用，一旦需要修改某个通用行为，必须逐一修改所有相关场景文件，容易出错且维护成本高。
    4.  **灵活性不足**：难以快速调整场景参数（如速度、距离）以生成不同的测试变体。
- **重要性**：解决这些问题对于加速ADS的研发进程至关重要。一个高效、模块化的场景创建工具可以：
    - 显著缩短测试场景的开发周期。
    - 降低代码维护的复杂度和成本。
    - 通过复用已验证的场景模块，提高测试的一致性和可靠性。
    - 使研究人员能够更专注于测试逻辑本身，而非繁琐的编码工作。
- **挑战**：主要挑战在于如何设计一种**通用、灵活且与现有标准兼容**的模块化机制。需要在不破坏 OpenSCENARIO 标准的前提下，引入模块化概念，并确保生成的最终场景能被主流模拟器（如 CARLA）正确解析和执行。

## 2. 创新点（Innovation）

- **关键创新**：提出了一个**基于 OpenSCENARIO 格式的模块化场景构建框架和工具**。其核心创新在于：
    1.  **模块化场景包（`.mosc` 格式）**：定义了一种新的文件格式（`.mosc`），用于将多个简单的或模块化的场景文件打包成一个逻辑单元。这个包包含一个主文件（`main.xosc`）和所有引用的子场景文件，实现了场景的物理和逻辑封装。
    2.  **`ScenarioReference` 标签**：在模块化场景的主文件（`main.xosc`）中，使用自定义的 `<ScenarioReference>` 标签来引用子场景，而不是直接编写具体的驾驶行为。这实现了场景的**组合**。
    3.  **`parameterAttributes` 属性与 `ParameterReference` 标签**：在简单场景文件中，通过 `parameterAttributes` 属性声明哪些参数可以被外部定制。在模块化场景的主文件中，通过 `<ParameterReference>` 标签为这些参数赋值。这实现了场景的**参数化复用**。
- **与现有方案的不同**：与 GeoScenario、SceML 等基于特定领域语言（DSL）或图形化建模的方法不同，Modular2Simple 直接**工作在 OpenSCENARIO 标准之上**，通过扩展其语义而非语法来实现模块化。这使得它能够与任何支持 OpenSCENARIO 格式的模拟器（如 CARLA）无缝集成，无需修改模拟器本身。它更像是一个**预处理器**，将模块化的设计转化为标准的、可执行的场景文件。
- **为什么能奏效**：该创新之所以有效，是因为它抓住了软件工程中“**高内聚、低耦合**”的模块化设计原则。它将复杂的场景分解为独立的、可复用的简单行为模块（高内聚），并通过清晰的接口（参数化标签）将它们组合起来（低耦合）。这直接解决了传统方法中代码冗余和难以复用的问题。

## 3. 主要内容（Main Content）【重点】

### 3.1 设计架构与方法

Modular2Simple 的架构围绕两个核心操作展开，如图 1 所示：

1.  **打包（Packaging）**：将一组简单的 `.xosc` 文件或已有的 `.mosc` 模块化文件打包成一个单一的 `.mosc` 模块化场景文件。
    - 命令：`java Modular2Simple -cxm <input_xosc_and_mosc_files> <output_mosc_file>`
2.  **转换（Converting）**：将一个 `.mosc` 模块化场景文件转换为一个单一的、标准的 `.xosc` 场景文件，供模拟器直接使用。
    - 命令：`java Modular2Simple -cmx <input_mosc_file> <output_xosc_file>`

**工作流程**：
- 开发者首先创建多个描述单一行为（如“车辆A从路口驶出”、“行人B横穿马路”）的简单 OpenSCENARIO 文件。
- 然后，使用 `-cxm` 命令，通过一个主文件（`main.xosc`）引用这些简单场景，并定义它们之间的组合关系和参数，打包成一个 `.mosc` 文件。
- 最后，使用 `-cmx` 命令，工具会解析 `.mosc` 包，将所有引用的子场景和参数展开，合并成一个完整的、标准的 `.xosc` 文件，供 CARLA 等模拟器执行。

### 3.2 关键算法与数学推导

本论文的核心并非提出新的算法或数学推导，而是提出一种**场景描述的结构化方法和转换逻辑**。其“算法”本质上是**一个基于规则的文本解析与代码生成过程**。

**核心逻辑**：
1.  **解析主文件**：读取 `.mosc` 包中的 `main.xosc` 文件，识别其中的 `<ScenarioReference>` 标签。
2.  **查找子场景**：根据 `scenarioFileName` 属性，在 `.mosc` 包内或 `SCENARIO_PATH` 指定的路径下找到对应的子场景文件（`.xosc` 或 `.mosc`）。
3.  **参数注入**：对于每个 `<ScenarioReference>`，解析其内部的 `<ParameterReference>` 标签，获取需要定制的参数名（`key`）和值（`value`）。
4.  **递归展开**：如果引用的子场景本身是 `.mosc` 文件，则递归执行步骤 1-3，直到所有引用都被解析为简单的 `.xosc` 文件。
5.  **代码合并与替换**：将子场景文件中的 `<ManeuverGroup>` 等核心内容复制到最终输出的 `.xosc` 文件中。同时，根据步骤 3 中获取的参数，替换子场景中由 `parameterAttributes` 声明的属性值。如果主文件中未提供某个参数的值，则使用子场景中定义的默认值。

**关键数据结构**（用伪代码表示）：
```
// 主文件中的引用结构
ScenarioReference {
    scenarioFileName: String
    ManeuverGroupReference {
        maneuverGroupName: String
        ParameterReference {
            key: String
            value: String
        }[]
    }[]
}

// 简单场景中的可定制声明
ManeuverGroup {
    parameterAttributes: String[]  // 声明哪些属性可被外部定制
    // ... 其他标准 OpenSCENARIO 内容
    Condition parameterAttributes="key1" key1="defaultValue1" ... 
}
```

### 3.3 模型、训练与数据集

- **模型**：本论文不涉及机器学习模型。Modular2Simple 本身是一个**基于规则的软件工具**，其“模型”是它所定义的模块化场景结构（`.mosc` 格式）。
- **训练**：无。
- **数据集**：论文中用于验证的“数据集”是一个由作者创建的**小型场景库**，包含 31 个不同的测试场景。这些场景涉及行人过马路、车辆交叉路口、车辆从后方驶来等多种交通参与者行为。

### 3.4 实验设置与结果

- **实验设置**：
    - **模拟器**：CARLA 0.9.15
    - **硬件**：Intel i7-10750H CPU, 8GB RAM, NVIDIA GTX 1650 GPU
    - **案例研究**：展示了一个复杂的多车事故场景（两车碰撞 -> 警车和救护车到达 -> 所有车辆离开），如图 2 所示，验证了工具构建复杂场景的能力。
    - **量化实验**：创建了一个包含 31 个场景的小型库，这些场景的行为由**仅 2 个**可复用的简单场景控制。
- **实验结果**：
    - **代码量对比**：如表 1 所示，使用 Modular2Simple 的模块化方法，整个场景库的代码行数（LoC）为 **10,810 行**，而使用传统的单个 OpenSCENARIO 文件方法则需要 **21,243 行**。
    - **代码缩减率**：整体代码量减少了 **49.1%**。单个场景的代码缩减率在 44.1% 到 54.3% 之间。
    - **结论**：模块化方法显著减少了代码冗余，提高了开发效率。论文指出，通过更广泛地复用更复杂的行为，代码缩减率可以进一步提高。

### 3.5 技术细节

- **实现语言**：Java。
- **交互方式**：命令行界面（CLI）。
- **文件格式**：
    - `.xosc`：标准的 OpenSCENARIO 格式文件，用于定义简单的、不可再分的场景行为。
    - `.mosc`：自定义的模块化场景包格式，本质上是包含 `main.xosc` 和其他子场景文件的压缩包或目录结构。
- **环境变量**：`SCENARIO_PATH` 用于指定工具搜索子场景文件的路径，增强了模块的跨目录复用能力。
- **默认值机制**：在简单场景中通过 `parameterAttributes` 声明的属性可以设置默认值。如果主文件未提供新值，则使用默认值，这保证了模块的鲁棒性和易用性。

## 4. 意义与影响（Significance and Impact）

- **解决的问题与贡献**：
    - **解决了**自动驾驶测试场景创建中**效率低、冗余高、复用难**的核心痛点。
    - **贡献**了一个**开源、轻量级、与标准兼容**的工具 Modular2Simple，它通过引入模块化设计，将场景创建从“手写长篇代码”转变为“组合可复用模块”，极大地简化了流程。
- **对未来研究/应用的影响**：
    - **加速ADS开发**：为研究人员和工程师提供了一种高效构建和迭代测试场景的方法，从而加速ADS的验证和部署。
    - **促进场景标准化**：鼓励社区创建和共享标准化的场景模块库，推动整个领域向更高效、更一致的测试实践发展。
    - **降低测试门槛**：使非编程专家也能通过组合预定义的模块来创建复杂的测试场景。
- **作者提到的局限与开放问题**：
    - **兼容性问题**：OpenSCENARIO 标准与 CARLA 等模拟器之间存在不完全兼容的情况，部分标签和功能可能无法被正确解析，导致场景行为与预期不符。
    - **缺乏预定义模块库**：工具本身没有提供一个丰富的、开箱即用的场景模块库，用户需要从零开始创建基础模块。
    - **未来方向**：
        1.  扩展对其他场景格式（如 GeoScenario）的支持。
        2.  改进 OpenSCENARIO 元素到不同模拟器的映射，确保跨平台一致性。
        3.  建立一个全面的、社区驱动的预定义场景模块库。

## 5. 澄清与简化（Clarifications and Simplifications）

- **类比理解**：可以把 Modular2Simple 想象成一个**乐高积木系统**。
    - **简单场景**：就是一块块**基础乐高积木**，比如一块红色的 2x4 砖块（代表“车辆直行”），一块蓝色的 1x2 砖块（代表“行人停止”）。每个积木都有特定的形状和功能。
    - **`parameterAttributes`**：就像积木上的**插孔**，你可以通过它来改变积木的颜色或大小。例如，一个“车辆直行”积木，可以通过插孔设定它的速度是 30km/h 还是 60km/h。
    - **`ScenarioReference`**：就是**搭建说明书**上的步骤，它告诉你“把红色积木 A 放在蓝色积木 B 的上面”。
    - **`.mosc` 包**：就是**一个装有说明书和所有需要用到的积木的盒子**。
    - **`-cmx` 转换**：就像**把说明书和所有积木拼成一个完整的、不可拆分的最终模型**（比如一辆完整的乐高汽车），这个模型可以直接拿去玩（在模拟器中运行）。
- **为什么能减少代码**：传统方法相当于为每一辆不同的乐高汽车都写一份全新的、从头到尾的搭建说明书。而 Modular2Simple 允许你为“车轮”、“车身”、“车灯”等通用部件各写一份说明书（简单场景），然后通过组合和参数调整（比如“车轮”的说明书里设定轮毂颜色为黑色或银色），就能快速拼出几十种不同的汽车。你只需要维护那几份通用部件的说明书，而不是几十份完整的汽车说明书，代码量自然就大幅减少了。

## 6. 补充说明（Additional Notes）

- **相关工作关联**：论文在“相关工作”部分提到了几种主流方法：
    - **基于SDL的方法**（GeoScenario, stiEF, SceML）：这些方法通过创建新的领域特定语言或图形化模型来定义场景，虽然强大，但学习成本高，且与现有模拟器的集成可能不如直接基于 OpenSCENARIO 标准来得直接。Modular2Simple 选择直接工作在 OpenSCENARIO 之上，是一种更轻量、更易集成的方案。
    - **数据驱动的方法**：利用真实事故数据（如CIDAS, GES）生成场景。这种方法能产生高度真实的场景，但通常需要大量数据，且生成的场景可能难以直接模块化。Modular2Simple 可以与数据驱动方法互补：从数据中提取出的典型行为模式可以被封装成 Modular2Simple 的简单场景模块，供后续组合复用。
