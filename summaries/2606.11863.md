# Enhancing LLM-Based Code Translation with Verified Multi-Semantic Representations

- arXiv: https://arxiv.org/abs/2606.11863v1
- arXiv ID: 2606.11863

---

## 0. 概览（Concise Summary）

- **主题（Topic）**：基于大语言模型（LLM）的代码翻译。
- **问题（Problem）**：现有LLM代码翻译方法主要依赖词级别的统计模式进行映射，缺乏对程序语义的充分理解，导致翻译结果中存在可干预的逻辑和语义错误。同时，在缺乏外部语义资源（如文档、测试用例）的现实场景中，如何从源代码自身构建丰富、多样且可靠的语义信息来指导翻译是一个关键挑战。
- **方法（Method）**：提出了一个名为**Multisage**的多语义增强与自校准框架。该框架包含三个核心模块：1）**语义表示解析模块**，从源代码中提取结构化的基础语义表示（如数据流图、类型约束、外部API信息）；2）**多语义增强模块**，基于基础语义表示构建增强语义（如函数级测试用例、代码摘要、API描述与测试用例）；3）**语义一致性校准模块**，通过语义保持的代码变异和跨语义一致性验证，自动过滤、校准和精炼增强后的语义，确保其准确性和可靠性。
- **创新（Innovation）**：1）系统性地识别并量化了LLM代码翻译失败中“可干预语义错误”的主导地位。2）提出了一个无需外部资源、仅从源代码出发，就能自动构建并校准多样化、互补性语义信息的完整框架。3）引入了一种基于语义保持变异的自校准机制，通过跨视图一致性验证来确保生成语义的可靠性。
- **意义（Significance）**：证明了显式的语义增强可以有效提升LLM代码翻译的可靠性，尤其是在小模型上效果显著。该工作为提升代码翻译的鲁棒性和正确性提供了一种实用且有效的新范式，表明改进代码翻译的关键不在于更大的模型，而在于如何显式地利用和验证语义信息。
- **一句话总结**：本文提出Multisage框架，通过从源代码中自动构建并自校准多种互补的语义表示，显著提升了LLM在代码翻译任务中的成功率和可靠性。

## 1. 研究动机（Motivation）

- **问题与空白**：现有的LLM代码翻译方法，如CodeBERT和TransCoder，虽然取得了显著进展，但其本质上是基于词级别的统计模式进行源语言到目标语言的映射。这种“表面”的映射方式导致模型在处理复杂的跨语言差异（如控制流、API依赖、类型约束）时，无法深入理解程序语义，从而产生逻辑错误、依赖错误或数据解析错误等“可干预”的语义错误。这些错误并非模型能力不足的随机结果，而是由于翻译过程中缺乏显式、可靠的语义约束所致。
- **重要性**：代码翻译是一项严格保持语义等价的任务，即使微小的语义偏差也会导致程序行为错误。在大型软件系统迁移和现代化改造中，这些错误会带来巨大的维护和集成成本。因此，提升代码翻译的可靠性至关重要。
- **挑战**：在现实场景中，高质量的语义资源（如功能描述、测试用例、正式规范）往往不可用。因此，必须直接从源代码中构建语义信息来指导翻译。这带来了两个核心挑战：
    1.  **如何构建丰富多样的多语义信息**：单一语义视角（如仅控制流或仅类型）只能捕捉程序行为的一个有限侧面。需要系统地构建多种互补的语义形式（如代码摘要、API约束、测试用例），以全面覆盖程序行为。
    2.  **如何确保语义信息的准确性和可靠性**：通过静态分析或近似推断构建的语义信息可能包含噪声、不完整或相互矛盾。如果直接将这种不可靠的语义用于指导模型，不仅无法纠正错误，反而可能误导模型，产生看似合理但实际偏离意图的输出。因此，需要一种机制来验证、过滤和校准这些语义信息。

## 2. 创新点（Innovation）

- **关键创新**：
    1.  **问题驱动的创新**：通过系统的实证分析，首次明确指出并量化了LLM代码翻译失败的主要原因是“可干预的语义错误”，而非模型容量限制。这为后续研究指明了方向：提升翻译可靠性的关键在于引入显式的语义约束。
    2.  **框架创新：Multisage**：提出了一个端到端的、无需外部资源的“多语义增强与自校准”框架。该框架将语义的**构建**、**增强**和**校准**整合为一个协同工作的流水线。
    3.  **方法创新：语义一致性校准模块**：这是本文最核心的创新。它提出了一种基于**语义保持变异**和**跨视图一致性验证**的自校准机制。通过生成与原程序语义等价的变异体，并比较原始语义与变异体语义的一致性，来自动识别和过滤不可靠的语义信号。这解决了“如何确保自建语义可靠”这一关键难题。
    4.  **训练策略创新：多任务联合微调**：在多语义增强模块中，采用了基于自适应权重（如FAMO）的多任务联合微调策略，动态平衡不同语义生成任务（摘要、测试用例等）的收敛速度，提升了模型生成多样化、高质量语义的能力。

- **与现有方案的不同**：
    - 与依赖外部资源（如文档、测试用例）的方法不同，Multisage完全从源代码自身出发。
    - 与仅使用单一语义（如AST）的方法不同，Multisage构建了多种互补的语义视图。
    - 与直接使用LLM生成语义（如CoT）的方法不同，Multisage引入了严格的校准机制来确保生成语义的可靠性，而非盲目信任。

## 3. 主要内容（Main Content）【重点】

### 3.1 设计架构与方法

Multisage框架的工作流程如图4所示，包含三个核心模块：

1.  **M1: 代码表示解析模块**：对输入的源代码进行静态分析，提取结构化的**基础语义表示（Rbase）**。这包括：
    - **控制流提取**：通过AST和CFG分析，生成线性化的执行模板。
    - **类型约束解析**：提取函数签名中的参数和返回类型。
    - **外部接口提取**：识别库函数调用、方法调用等外部依赖，并将其抽象为规范形式。

2.  **M2: 多语义增强模块**：以Rbase为输入，构建更丰富的**增强语义**。其工作流程如图5所示，分为三步：
    - **多语义数据构建**：利用LLM（如GPT-4 Turbo）作为生成代理，基于种子数据集（XLCoST, XCodeEval）和Rbase，生成代码摘要、函数级测试用例、API级描述与测试用例。并通过执行验证器进行校验和迭代修正。
    - **多语义增强模型微调**：使用多任务联合微调策略（如FAMO），训练一个专门的模型来同时生成多种语义。
    - **多语义增强模型推理**：对于新的源代码，使用训练好的模型并行生成多种语义，并对生成的测试用例进行执行验证，最多进行三轮修正。

3.  **M3: 语义一致性校准模块**：对M2生成的增强语义进行可靠性验证和校准。
    - **等价变异器**：通过静态规则（如表达式重写、控制流重写、API替换）生成3-5个与原程序语义等价的变异体。
    - **一致性评估器**：将原始增强语义（S0）与从变异体重构的增强语义（{S'}）进行比较。通过计算语义单元（如API约束、功能陈述）的Jaccard相似度，并基于多数投票原则，计算一致性得分C。若C超过阈值τ（设为0.6），则认为S0可靠，直接使用；否则，触发校准单元。
    - **语义校准单元**：当C < τ时，从S0和{S'}中聚合获得多数支持的语义单元，重构出校准后的最终语义（S_final）。

### 3.2 关键算法与数学推导

- **多任务联合微调损失函数**：
  假设有N个语义生成任务，每个任务T_i的损失函数为L_i(θ)，则总体优化目标为：
  $$L(\theta) = \sum_{i=1}^{N} w_i \cdot L_i(\theta)$$
  其中，θ是模型参数，w_i是第i个任务的归一化自适应权重，用于动态平衡不同任务的收敛速度。

- **语义一致性评估**：
  1.  **语义单元匹配**：对于原始语义S0中的一个语义单元u，定义能在变异体集合{S'}中被匹配的变异体集合为：
      $$M(u) = \{ S' \in \{S'\} \mid \exists v \in U(S') \text{ s.t. } \text{sim}(u, v) \ge \delta \}$$
      其中，相似度函数sim(u, v)为Jaccard相似度：
      $$\text{sim}(u, v) = \frac{|T(u) \cap T(v)|}{|T(u) \cup T(v)|}$$
      T(u)和T(v)分别是u和v的归一化token集合。δ是相似度阈值（设为0.7）。

  2.  **语义单元支持度**：语义单元u的支持度定义为：
      $$\text{supp}(u) = \frac{|M(u)|}{|\{S'\}|}$$

  3.  **稳定语义单元集**：支持度超过阈值ρ（设为0.5）的单元被认为是稳定的：
      $$U_{\text{stable}} = \{ u \in U(S_0) \mid \text{supp}(u) \ge \rho \}$$

  4.  **一致性得分**：最终的一致性得分C定义为稳定单元占原始语义单元总数的比例：
      $$C = \frac{|U_{\text{stable}}|}{|U(S_0)|}$$

### 3.3 模型、训练与数据集

- **数据集**：
    - **种子数据集**：XCodeEval（95,425个C++样本）和XLCoST（11,198个C++样本），用于构建多语义增强数据集。
    - **多语义增强数据集**：通过M2模块的数据构建流程，从种子数据集生成，包含309,212个C++样本和311,890个Java样本。
    - **测试数据集**：HumanEval-X，包含164个编程挑战，每个任务都有单元测试和参考实现，用于评估C++到Java的翻译。

- **模型**：
    - **多语义增强模型**：基于MFTCoder配置进行微调的模型，使用QLoRA INT4量化，参数比例为2.52%。
    - **翻译骨干模型**：评估了多种LLM，包括：
        - HP-LLMs (>100B): DeepSeek-V3, GPT-4 Turbo, Qwen-Plus
        - MS-LLMs (10B-100B): Qwen2.5-Coder-Instruct-33B, StarCoder2-15B
        - ℓLMs (<10B): StarCoder2-7B, Qwen2.5-Coder-7B

- **训练细节**：
    - 优化器：Adam
    - 全局批次大小：128
    - 学习率：初始2×10⁻⁴，采用余弦退火策略，warmup比例为3×10⁻²，最小学习率为1×10⁻⁵。

### 3.4 实验设置与结果

- **评估指标**：
    - **代码翻译成功率**：主要指标，翻译后的代码必须通过HumanEval-X中所有对应的单元测试。
    - **CodeBLEU**：辅助指标，用于衡量结构化和数据流层面的语义一致性。

- **基线方法**：
    - **LLM基线**：上述提到的多种LLM，使用vanilla prompting。
    - **专用代码翻译基线**：TransCoder, DOBF, TransCoder-ST, TransCoder-IR, INTERTRANS。
    - **语义增强能力基线**：使用DeepSeek-V3, GPT-4 Turbo, Qwen-Plus直接生成语义约束。

- **主要结果**：
    1.  **与Vanilla Prompting对比**：Multisage在所有评估模型上均提升了翻译成功率，最高提升**2.22倍**（在StarCoder2-7B上）。
    2.  **与专用模型对比**：Multisage在CodeBLEU得分上显著优于TransCoder等专用模型，同时保持了有竞争力的执行成功率。
    3.  **与语义增强策略对比**：与CoT推理和单阶段语义提示相比，Multisage在小、中、大规模模型上分别实现了最高1.42倍、1.28倍和1.17倍的翻译成功率提升。
    4.  **错误类型分析**：如图2所示，实验证实了大部分翻译失败（尤其是在大模型上）属于“可干预”的语义错误，而非模型自身错误。

### 3.5 技术细节

- **语义表示解析**：使用Tree-sitter库进行AST解析，使用Joern工具进行CFG生成。
- **等价变异**：变异规则是纯静态和确定性的，基于AST、CFG和符号表进行操作，不涉及学习组件。变异后失败的编译或执行检查的变体会被丢弃。
- **语义单元**：将自然语言描述（如摘要、API描述）和测试用例中的断言都视为“语义单元”，通过token级别的Jaccard相似度进行比较。
- **阈值设置**：默认参数为δ=0.7（单元级相似度），ρ=0.5（多数支持），τ=0.6（全局一致性）。实验表明，框架对这些参数的变化是鲁棒的。

## 4. 意义与影响（Significance and Impact）

- **解决的问题与贡献**：
    - 解决了LLM代码翻译中因缺乏语义理解而导致可靠性低的问题。
    - 提供了一个实用的、无需外部资源的解决方案，通过自动构建和校准语义来提升翻译质量。
    - 提出了一个新颖的“语义自校准”范式，通过代码变异和一致性验证来确保自建语义的可靠性，这是该领域的核心贡献。

- **对未来研究/应用的影响**：
    - **研究影响**：为代码翻译研究开辟了新方向，即从“提升模型容量”转向“显式利用和验证语义”。其自校准思想可以推广到其他需要语义保证的代码生成任务（如程序修复、代码补全）。
    - **应用影响**：可以集成到现有的代码迁移工具中，显著提高自动化程度和翻译质量，降低人工审查成本。尤其对于资源受限的小模型，其提升效果更为显著，具有很高的实用价值。

- **局限与开放问题**：
    - 框架的复杂性：Multisage包含多个模块和步骤，其端到端的延迟和计算成本可能高于简单的prompting方法。
    - 对变异规则的依赖：校准模块的有效性依赖于等价变异规则的质量和覆盖度。对于某些特定领域的代码，可能难以设计完备的变异规则。
    - 多语义增强模型的泛化性：虽然实验验证了跨模型泛化能力，但该模型本身是在特定数据集上训练的，其对于完全未见过的编程范式或库的泛化能力有待进一步研究。

## 5. 澄清与简化（Clarifications and Simplifications）

- **核心思想类比**：可以把Multisage想象成一个“翻译助手”，它不像普通翻译器那样逐词翻译，而是先做“阅读理解”。
    1.  **M1（代码表示解析）**：相当于助手先画出原文的**逻辑结构图**（控制流）、**词性标注**（类型约束）和**引用的外部资料**（API信息）。
    2.  **M2（多语义增强）**：助手根据逻辑结构图，自己编写了**内容摘要**、**关键术语解释**和**几个简单的测试题**（测试用例），以确保自己理解了原文的“意思”。
    3.  **M3（语义一致性校准）**：这是最聪明的一步。助手为了确保自己写的“摘要”和“测试题”是正确的，会**用不同的句式重写原文**（等价变异），然后看看自己根据这些改写版本写出的“摘要”和“测试题”是否和之前的一致。如果一致，说明理解是可靠的；如果不一致，就说明之前的理解可能有偏差，需要**综合所有版本中最一致的部分**来修正自己的理解。

- **“可干预语义错误”的通俗解释**：想象一下，把中文的“他跑得很快”翻译成英文。一个基于统计的模型可能会翻译成“He runs very fast”，这没问题。但如果句子是“他跑得快，但跑得不远”，模型可能只看到“跑得快”就翻译成“He runs fast”，忽略了“但”带来的转折关系，这就是一个“可干预的语义错误”。它不是模型不认识单词，而是没有理解句子内部的逻辑关系。Multisage就是通过提供“这是一个转折关系的句子”这样的语义提示，来帮助模型避免这种错误。
