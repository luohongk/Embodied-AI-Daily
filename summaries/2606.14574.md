## 0. 概览（Concise Summary）

- **主题（Topic）**：大型语言模型（LLM）在可执行任务规划中的可靠性评估，特别是针对家庭环境（厨房场景）的自主代理。
- **问题（Problem）**：现有基准测试主要评估LLM生成的计划是否能成功执行，但忽略了**潜在故障（Latent Failures）**。这类故障不会立即阻止计划执行，而是悄无声息地损害最终目标的达成，甚至导致不可逆的后果。
- **方法（Method）**：提出了一个名为**SIMMER**的基准测试框架。它包含三个核心组件：1）一个基于真实烹饪脚本构建的、包含77个动作和262个对象的符号化世界模型；2）一个将故障区分为即时故障和潜在故障的分类法；3）一个状态机执行器，用于模拟计划执行，跟踪细粒度状态，并检测传统指标无法发现的故障。
- **创新（Innovation）**：首次系统性地定义和评估了LLM规划中的“潜在故障”问题。通过构建一个更复杂、更贴近现实的符号世界模型，揭示了LLM在跟踪隐式状态依赖（如交叉污染）方面的严重缺陷。此外，提出了**反事实预想模拟（Counterfactual Foresight Simulation）** 这一提示策略，通过强制模型在每一步行动前进行显式状态推理，显著减少了潜在故障。
- **意义（Significance）**：该工作揭示了当前LLM规划能力评估中一个被忽视的关键盲点。它表明，仅凭表面上的执行成功率会严重高估LLM的规划可靠性。SIMMER为开发更安全、更鲁棒的LLM规划器提供了重要的评估工具和方向指引，尤其是在高风险领域（如家庭助手、机器人烹饪）。
- **一句话总结**：本文提出了SIMMER基准，通过一个精细的符号世界模型揭示了LLM在规划中普遍存在且常被忽视的“潜在故障”，并证明显式的状态推理是缓解该问题的有效途径。

## 1. 研究动机（Motivation）

- **解决的问题与空白**：当前评估LLM规划能力的方法存在两大局限，导致它们无法检测到一类关键的错误——**潜在故障**。
    1.  **基于虚拟环境的方法**（如TextWorld, ALFWorld）：虽然能进行精确的状态跟踪，但环境过于简化（如TextWorld仅定义了10种对象类型和26个动作规则），无法模拟现实世界中复杂的隐式状态变化，如**污染传播、温度变化、化学转化**等。
    2.  **基于自然语言语义相似度的方法**：通过衡量生成计划与参考计划的语义相似度来评估。这种方法过于表面化，一个由孤立看来合理的动作组成的计划，可能因为忽略了动作间的状态依赖而包含致命错误。
- **为什么重要**：LLM正被部署到家庭助手等安全关键领域。如果规划器无法检测到像“用切过生鸡肉的砧板直接切沙拉蔬菜”这样的潜在故障，将导致严重的食品安全问题。这种故障在计划执行时不会报错，但后果是灾难性的。
- **挑战在哪里**：潜在故障的检测需要模型不仅理解“能否执行”，还要理解“是否应该执行”。这要求模型能够模拟动作的**下游效应**和**隐式状态传播**，而不仅仅是检查显式的先决条件。现有方法缺乏对这种深层因果推理能力的评估。

## 2. 创新点（Innovation）

- **关键创新**：提出了一个全新的基准测试框架**SIMMER**，用于系统性地评估LLM规划中的潜在故障。
- **与现有方案的不同**：
    1.  **更真实的世界模型**：SIMMER的世界模型基于真实世界的烹饪脚本（wikiHow, Instructables）构建，包含77个动作和262个对象，支持约46,800种交互。其规模和领域特异性远超TextWorld、ALFWorld等通用或简化环境，能够模拟污染、温度等复杂的隐式状态。
    2.  **全新的故障分类法**：明确区分了**即时故障**（执行时立即触发）和**潜在故障**（执行时无反馈，但最终导致失败）。在潜在故障中，进一步区分了**可逆**和**不可逆**的故障，后者代表了灾难性的规划错误。
    3.  **细粒度的状态机执行器**：该执行器不仅检查动作的先决条件，还能在模拟过程中跟踪并传播隐式状态（如污染），并在执行后审计最终状态，从而检测出传统方法无法发现的潜在故障。
- **为什么能奏效**：通过构建一个足够复杂且领域知识丰富的符号世界，SIMMER能够模拟出导致潜在故障的因果链。其状态机执行器能够精确地捕捉到这些因果链中的每一个环节，从而将隐式的、无声的故障显式地暴露出来。

## 3. 主要内容（Main Content）【重点】

### 3.1 设计架构与方法

SIMMER的整体架构如图2所示，包含三个核心组件：

1.  **符号化世界模型（Symbolic World Model）**：定义了厨房领域的动作和对象。
    - **动作**：定义为元组 `<args, preconditions, effects>`（参数，先决条件，效果）。例如，`grab`动作的参数是`object`，先决条件是`agent hands empty`（手空）和`object grabbable`（对象可抓取），效果是`agent holding object`（拿着对象）。
    - **对象**：定义为元组 `<properties, states, location>`（属性，状态，位置）。例如，`chicken breast`的属性包括`cuttable, grabbable, protein`，状态包括`raw, whole`，位置在`fridge`。
    - **规模**：包含77个动作和262个对象，支持约46,800种交互，远超其他虚拟环境（见表2）。

2.  **故障分类法（Failure Taxonomy）**：
    - **即时故障（Immediate Failures）**：违反当前动作的先决条件，阻止执行。例如，在没有拿刀的情况下尝试`chop`。
    - **潜在故障（Latent Failures）**：不违反任何先决条件，执行时无反馈，但最终损害目标。例如，忘记加发酵粉做蛋糕，蛋糕最终无法膨胀。
        - **不可逆潜在故障（Irreversible Latent Failures）**：一旦发生，无法通过后续任何动作恢复。例如，生鸡肉污染了砧板，之后用同一砧板切蔬菜导致交叉污染。

3.  **状态机执行器（State Machine Executor）**：模拟计划执行并检测故障。
    - **第一阶段：逐步执行**。解析每个动作，检查先决条件（检测即时故障），应用状态转换（包括传播隐式状态如污染）。
    - **第二阶段：执行后审计**。扫描最终状态，检查是否存在未解决的异常情况，如食物未煮熟、电器未关等（检测潜在故障）。

### 3.2 关键算法与数学推导

核心算法是状态机执行器的执行逻辑，如算法1所示。

**算法1：通过状态机执行计划**

**输入**：计划 $P = \langle a_1, a_2, \ldots, a_n \rangle$，环境 $E = (O, A)$
**输出**：故障集合 $F$

1.  **初始化**：
    - 世界状态 $S \leftarrow \text{INITSTATE}(E)$
    - 代理状态 $\sigma \leftarrow (\text{loc: kitchen, hold: } \emptyset)$
    - 故障集合 $F \leftarrow \emptyset$

2.  **逐步执行**（对于 $i = 1$ 到 $n$）：
    - 解析动作 $a_i$ 和参数 $r$。
    - **先决条件检查**：对于动作 $a$ 的每个先决条件 $\pi$：
        - 如果 $\neg \pi(S, \sigma, r)$，则记录一个即时故障：$F \leftarrow F \cup \{(i, \text{IMMEDIATE}, \pi)\}$
    - **状态转换**：$(S, \sigma) \leftarrow \text{APPLY}(a, r, S, \sigma)$

3.  **执行后审计**：
    - 对于每个食材 $o \in S$：
        - 如果 $o.\text{contam} \neq \emptyset$，则记录潜在故障：$F \leftarrow F \cup \{(n+1, \text{LATENT}, \text{contamination})\}$
        - 如果 $\neg \text{EDIBLE}(o)$，则记录潜在故障：$F \leftarrow F \cup \{(n+1, \text{LATENT}, \text{uncooked})\}$
    - 对于每个电器 $o \in S$：
        - 如果 $\text{on} \in o.\text{states}$，则记录潜在故障：$F \leftarrow F \cup \{(n+1, \text{LATENT}, \text{safety})\}$

4.  **返回** $F$

**关键点**：该算法的核心在于 `APPLY` 函数，它不仅更新对象的位置和持有状态，还会根据领域规则（如“生肉接触表面导致表面污染”）更新隐式状态（如 `contam` 属性）。而 `EDIBLE` 函数则检查食物是否经过了必要的烹饪步骤。

### 3.3 模型、训练与数据集

- **任务数据集**：从wikiHow和Instructables收集了100个烹饪脚本，涵盖12种烹饪技术，任务复杂度从2到18个自然语言步骤不等。平均每个任务涉及26.6个动作和31.5个对象。
- **评估模型**：评估了6个LLM，包括：
    - **专有模型**：GPT-5.4, Gemini 3 Flash, Claude Opus 4.6。
    - **开源模型**：Llama 3.3 70B, DeepSeek V3.2, Qwen 3.5 27B。
- **训练**：本文不涉及模型训练，所有模型均为零样本或少样本评估。
- **数据集构建**：通过GPT-5.4从烹饪脚本中提取动作和对象，然后进行人工验证、规范化、去重和低频率项筛选。最后，使用GPT-5.4和人工审核来标注每个动作和对象的正式定义（先决条件、效果、属性、状态）。

### 3.4 实验设置与结果

- **实验设置**：所有模型在温度 $T=0$ 下进行确定性生成。评估指标包括：包含故障的计划比例、每个计划的平均故障数，并细分为即时、潜在和不可逆故障。
- **主要结果（表3）**：
    - **整体表现堪忧**：即使是最先进的模型，生成无错误计划的比例也低于17%（平均仅7.2%）。
    - **潜在故障普遍**：29-56%的计划包含潜在故障，其中大部分是不可逆的。
    - **模型间差异**：专有模型（如GPT-5.4）表现优于开源模型，但所有模型都存在严重的潜在故障问题。
- **故障缓解效果（图4）**：
    - **Self-Refine**：对GPT-5.4和Claude Opus 4.6等前沿模型有效，能将总故障减少40%以上，但对弱推理能力的模型效果甚微甚至为负。
    - **反事实预想模拟（Counterfactual Foresight）**：在所有故障类别上均优于Self-Refine。对于GPT-5.4，它将即时故障减少了73%，潜在故障减少了69%，不可逆故障减少了45%。对于Claude Opus 4.6，效果更佳，分别减少了80%、72%和75%。
- **案例分析（表4）**：展示了GPT-5.4在“制作培根鸡蛋奶酪三明治”任务中的失败计划和成功计划。失败计划中，模型先处理生培根，然后直接用手接触面包，导致污染，并且忘记关火。而使用反事实预想模拟的成功计划则调整了顺序，先处理即食食材，再处理生蛋白，并在上菜前关火。

## 4. 意义与影响（Significance and Impact）

- **解决的问题与贡献**：
    1.  **填补评估空白**：首次系统性地定义并量化了LLM规划中的“潜在故障”问题，揭示了现有评估方法的重大盲点。
    2.  **提供评估工具**：发布了SIMMER基准，包括一个丰富的符号世界模型、一个故障分类法和一个状态机执行器，为社区提供了一个更严格、更全面的LLM规划能力评估平台。
    3.  **揭示模型缺陷**：实验证明，即使是当前最强大的LLM，在需要跟踪隐式状态依赖的复杂规划任务中表现极差，其规划可靠性被严重高估。
    4.  **指明改进方向**：提出的“反事实预想模拟”策略被证明是缓解潜在故障的有效方法，为开发更鲁棒的LLM规划器提供了明确的技术路线。
- **对未来研究/应用的影响**：
    - **评估标准**：未来的LLM规划评估不应仅关注执行成功率，必须将潜在故障，特别是不可逆故障作为关键指标。
    - **模型设计**：研究应聚焦于如何增强LLM的“世界模型”推理能力，使其能够显式地模拟和跟踪动作的长期和隐式影响。
    - **安全部署**：在将LLM规划器部署到真实世界（如机器人、家庭助手）之前，必须通过类似SIMMER的基准进行严格测试，以确保其安全性。
- **局限与开放问题**：
    - 当前世界模型仅限于厨房领域，未来需要扩展到更多样化的场景。
    - 反事实预想模拟虽然有效，但增加了计算成本和推理时间。如何在不牺牲效率的前提下实现鲁棒的规划，是一个开放问题。
    - 对于推理能力较弱的模型，Self-Refine和反事实预想模拟的效果不佳，甚至可能适得其反。如何提升这些模型的底层推理能力是根本挑战。

## 5. 澄清与简化（Clarifications and Simplifications）

- **什么是“潜在故障”？**
    - **通俗解释**：想象一下，你正在按照一个菜谱做蛋糕。菜谱上写着“混合面粉、糖、鸡蛋”，你照做了。然后“放入烤箱烘烤”，烤箱也正常工作了。但是，蛋糕最终没有发起来。为什么？因为菜谱忘了写“加入发酵粉”。这个“忘记加发酵粉”就是**潜在故障**。在混合和烘烤的每一步，都没有任何错误提示（没有“无法混合”或“烤箱报错”），但最终结果是失败的。这种错误在发生时不声不响，直到最后才显现出来。
    - **类比**：就像写代码时的一个逻辑错误。程序不会崩溃（没有即时故障），但计算出的结果是错误的。这个错误在代码执行过程中是“潜伏”的。

- **为什么SIMMER的世界模型更“真实”？**
    - **类比**：之前的评估环境（如TextWorld）就像一个只有“积木”的儿童玩具世界。你可以拿起、放下、堆叠积木，但无法模拟“水”或“火”的效果。而SIMMER的世界模型更像一个“化学实验室”或“模拟城市”，它定义了更复杂的规则，比如“生肉（蛋白质）接触砧板 → 砧板被污染 → 其他食物接触砧板 → 也被污染”。这种“状态传播”是现实世界中常见但被之前模型忽略的。

- **“反事实预想模拟”是如何工作的？**
    - **通俗解释**：这就像让一个厨师在动手做菜前，先在脑子里“过一遍电影”。每做一个动作前，他都问自己：“如果我这样做，接下来会发生什么？会不会出问题？”
    - **具体操作**：在生成计划时，模型不是直接输出动作序列，而是被要求在每个动作前，先**预测**这个动作会如何改变当前的世界状态，并**自我检查**这个改变是否会导致任何潜在危害。如果预测到问题，就修改动作。这迫使模型进行显式的、逐步的因果推理，而不是凭直觉或模式匹配来生成计划。
