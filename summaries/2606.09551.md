## 0. 概览（Concise Summary）

- **主题（Topic）**：两服务器安全推理（Two-server Secure Inference），特别是针对大型语言模型（LLM）的隐私保护推理。
- **问题（Problem）**：在基于函数秘密共享（FSS）的安全推理系统中，每个非线性算子（如激活函数、归一化）都需要设计专门的、复杂的协议，导致系统脆弱、难以扩展，且成为性能瓶颈。
- **方法（Method）**：提出一个名为 FuseFSS 的编译器。它将每个标量定点算子抽象为一个统一的“算子规范”（Operator Specification），然后通过两个标准化的 FSS 原语（打包比较和向量区间查找）来编译执行，取代了为每个算子单独设计协议的做法。
- **创新（Innovation）**：核心创新在于用一个统一的编译流水线替代了传统的“每个算子一个协议”的工程范式。通过将算子分解为区间划分、低次多项式片段和谓词位，实现了协议设计的标准化和GPU友好化。
- **意义（Significance）**：显著降低了安全LLM推理的在线延迟和通信开销，同时简化了预处理过程，使得系统更高效、更易于扩展和维护。为构建实用化的隐私保护LLM服务提供了关键技术支撑。
- **一句话总结**：FuseFSS 通过一个统一的编译器，将安全推理中复杂的非线性算子协议简化为两个标准化的FSS调用，从而在保持精度的前提下，大幅提升了端到端性能和降低了系统复杂度。

## 1. 研究动机（Motivation）

在基于两服务器模型的安全推理中，线性层（如矩阵乘法）可以通过函数秘密共享（FSS）高效实现。然而，主要的性能瓶颈来自于**标量定点非线性算子**和**辅助操作**，例如：
- **非线性激活函数**：如 GELU、ReLU。
- **归一化操作**：如 LayerNorm 中的均值和方差计算。
- **定点算术辅助操作**：如截断（Truncation）、算术右移（ARS）、指数函数（nExp）、倒数（Reciprocal）等。

这些操作在模 $2^n$ 的环上执行，需要进行比较、分段选择、处理模数环绕（wrap-around）等问题。现有的高性能系统（如 Sigma）为每个这样的算子都设计了一个**专门的、定制的协议**。这种“每个算子一个协议”的方法存在以下问题：
1.  **脆弱性**：每个协议都需要单独进行正确性论证，容易引入与环绕或符号相关的细微错误。
2.  **重复劳动**：密钥生成逻辑、预处理材料等在不同算子间重复，增加了工程复杂性。
3.  **难以扩展**：添加一个新的算子需要从头设计、实现和验证一个完整的协议，阻碍了系统的灵活性和演进。

因此，**核心挑战**在于：如何设计一种通用的、统一的框架，能够高效、安全地处理所有常见的定点非线性算子，从而摆脱对特定算子协议的依赖，并提升整体性能。

## 2. 创新点（Innovation）

FuseFSS 的核心创新在于提出了一个 **“算子规范”（Operator Specification）** 和一个基于此规范的**统一编译流水线**。

1.  **统一的算子抽象（Operator Specification）**：
    - **创新点**：将任何标量定点算子（如 GELU, nExp, 截断等）抽象为一个包含三部分的规范：
        1.  **区间划分**：将输入值域 $[0, 2^n)$ 划分为多个区间。
        2.  **低次多项式**：在每个区间内，算子的算术输出由一个低次多项式近似。
        3.  **谓词位**：在每个区间内，算子的布尔输出（如溢出标志）由一组基本谓词（如比较）的逻辑组合定义。
    - **与现有方案的不同**：现有方案（如 Sigma）为每个算子（如 GELU）编写一个独立的协议，包含自定义的比较、环绕修正和预处理逻辑。FuseFSS 则用一个统一的“描述”来定义算子，将协议设计问题转化为一个编译问题。

2.  **统一的编译与执行流水线（Two-Call Evaluation）**：
    - **创新点**：给定一个“算子规范”和预处理掩码，FuseFSS 编译器将每个算子实例编译成**两个标准化的 FSS 调用**：
        1.  **打包比较（Packed Comparison）**：一次调用返回所有需要的谓词位的 XOR 分享。
        2.  **向量区间查找（Vector Interval Lookup）**：一次调用返回当前输入所在区间的多项式系数和常数的加法分享。
    - **与现有方案的不同**：现有方案中，每个算子需要多个不同的 FSS 原语调用和复杂的后处理。FuseFSS 将这个过程标准化，使得所有算子的在线执行路径几乎相同，极大地简化了系统并提升了GPU的并行效率。

3.  **掩码感知的编译（Mask-Aware Compilation）**：
    - **创新点**：编译器能够自动处理掩码带来的影响。例如，将原始输入 $x$ 上的谓词（如 $x < \beta$）重写为关于公开掩码值 $\hat{x} = x + r$ 的谓词，并引入秘密共享的进位位 $w$ 来修正掩码带来的偏移和环绕。
    - **为什么能奏效**：通过数学推导（引理4.3），证明了这种重写是正确且安全的。它确保了所有依赖于掩码的中间值（如进位位）都保持秘密共享状态，不会泄露关于输入 $x$ 或掩码 $r$ 的信息。

## 3. 主要内容（Main Content）【重点】

### 3.1 设计架构与方法

FuseFSS 的整体架构是一个编译流水线，其核心组件如下：

1.  **算子规范（Operator Specification）**：用户或系统开发者使用一个类型化的描述来定义一个新的标量定点算子。这个描述是公开的，不依赖于任何秘密数据。
2.  **编译器（Compiler）**：接收算子规范和预处理阶段生成的掩码 $r_{in}$，执行以下步骤：
    - **边界平移**：将算子规范中的区间边界 $\alpha_i$ 根据掩码 $r_{in}$ 平移到 $\hat{x}$ 空间，得到新的边界 $(\alpha_i + r_{in}) \mod 2^n$。
    - **谓词重写**：利用掩码重写引理（如引理4.3），将关于 $x$ 的谓词（如 $x < \beta$）转换为关于 $\hat{x}$ 的谓词（如 $\hat{x} < \theta$）和秘密共享的进位位 $w$ 的组合。
    - **实例化**：将所有重写后的比较操作打包成一个“打包比较”FSS实例；将所有区间对应的多项式系数和常数打包成一个“向量区间查找”FSS实例。通过填充（padding）确保这两个实例的公开形状（如比较次数、区间数量）与掩码无关，防止信息泄露。
3.  **在线执行（Online Evaluation）**：对于每个算子实例，执行以下步骤：
    - **公开掩码值**：双方打开 $\hat{x} = x + r_{in}$。
    - **执行FSS调用**：在 $\hat{x}$ 上执行打包比较和向量区间查找，分别得到谓词位的 XOR 分享和系数/常数的加法分享。
    - **本地后处理**：双方本地计算 $x = \hat{x} - r_{in}$ 的加法分享。然后，利用得到的系数和谓词位，通过 Beaver 三元组进行多项式求值（Horner方法），并通过布尔电路计算最终的布尔输出。

### 3.2 关键算法与数学推导

**核心引理：掩码下的无符号比较重写（Lemma 4.3）**

这是整个编译过程的理论基石。它展示了如何将关于秘密输入 $x$ 的比较 $I[x < \beta]$，转换为关于公开掩码值 $\hat{x}$ 的比较和秘密进位位 $w$ 的 XOR 组合。

设 $N = 2^n$，$r$ 是均匀随机的掩码，$\hat{x} = (x + r) \mod N$。对于阈值 $\beta \in \{0, ..., N\}$，定义：
- $\theta = (r + \beta) \mod N$
- $w = I[r + \beta \ge N]$ （进位位）

那么，对于所有 $x \in \mathbb{R}$，有：
$$I[x < \beta] = I[\hat{x} < \theta] \oplus I[\hat{x} < r] \oplus w$$

**证明思路**：该等式通过分析整数加法 $x + r$ 在模 $N$ 下的行为得出。关键在于考虑 $r + \beta$ 是否产生进位 $w$，以及 $x + r$ 是否小于 $r$。这个引理将秘密的比较 $I[x < \beta]$ 分解为两个关于公开值 $\hat{x}$ 的比较（$I[\hat{x} < \theta]$ 和 $I[\hat{x} < r]$）和一个秘密的进位位 $w$ 的 XOR。其中 $I[\hat{x} < r]$ 可以通过另一个类似的引理进一步处理，最终所有依赖于 $r$ 的部分都被转化为秘密共享的常数。

**推论：掩码下的区间指示器（A Useful Corollary）**

结合引理4.3和区间定义，可以得到区间指示器 $I[x \in I_i]$ 的表达式，其中 $I_i = [\alpha_i, \alpha_{i+1})$：
$$I[x \in I_i] = I[\hat{x} < (\alpha_{i+1} + r) \mod 2^n] \oplus I[\hat{x} < (\alpha_i + r) \mod 2^n] \oplus w_{i+1} \oplus w_i$$
其中 $w_t = I[r + \alpha_t \ge 2^n]$ 是预处理时计算的秘密进位位。这个公式巧妙地**消去了** $I[\hat{x} < r]$ 项，使得区间选择只需要对 $\hat{x}$ 进行一系列比较，以及使用秘密共享的进位位。

### 3.3 模型、训练与数据集

- **模型**：论文评估了 BERT（tiny, base, large）和 GPT（GPT-2, GPT-Neo）系列模型。这些是典型的 Transformer 架构的 LLM。
- **训练**：论文不涉及模型训练。它关注的是**推理**阶段，假设模型权重是公开的，而输入（prompt）和中间激活值是秘密的。
- **数据集**：论文没有使用特定的数据集来评估模型精度，而是通过端到端推理的延迟、通信量和预处理开销来衡量系统性能。精度方面，论文声称 FuseFSS 与基线系统 Sigma 相比，**保持了模型精度**（matches model accuracy），这意味着它没有引入额外的精度损失。

- **实验设置**：
    - **硬件**：2 × RTX PRO 6000 Blackwell GPU（每方一个），2 × EPYC 9654 CPU。
    - **基线**：Sigma，当前最先进的基于 FSS 的 GPU 安全推理系统。
    - **指标**：在线延迟（Online time）、在线通信量（Online Comm）、密钥生成时间（Keygen time）、密钥大小（Key size）。

- **主要结果（表1）**：
    - **端到端性能**：在所有测试模型上，FuseFSS 相比 Sigma 实现了 **1.24×–1.50× 的端到端加速**（在线延迟降低 19%–33%）。
    - **通信量**：在线通信量减少了 **9%–16%**。
    - **预处理成本**：密钥生成时间减少了 **14%–23%**，密钥大小减少了 **20%–24%**。

- **Softmax 子步骤分解（表3）**：FuseFSS 对 Softmax 中编译的路径（nExp + reciprocal）实现了 **4.5×–4.7× 的加速**，使得 Softmax 总时间减少了超过一半。

### 3.5 技术细节

- **类型化分享域（Typed Sharing Domains）**：系统使用两种基础类型：算术类型 $A_n$（值在 $\mathbb{Z}_{2^n}$ 中，加法分享）和比特类型 $B$（值在 $\{0,1\}$ 中，XOR 分享）。并提供了两者之间的转换协议（B2A, A2B）。
- **标准预处理子协议**：使用 Beaver 三元组实现环上的乘法，使用布尔 AND 关联性实现 XOR 分享的比特与操作。
- **后端接口**：编译器依赖于两个标准 FSS 原语：
    - **打包比较**：给定一个查询列表 $(k_t, c_t, \theta_t)$，返回 $I[\text{view}_{k_t, c_t}(u) < \theta_t]$ 的 XOR 分享。其中 $\text{view}_{k_t, c_t}(u) = ((u \mod 2^{k_t}) + c_t) \mod 2^{k_t}$。
    - **向量区间查找**：给定边界和载荷向量，返回 $u$ 所在区间的载荷向量的加法分享。
- **编译后处理电路**：在线阶段，在获得 FSS 输出后，还需要一个固定的后处理电路 $\Phi$。这个电路使用标准的预处理原语（Beaver 乘法、布尔 AND、B2A 转换）来完成最终的计算，例如多项式求值、布尔逻辑组合等。

## 4. 意义与影响（Significance and Impact）

- **解决的问题与贡献**：
    - **解决了安全推理中非线性算子的协议设计瓶颈**。FuseFSS 用一个统一的编译器替代了繁琐且易错的手工协议设计，极大地简化了系统开发和维护。
    - **显著提升了性能**。在端到端延迟、在线通信和预处理成本上均取得了可观的改进，使得基于 FSS 的 LLM 安全推理更加实用。
    - **提供了一个可扩展的框架**。通过定义新的“算子规范”，可以轻松地将新的非线性算子集成到系统中，而无需修改核心协议。

- **对未来研究/应用的影响**：
    - **推动安全LLM推理的实用化**：FuseFSS 的高效性使得在 GPU 上部署隐私保护的 LLM 服务更具可行性，特别是在对延迟敏感的实时应用场景中。
    - **启发新的系统设计范式**：其“编译而非设计”的思路可能被其他安全计算系统（如基于 HE 或 Garbled Circuit 的系统）借鉴，用于处理复杂的计算模块。
    - **为更复杂的算子提供基础**：该框架可以扩展到支持更复杂的、非标量的操作，或者用于安全训练中的反向传播。

- **作者提到的局限与开放问题**：
    - **向量归约操作**：FuseFSS 主要针对标量元素级算子。像 `max`、`sum` 这样的向量归约操作不在其“算子规范”IR 的范围内，需要由标准的 MPC 子协议处理。作者指出，优化这些向量归约是未来与 FuseFSS 互补的工作方向。
    - **更强的安全模型**：当前工作基于半诚实（semi-honest）安全模型。将 FuseFSS 扩展到恶意（malicious）安全模型是一个重要的未来方向。

## 5. 澄清与简化（Clarifications and Simplifications）

- **什么是“函数秘密共享（FSS）”？**
    - **通俗解释**：想象有两个服务器，它们各自持有一把“钥匙”的一部分。当它们看到一个公开的输入（比如一个数字 $\hat{x}$）时，它们可以各自用自己的一半钥匙，独立地计算出一个“碎片”。这两个碎片合在一起，才能得到真正的结果（比如“这个数字是否大于10？”的答案）。FSS 保证，单个服务器只看自己的碎片，完全无法推断出任何关于输入或结果的信息。FuseFSS 就是利用这种技术来高效地执行比较和查找操作。

- **“算子规范”是什么？**
    - **类比**：可以把一个复杂的函数（比如 GELU）想象成一个分段函数。它的图像可以分成几段，每一段都是一条简单的直线或抛物线。“算子规范”就是对这个分段函数的精确描述：它告诉你分段的边界在哪里，每一段是哪种直线/抛物线，以及每一段上是否有一些额外的标志位（比如是否溢出）。FuseFSS 的编译器就是读取这个“描述”，然后自动生成在两台服务器上安全执行这个分段函数的代码。

- **“打包比较”和“向量区间查找”如何工作？**
    - **打包比较**：想象你要同时检查一个数字是否满足多个条件，比如“是否小于A？”、“是否小于B？”、“是否小于C？”。传统方法需要逐个检查。FuseFSS 的“打包比较”就像一个高效的批处理机，一次调用就能同时得到所有比较结果的秘密碎片。
    - **向量区间查找**：想象你有一个表格，里面列出了每个区间对应的系数（比如区间1对应系数a1,b1；区间2对应系数a2,b2...）。你输入一个数字，需要找到它属于哪个区间，并取出对应的系数。FuseFSS 的“向量区间查找”就像一个秘密的“翻牌”游戏，它不会告诉你翻的是哪张牌（哪个区间），但会秘密地给你那张牌上的数字（系数）。
