## 0. 概览（Concise Summary）

- **主题（Topic）**：本文属于AI for Code Security领域，具体任务是评估大型语言模型（LLM）在安全多方计算（MPC）软件上的代码修复能力。
- **问题（Problem）**：现有通用的代码修复基准（如SWE-bench）无法直接用于评估MPC软件，原因有三：1）MPC仓库中大部分是通用Python代码而非密码学逻辑；2）MPC修复任务常缺乏标准化的测试用例；3）仅通过功能测试（fail-to-pass）无法检测密码学安全性和数值精度问题。因此，缺乏一个专门针对MPC的、仓库级别的LLM代码修复基准。
- **方法（Method）**：提出了一个名为**MPC-Patch-Bench**的基准，包含两个核心框架：1）**数据策展框架**，通过一个领域特定的策展代理和一个人机协作完成引擎，从原始PR中筛选并构建出205个经过验证的MPC修复实例；2）**MPC验证器**，通过动态差分测试和领域特定的静态分析，对LLM生成的补丁进行功能、安全性和数值保真度的全面评估。
- **创新（Innovation）**：1）首次提出了专门针对MPC的仓库级LLM代码修复基准；2）设计了结合领域知识和人机协作的数据策展流程，解决了MPC数据稀缺和测试不完整的问题；3）提出了一个双流验证器，将功能正确性与密码学安全性和数值保真度解耦，揭示了仅凭功能测试会严重高估LLM在安全敏感软件中的可靠性。
- **意义（Significance）**：该工作不仅为MPC领域提供了一个严格的评估标准，更重要的是，它揭示了一个普遍性问题：在安全关键领域，功能正确的代码修复基准会系统性高估LLM的可靠性。这为未来在安全敏感软件中评估LLM提供了方法论模板。
- **一句话总结**：本文构建了首个面向安全多方计算的仓库级LLM代码修复基准MPC-Patch-Bench，通过引入领域特定的数据策展和双流安全验证器，揭示了功能正确性远非评估LLM在隐私保护软件中可靠性的充分条件。

## 1. 研究动机（Motivation）

- **解决的问题**：当前缺乏一个能够有效评估LLM在真实世界MPC软件上进行代码修复能力的基准。
- **为什么重要**：MPC技术正被广泛应用于隐私保护机器学习、生物医学协作和安全分析等关键领域。这些领域的软件一旦出现安全漏洞，后果极其严重。因此，利用LLM辅助MPC代码修复具有巨大的潜力，但前提是必须有一个可靠的评估方法。
- **挑战在哪里**：直接将通用基准（如SWE-bench）移植到MPC领域会遭遇三个结构性失败：
    1.  **数据噪声大**：MPC仓库中的大部分Pull Request（PR）是关于通用Python基础设施（如CI配置、API封装）的修改，而非核心密码学逻辑。直接使用这些PR会引入大量无关噪声。
    2.  **测试缺失**：许多高质量的MPC修复PR没有附带标准化的Fail-to-Pass（F2P）和Pass-to-Pass（P2P）测试，导致严格的SWE-bench提取流程只能保留极少数实例（从7305个PR中仅得到42个）。
    3.  **评估维度单一**：MPC代码的“正确性”不仅包括功能正确，还必须满足密码学安全（如不泄露秘密）和数值保真度（如有限环上的定点数运算不产生漂移）。标准的F2P测试完全无法检测这些安全性和数值性问题。

## 2. 创新点（Innovation）

- **关键创新**：提出了**MPC-Patch-Bench**，一个全新的、专门为评估LLM在MPC软件上的代码修复能力而设计的仓库级基准。
- **新方法与技术**：
    1.  **数据策展框架**：
        - **领域特定策展代理**：设计了一个三层启发式过滤机制（原始密码学原语检查、协议逻辑检查、领域特异性检查），从海量PR中筛选出真正涉及MPC推理的候选任务。
        - **人机协作完成引擎**：针对缺乏测试的PR，引入人类专家与AI代理协作，共同合成问题描述、F2P/P2P测试和黄金补丁，从而“拯救”了这些高价值但“不完整”的实例，将可用实例从42个大幅提升至205个。
    2.  **MPC验证器**：
        - **双流验证架构**：将评估解耦为功能测试、动态测试和静态分析三个维度。
        - **动态测试**：通过将加密输出与明文预言机进行差分比较，检测数值精度漂移。
        - **领域特定静态分析（SAST）**：设计了一套针对MPC的静态分析规则，自动检测不安全揭示、非法类型转换、非密码学随机数等安全反模式。
- **与现有方案的不同**：与SWE-bench等通用基准不同，MPC-Patch-Bench不仅关注功能正确性，更将密码学安全性和数值保真度作为核心评估指标。与SPDZCoder等MPC代码生成工具不同，它专注于评估LLM在真实仓库上下文中的修复能力，而非单函数或单框架的代码生成。
- **为什么能奏效**：数据策展框架通过领域知识过滤噪声，并通过人机协作解决了数据稀缺问题，确保了基准的质量和规模。双流验证器则从执行和源码两个层面，全面覆盖了MPC代码特有的失败模式，使得评估结果更真实、更可靠。

## 3. 主要内容（Main Content）【重点】

### 3.1 设计架构与方法

MPC-Patch-Bench的整体架构如图1所示，由两个核心框架组成：

1.  **数据策展框架**：负责从原始GitHub PR中构建高质量的基准实例。其流程如图2所示：
    - **输入**：从5个主流MPC框架（CrypTen, tf-encrypted, MP-SPDZ, SecretFlow, PySyft）爬取的7305个原始PR。
    - **领域特定策展代理**：使用一个AI代理，通过三层启发式规则（原始密码学原语、协议逻辑、领域特异性）对每个PR进行评分。评分高于0.9的自动接受，介于0.75和0.9之间的交由人工审核，低于0.75的丢弃。此步骤将候选实例从7305个缩减至1175个。
    - **人机协作完成引擎**：对于1175个候选实例，人类专家与AI合成代理协作，为那些缺少问题描述和测试的实例补全这些关键组件，最终生成205个完全可执行且经过验证的实例。

2.  **MPC验证器**：负责对LLM生成的补丁进行全面评估。其架构如图3所示：
    - **功能测试**：首先执行标准的F2P和P2P测试，判断补丁是否功能正确。
    - **动态测试**：对通过功能测试的补丁，执行MPC协议，并将其输出与一个数学上的明文预言机进行比较，以检测数值精度问题。
    - **领域特定SAST**：对补丁源码进行静态扫描，检测不安全揭示、非法类型转换、非密码学随机数等安全反模式。
    - **最终决策**：只有同时通过功能测试、动态测试和静态分析的补丁，才被标记为“验证通过”。

### 3.2 关键算法与数学推导

论文的核心算法主要体现在数据策展代理的自动路由逻辑中，其伪代码如Algorithm 1所示。

**Algorithm 1: Domain-Specific Curation Agent**

**输入**：未过滤的候选集 $C_{raw}$，AI代理 $A_{agent}$，固定的MPC特定启发式规则 $H_{mpc}$，置信度阈值 $\tau_{high}$ 和 $\tau_{med}$（当前设为 $\tau_{high} = 0.9$ 和 $\tau_{med} = 0.75$）。

**输出**：潜在MPC任务集 $C_{potential}$，人工审核池 $C_{review}$。

1.  初始化 $C_{potential} \leftarrow \emptyset$, $C_{review} \leftarrow \emptyset$
2.  **对于** 每个候选 $c \in C_{raw}$ **执行**：
3.    $diff, desc \leftarrow ExtractData(c)$
4.    $prompt \leftarrow ConstructPrompt(H_{mpc}, diff, desc)$  // 构建包含三层评估的提示
5.    $layer\_outputs, conf \leftarrow A_{agent}.Evaluate(prompt)$  // AI代理评估并返回置信度
6.    $c.metadata \leftarrow layer\_outputs$  // 保存分析结果
7.    **如果** $conf \ge \tau_{high}$ **则**：
8.      $C_{potential} \leftarrow C_{potential} \cup \{c\}$  // 自动接受
9.    **否则如果** $\tau_{med} \le conf < \tau_{high}$ **则**：
10.     $C_{review} \leftarrow C_{review} \cup \{c\}$  // 标记为人工审核
11.   **否则**：
12.     丢弃 $c$ 作为噪声
13.   **结束如果**
14. **结束对于**
15. **返回** $C_{potential}$, $C_{review}$

**数学推导**：论文未提出新的数学公式，但其验证器依赖于MPC的数学基础。例如，动态测试中的“精度检测”涉及对定点数截断误差的监控，其容忍度（atol）被设置为一个极小的值（如0.005），这背后是有限环上算术运算的误差分析。静态分析中的“不安全揭示”规则，其数学基础是确保秘密共享值在协议结束前不被泄露，这直接关系到MPC的安全性定义。

### 3.3 模型、训练与数据集

- **数据集**：MPC-Patch-Bench数据集包含**205个**完全验证且可执行的实例。这些实例来自5个MPC库，分布如图4所示：CrypTen (29.8%)，tf-encrypted (21.0%)，MP-SPDZ (20.0%)，PySyft (16.6%)，SecretFlow (12.7%)。表2展示了实例的统计信息，如平均问题描述长度、补丁大小、测试数量等。图5展示了补丁大小、F2P测试数量和问题描述长度的累积分布函数（CDF），证明了数据集的多样性和复杂性。
- **模型**：评估了8个当代LLM，包括Anthropic的Claude系列（Sonnet 4.6, Opus 4.6, Haiku 4.5）、OpenAI的GPT系列（GPT-5.4, GPT-4.1, GPT-4o-mini）和Google的Gemini系列（Gemini 2.5 Pro, Gemini 2.5 Flash-Lite）。
- **训练**：论文不涉及模型训练，所有模型均以零样本（zero-shot）方式被查询，温度为0（贪婪解码）。

### 3.4 实验设置与结果

- **实验设置**：
    - **提示**：每个模型接收一个结构化的提示，包含问题描述、相关源码文件和输出格式指令。
    - **执行**：生成的补丁被应用到仓库的基提交上，并使用库特定的测试运行器执行（如pytest或自定义编译仿真管道）。功能解决的标准是所有F2P测试通过且所有P2P测试继续通过。
    - **验证**：功能解决的补丁还需通过MPC验证器的动态测试和静态分析，才能被标记为“验证解决”。

- **实验结果**：
    1.  **功能解决率**（图6）：最强的模型（Sonnet 4.6）仅解决了22.9%的任务。不同库的难度差异巨大，CrypTen最易，MP-SPDZ最难（最高仅解决2/41个实例）。这表明MPC修复对LLM来说是一个极具挑战性的任务。
    2.  **验证影响**（图7）：引入MPC验证器后，模型排名发生显著变化。Opus 4.6在验证后排名反超Sonnet 4.6。高达40%的功能通过补丁被验证器拒绝（如GPT-5.4），其中大部分是由SAST标记的安全问题。这有力地证明了功能正确性会严重高估LLM在MPC领域的可靠性。

### 3.5 技术细节

- **数据策展**：三层启发式规则的具体内容在3.2.1节有详细描述，包括检查是否操作了秘密共享、OT等原语，是否涉及A2B转换等协议逻辑，以及是否处理了数据无关性、定点数截断等MPC特有的工程约束。
- **人机协作**：人类专家的角色是提供领域直觉并验证密码学正确性，而AI代理负责结构化生成缺失的程序组件（如测试代码）。
- **MPC验证器**：
    - **动态测试**：采用MPCDiff风格的差分测试，将加密输出与数学明文预言机比较。还通过注入边界值（如0， -1， MAX_INT）来探测代数边界情况。
    - **领域特定SAST**：使用Semgrep等工具，编写了针对MPC的YAML规则。例如，一个规则会标记在循环中调用`reveal()`或`get_plain_text()`的行为，因为这可能导致秘密泄露。附录F提供了一个具体的规则示例和标记的补丁。
- **数据污染分析**：论文在附录D中讨论了数据污染问题，指出其任务特性（需要理解特定仓库状态）和低解决率（远低于纯记忆所能达到的水平）表明，记忆不是模型表现的主要驱动力。

- **解决了什么问题**：解决了MPC领域缺乏仓库级LLM代码修复评估基准的空白。它提供了一个标准化的、可复现的评估平台，能够全面衡量LLM在功能、安全性和数值精度三个维度的表现。
- **贡献是什么**：
    1.  **首个MPC仓库级基准**：为MPC社区提供了一个宝贵的资源。
    2.  **创新的数据策展方法**：其领域特定策展和人机协作完成引擎为其他专业领域构建基准提供了可复用的方法论。
    3.  **双流验证框架**：揭示了功能测试的局限性，并提供了一个更全面的评估范式，对安全关键软件领域具有普遍指导意义。
- **对未来研究/应用的影响**：
    - 激励研究者开发更安全、更可靠的LLM代码修复模型，尤其是在隐私保护和安全计算领域。
    - 其方法论可以推广到其他安全敏感领域（如密码学、区块链、形式化验证），推动“安全感知”的代码生成和修复评估。
    - 为LLM在MPC领域的实际应用（如辅助开发者修复漏洞）提供了更可靠的评估基础。
- **局限与开放问题**：
    - **覆盖范围有限**：当前仅覆盖5个MPC框架，未来需要扩展到更多框架（如全同态加密、零知识证明环境）。
    - **数据污染**：尽管有分析，但无法完全避免。作者建议社区使用其方法论在更新的数据上重新运行，以控制污染影响。
    - **验证器完备性**：SAST规则和动态测试可能无法覆盖所有MPC安全漏洞，未来需要更全面的验证方法。

## 5. 澄清与简化（Clarifications and Simplifications）

- **MPC是什么？**：可以想象成几个互不信任的人想一起算一笔账，但每个人都不想让别人看到自己的账本。MPC就是一种密码学技术，允许他们通过一种特殊的“加密计算”方式，在不泄露各自原始数据的情况下，共同得到最终结果。
- **为什么通用基准不行？**：通用基准（如SWE-bench）就像是一个“通用驾照考试”，只考你开车能不能从A点到B点。但MPC代码修复更像是在开一辆“装甲运钞车”，不仅要能开到目的地，还要确保车上的钱（秘密数据）在任何情况下都不会被抢走（泄露），并且计算出的金额（数值）必须分毫不差。通用考试完全不考这些安全性和精度要求。
- **数据策展框架在做什么？**：想象一下，要从一个巨大的图书馆（GitHub）里找出所有关于“如何修理装甲运钞车”的维修手册。首先，一个“图书管理员AI”会快速浏览所有书名，只留下那些看起来跟“装甲”、“运钞”、“安全锁”有关的书（三层过滤）。然后，对于那些内容不完整、缺少关键章节（如测试步骤）的书，一个“人类专家”会和“AI助手”一起，根据书中的线索，把缺失的章节补全，最终形成一本本完整的、可操作的维修手册（205个实例）。
- **MPC验证器在做什么？**：它就像一个“双重安全检查员”。
    - **动态测试**：检查员会亲自开一下修好的车，看它能不能正常行驶（功能测试）。然后，他会把车开到一个“标准测试场”（明文预言机），比较车上的仪表盘读数（加密输出）和标准答案是否一致，看有没有因为改装（MPC协议）导致的误差（数值精度）。
    - **静态分析**：另一个检查员会仔细阅读维修手册（源码），看有没有违反安全规定的操作，比如“在行驶中打开保险柜门”（不安全揭示）、“用普通螺丝代替防爆螺丝”（非密码学随机数）等。
