## 0. 概览（Concise Summary）

- **主题（Topic）**：大型语言模型（LLM）智能体的技能文档自动生成与优化。
- **问题（Problem）**：LLM 自动编写的技能文档（skill documents）质量低下，无法有效提升智能体性能，而现有评估方法缺乏可操作的诊断反馈，导致技能改进困难。
- **方法（Method）**：提出 **SKILLAXE** 框架，一种完全无监督的评估引导式自我精炼方法。它通过对比智能体在有/无技能文档时的行为，从四个可解释维度（质量影响、触发精度、指令遵循与故障归因、解决方案路径覆盖）进行诊断，并生成结构化改进简报，指导 LLM 重写技能。
- **创新（Innovation）**：首次将技能失败模式分解为多个可诊断的维度，并利用技能文档本身隐含的约束作为评估标准，无需任何人工标注、测试用例或环境奖励信号，实现了对 LLM 自写技能的自动化、可解释性改进。
- **意义（Significance）**：显著缩小了 LLM 自写技能与人类编写技能之间的性能差距（在 SkillsBench 上缩小了 47-67%），并验证了其在生产环境中作为持续学习引擎的潜力，降低了技能工程对人工专家的依赖。
- **一句话总结**：SKILLAXE 通过一种无监督的、多维度的行为对比诊断方法，让 LLM 能够像人类专家一样，系统地诊断并改进自己编写的智能体技能文档。

## 1. 研究动机（Motivation）

- **核心问题**：LLM 智能体严重依赖“技能文档”（结构化自然语言指令）来提升性能，但 LLM 自身编写的技能文档质量很差。在 SkillsBench 基准测试中，人类编写的技能能将任务通过率提升 16.2 个百分点，而 LLM 编写的技能则几乎没有带来任何可衡量的提升。
- **为什么重要**：技能文档的编写和维护目前主要依赖人工，过程繁琐、经验化且不透明。如果 LLM 无法自主编写和优化技能，将严重制约智能体系统的可扩展性和自动化水平，尤其是在需要大量、多样化技能的复杂生产环境中。
- **挑战在哪里**：
    1.  **缺乏可操作的诊断反馈**：现有的评估方法（如任务级通过/失败信号）过于粗糙，无法告诉开发者或 LLM 技能失败的具体原因（例如，是技能本身指令错误？触发时机不对？还是智能体没有遵循指令？）。
    2.  **失败模式复杂且隐蔽**：LLM 编写的技能在语法上通常很流畅，但失败模式是系统性的，例如：在错误的任务上被触发、指令与任务约束冲突、或提供看似合理但实际会误导智能体的建议。
    3.  **需要无监督方法**：在实际应用中，为每个任务编写测试用例或设计奖励函数是不现实的。因此，需要一个完全无监督的、仅依赖任务指令和智能体行为就能进行诊断和优化的框架。

## 2. 创新点（Innovation）

- **核心创新**：提出了 **SKILLAXE** 框架，这是一个**评估引导的自我精炼**框架，用于自动诊断和改进 LLM 智能体的技能文档。
- **与现有方案的不同**：
    - **从“结果评估”到“过程诊断”**：现有工作（如 Reflexion, Self-Refine）主要关注改进单次执行或模型行为，而 SKILLAXE 专注于改进**可复用的技能文档**。它不满足于知道任务失败，而是要找出技能文档中导致失败的具体原因。
    - **多维度的故障归因**：SKILLAXE 将技能质量分解为四个可解释的维度（质量影响、触发精度、指令遵循与故障归因、解决方案路径覆盖），每个维度对应一种特定的失败模式，提供了比单一标量分数丰富得多的诊断信息。
    - **完全无监督**：SKILLAXE 的诊断过程**不需要任何人工标注、预定义的测试用例、或环境奖励函数**。它通过对比智能体在有/无技能文档时的行为差异，并利用技能文档本身隐含的规则作为评估标准，实现了零监督信号的自我改进。
- **为什么能奏效**：关键洞察在于，技能文档本身隐式地定义了可评估的行为约束。通过将技能的规则视为评估标准，并对比有/无技能时的输出，SKILLAXE 能够产生可操作的反馈，从而指导 LLM 进行有针对性的重写，而不是盲目地试错。

## 3. 主要内容（Main Content）【重点】

### 3.1 设计架构与方法

SKILLAXE 框架是一个迭代循环，包含四个阶段（如图 1 所示）：

1.  **阶段 1：行为对比（Phase 1: Paired Execution）**：对于每个任务，智能体分别在**没有技能**（基线响应 $r_0$）和**注入当前技能**（技能响应 $r_s$）的情况下执行。通过对比 $r_0$ 和 $r_s$ 来评估技能的影响。
2.  **阶段 2：多维诊断（Phase 2: Diagnostic Evaluation）**：从四个维度对技能进行诊断，生成一个结构化的**改进简报（Improvement Brief）**。
3.  **阶段 3：结构化重写（Phase 3: Structured Rewriting）**：一个 LLM 重写器（LLM refiner）根据改进简报中的诊断信息，对技能文档进行有针对性的重写。
4.  **阶段 4：迭代收敛（Phase 4: Iteration）**：重复上述过程，直到技能性能收敛或达到预设的迭代次数。

### 3.2 关键算法与数学推导

SKILLAXE 的核心在于四个诊断维度的计算方法：

**1. 质量影响（Quality Impact）**
- **目的**：判断技能是否整体上帮助或损害了智能体性能。
- **方法**：通过两阶段对比评估。
    - **阶段 1：偏好方向**：LLM 判断器比较 $r_s$ 和 $r_0$，决定哪个更好，得到方向 $d \in \{-1, 0, +1\}$。
    - **阶段 2：改进幅度**：LLM 判断器估计改进的强度 $m \in [0, 1]$。
    - **公式**：$$Quality(s, \tau) = d \cdot m \in [-1, 1]$$
    - **作用**：作为外层循环信号，决定是否需要对技能进行深入诊断。

**2. 触发精度（Trigger Precision）**
- **目的**：诊断技能是否在正确的任务上被激活，避免误触发或漏触发。
- **方法**：在语义检索空间中评估触发词的质量。
    - 从技能文档中提取**正触发短语** $P^+$（应激活）和**负触发短语** $P^-$（不应激活）。
    - 为每个短语生成**上下文感知向量**，融合短语本身的语义和其在句子中的上下文逻辑。
    - 计算三个几何诊断指标：
        - **覆盖广度（Coverage Breadth）**：衡量正触发短语的多样性，避免触发范围过窄。
          $$CoverageBreadth = \frac{1}{|P^+|(|P^+|-1)} \sum_{i \neq j} \left(1 - \cos(e_i^+, e_j^+)\right)$$
        - **负特异性（Negative Specificity）**：衡量负触发短语与正触发短语的分离程度，避免误触发。
          $$NegativeSpecificity = \frac{1}{|P^-|} \sum_{e^- \in P^-} \min_{e^+ \in P^+} \left(1 - \cos(e^-, e^+)\right)$$
        - **边界锐度（Boundary Sharpness）**：衡量正负触发短语之间的最小距离，暴露决策边界上的模糊性。
          $$BoundarySharpness = \min_{e^- \in P^-, e^+ \in P^+} \left(1 - \cos(e^-, e^+)\right)$$

**3. 指令遵循与故障归因（Instruction Compliance with Fault Attribution）**
- **目的**：区分失败是由于智能体没有遵循指令（智能体错误），还是技能指令本身质量差（技能错误）。
- **方法**：将技能分解为可评估的规则，并对每条规则进行评估。
    - 从技能 $s$ 中提取规则集 $\{R_1, ..., R_n\}$ 和严重性权重 $w_i \in \{1,2,3\}$。
    - 对每条规则 $R_i$，LLM 评估器预测：
        - 相关性 $\rho_i \in \{0,1\}$
        - 遵循度 $a_i \in [0,1]$
        - 规则质量 $g_i \in [0,1]$
        - 技能故障 $f_i \in [0,1]$
    - **原始指令遵循度**：$$Compliance(s, \tau) = \frac{\sum_{i:\rho_i=1} w_i g_i a_i}{\sum_{i:\rho_i=1} w_i}$$
    - **故障调整后的技能信用分**：$$c_i = a_i + (1 - a_i)(1 - f_i)$$
      $$SkillScore(s, \tau) = \frac{\sum_{i:\rho_i=1} w_i g_i c_i}{\sum_{i:\rho_i=1} w_i}$$
    - **核心思想**：当 $f_i$ 高时（即失败归因于技能），$c_i$ 低，SkillScore 低，提示需要重写规则。反之，如果 $f_i$ 低（失败归因于智能体），$c_i$ 会部分恢复，保护了正确的规则不被修改。表 1 清晰地展示了这一区别。

**4. 解决方案路径覆盖（Solution-Path Coverage）**
- **目的**：诊断技能是否只支持了智能体可能采取的多种有效解决方案中的一小部分。
- **方法**：
    - 对于任务 $\tau$，LLM 枚举出 $k$ 个合理的解决方案路径 $\{P_1, ..., P_k\}$。
    - 将技能 $s$ 分割成多个块（chunks）。
    - 计算每个路径与技能块的最大语义相似度，然后取平均。
    - **公式**：$$PathCoverage(s, \tau) = \frac{1}{k} \sum_{j=1}^k \max_{c \in chunks(s)} \cos(e(P_j), e(c))$$
    - **作用**：确保技能提供的指导覆盖了智能体可能探索的各种有效策略，而不是只针对单一路径。

### 3.3 模型、训练与数据集

- **模型**：使用 **Claude Opus 4.5** 作为智能体骨干网络（agent backbone）和 LLM 判断器/重写器。SKILLAXE 框架本身不训练模型，而是利用 LLM 的推理和生成能力。
- **训练**：SKILLAXE 是一个**无训练**的框架。它不涉及对 LLM 本身的微调，而是通过提示（prompting）和迭代优化技能文档的内容。
- **数据集**：
    1.  **SkillsBench**：用于评估任务级技能改进。包含 89 个任务，191 个技能，688 次智能体运行。任务是多样化的，包括软件工程、数据科学等。
    2.  **SpreadsheetBench**：用于评估持续库改进。包含 912 个真实的电子表格操作任务。论文使用 200 个任务作为训练集（用于构建技能库），50 个作为测试集。

### 3.4 实验设置与结果

- **实验 1：SkillsBench 任务级改进**
    - **设置**：对比四种条件：无技能、LLM 自写技能、SKILLAXE 改进后的技能、人类编写的技能。
    - **评估**：使用两种评估协议：1）**多模态公平评分器**（Fair Grader），一个不依赖测试用例的 LLM 评判者；2）**SkillsBench 原生验证器**（Native Verifier），基于 pytest 的严格测试。
    - **主要结果（图 3，表 2）**：
        - SKILLAXE 将整体通过率从 LLM 自写技能的 32.5% 提升至 41.6%（公平评分器下从 39.0% 提升至 54.5%），相对提升 28%。
        - SKILLAXE 缩小了与人类技能之间 47-67% 的差距。
        - **关键发现**：性能提升主要来自**执行可靠性（Coverage）** 的大幅提升（从 49.4% 到 72.7%），而非答案质量（Quality）的提升。这表明技能主要提供了防止智能体崩溃的“执行知识”。

- **实验 2：SpreadsheetBench 持续库改进**
    - **设置**：模拟生产环境，智能体（Excel Copilot）从过去的任务轨迹中学习，构建并持续改进技能库。
    - **对比条件**：无技能库、LLM 自写技能库（69 个技能）、SKILLAXE 技能库（22 个技能）。
    - **主要结果（表 3）**：
        - 两个技能库都将通过率从 16.0% 大幅提升至 52.0%。
        - SKILLAXE 库**仅用 22 个技能就达到了与 69 个技能的 LLM 自写库相同的准确率**，技能数量减少了 68%。
        - SKILLAXE 技能的**激活率更高**（35.8% vs 20.0%），说明其描述更清晰、更易于智能体识别和使用。

### 3.5 技术细节

- **触发短语提取**：使用带否定追踪的依存句法分析（dependency parsing with negation tracking）从技能文档中提取正负触发短语。
- **上下文感知向量**：通过混合短语本身的嵌入和其所在句子的上下文嵌入来生成，以区分同一短语在不同语境下的含义（例如，“spreadsheet”在激活和排除语境中）。
- **故障归因**：评估基于文件级别的证据（如公式、格式代码、结构差异），而不是智能体的文本声明，以确保评估的客观性。
- **技能库路由**：在 SpreadsheetBench 实验中，使用基于嵌入的 3 区域路由（FORK/IMPROVE/SKIP）来决定新任务是创建新技能、改进现有技能还是跳过。

## 4. 意义与影响（Significance and Impact）

- **解决了什么问题**：
    - 解决了 LLM 智能体技能工程中**缺乏可操作诊断反馈**的核心瓶颈。
    - 证明了 LLM 可以**自主、系统地诊断和改进自己编写的技能**，无需人工干预。
    - 揭示了技能提升性能的主要机制是**提供执行知识**，从而提高任务的完成率，而非提升答案质量。

- **对未来研究/应用的影响**：
    - **自动化技能工程**：为构建能够自我进化的智能体系统铺平了道路，极大地降低了技能开发和维护的人力成本。
    - **生产级智能体部署**：SKILLAXE 的持续学习能力使其非常适合在生产环境中部署，能够从每次失败中学习，不断优化技能库，提升系统整体鲁棒性。
    - **可解释的 AI 系统**：通过将失败归因到具体的技能维度，SKILLAXE 提供了对智能体行为更深入的理解，增强了系统的可解释性和可调试性。

- **作者提到的局限与开放问题**：
    - **结构错位**：无法检测技能本身教授了根本错误的策略，但内部自洽的情况。
    - **技能交互**：未考虑多个技能同时激活时可能产生的冲突或交互影响。
    - **LLM 判断器偏差**：依赖 LLM 作为评判者，可能存在位置或校准偏差。
    - **与人类技能的差距**：虽然大幅缩小了差距，但仍有剩余，可能需要更强的在线执行反馈和多轮迭代。

## 5. 澄清与简化（Clarifications and Simplifications）

- **技能文档是什么？**：可以把它想象成一个给 AI 智能体的“**操作手册**”或“**最佳实践指南**”。比如，一个处理 PDF 的技能，手册里会写：“第一步，用 `pdfplumber` 库打开文件；第二步，如果遇到扫描件，先调用 OCR 服务；第三步，提取文本后，注意清理特殊字符...”。这个手册不是代码，而是自然语言写的指导。

- **SKILLAXE 的核心思想是什么？**：就像一个教练在观察运动员（智能体）比赛。教练让运动员**先不按手册做一次**，再**按手册做一次**。通过对比两次的表现，教练就能诊断出手册的问题：
    - **质量影响**：按手册做，成绩是变好了还是变差了？
    - **触发精度**：这个手册是不是在不该用的时候也被拿出来用了？（比如，处理图片的手册被用在了处理 PDF 上）
    - **指令遵循与故障归因**：运动员没做好，是因为他没看懂手册（运动员的错），还是手册本身写得太模糊、有误导性（手册的错）？比如，手册说“用黄色背景”，但没说是哪种黄，导致运动员用了浅黄，这就是手册的错。
    - **解决方案路径覆盖**：手册是不是只教了一种方法？比如，处理表格，手册只教了用 Python 的 `pandas`，但运动员可能更擅长用 Excel 的公式。如果手册没有覆盖后一种方法，那它就不是一个好手册。

- **为什么“无监督”很重要？**：想象一下，你要为一个能处理上万种不同任务的 AI 写测试用例，这几乎是不可能的。SKILLAXE 不需要这些，它只需要任务本身和 AI 的输出，就能自己“悟”出问题所在，这就像让 AI 自己当自己的老师。
