## 0. 概览（Concise Summary）

- **主题（Topic）**：大语言模型（LLM）剪枝的推理加速基准测试。
- **问题（Problem）**：现有的LLM剪枝方法种类繁多，其实际推理加速效果严重依赖于硬件和内核实现，导致基于理论指标（如FLOPs）的评估无法反映真实加速比。缺乏一个统一的、公平的框架来比较不同剪枝方法在实际部署中的性能。
- **方法（Method）**：提出一个以通用矩阵乘法（GEMM）为中心的剪枝分类法（GEMM-centric taxonomy），将所有剪枝方法统一映射到GEMM的M、N、K三个逻辑维度上。基于此分类法，构建了一个统一的基准测试框架，实现了不同剪枝方法在实现一致性下的公平比较。
- **创新（Innovation）**：
    1.  **GEMM-centric分类法**：首次从GEMM操作维度（M, N, K）统一理解LLM剪枝，揭示了不同剪枝方法在计算传播和实现行为上的本质差异。
    2.  **统一基准框架**：构建了一个硬件无关的、可复用的基准测试框架，为每种剪枝分类提供了标准化的加速实现，使得跨方法比较成为可能。
    3.  **关键洞察**：通过系统实验，首次绘制了当前剪枝加速的Pareto前沿：静态深度剪枝（Static M）在低性能损失下最优；动态深度剪枝（Dynamic M）在中等损失下领先；静态宽度剪枝（Static NK）在高损失下才具有竞争力。
- **意义（Significance）**：该工作揭示了名义稀疏度（nominal sparsity）是实际加速的弱预测因子，强调了剪枝分类法、结构传播和系统开销对真实加速比的决定性作用。为未来的LLM剪枝研究提供了统一的理论视角和实践指导，指明了混合剪枝设计是未来发展方向。
- **一句话总结**：本文通过提出一个以GEMM维度为核心的剪枝分类法，构建了统一的基准测试框架，系统性地揭示了不同LLM剪枝方法在实际推理加速中的真实性能表现和Pareto前沿，证明了名义稀疏度并非实际加速的有效指标。

## 1. 研究动机（Motivation）

- **解决的问题**：LLM剪枝领域存在严重的评估碎片化问题。各种剪枝方法（如深度剪枝、宽度剪枝、动态剪枝等）虽然目标相同，但它们在模型上产生的稀疏模式、计算流改变和硬件行为（如内存访问、内核调度效率）截然不同。因此，实际加速比与具体硬件和内核实现高度耦合。
- **为什么重要**：由于缺乏统一比较，研究者常依赖FLOPs减少量或单内核吞吐量等理论代理指标来评估加速效果，但这些指标与端到端实际加速比之间存在巨大差距。这导致：
    1.  无法准确判断哪种剪枝方法在实际部署中更有效。
    2.  新提出的剪枝方法难以在公平的环境下与现有方法进行比较。
    3.  阻碍了领域对剪枝加速潜力的根本理解。
- **挑战**：主要挑战在于如何建立一个**实现一致（implementation-consistent）** 的比较框架。不同剪枝方法需要不同的内核实现和系统优化，直接比较它们的实际延迟是不公平的。需要一种抽象方法，能够剥离方法特定的实现细节，在统一的底层操作（即GEMM）层面进行比较。

## 2. 创新点（Innovation）

- **关键创新**：
    1.  **GEMM-centric分类法**：这是本文最核心的理论创新。它将所有LLM剪枝方法重新组织，根据它们减少的是GEMM操作中的哪个逻辑维度（M: 批次/序列维度，N: 输出特征维度，K: 输入特征/规约维度）进行分类。例如，深度剪枝（如ShortenedLLaMA）被归类为静态M剪枝；宽度剪枝（如SliceGPT）被归类为静态NK剪枝；低秩近似（如SVD-LLM）被归类为静态K剪枝。
    2.  **统一的基准测试框架**：基于上述分类法，构建了一个可复用的基准测试框架。该框架为每个剪枝分类（如Static M, Static NK, Dynamic M等）提供了标准化的、硬件无关的加速实现（如使用Triton和TileLang编写内核），从而确保了不同方法在相同硬件和软件栈下进行公平比较。
- **与现有方案的不同**：现有工作通常只比较同一种类下的方法（如比较不同的宽度剪枝方法），或者使用不一致的模型、硬件和优化设置进行比较。本文的创新在于**跨种类**、**跨方法**地在一个统一框架下进行比较，首次揭示了不同剪枝家族之间的真实性能差异。
- **为什么能奏效**：该分类法抓住了LLM推理的核心——GEMM操作。无论剪枝方法多么复杂，其最终效果都可以归结为对GEMM的M、N、K维度进行缩减。这个抽象层成功地将“剪枝算法”与“硬件实现”解耦，使得在统一的“操作”层面进行比较成为可能。

## 3. 主要内容（Main Content）【重点】

### 3.1 设计架构与方法

本文的核心架构是围绕其GEMM-centric分类法构建的。主要分为两步：

1.  **分类与映射**：首先，将现有的LLM剪枝方法根据其剪枝的GEMM维度（M, N, K）进行分类，并分析其在不同GEMM操作间的传播行为。例如，对N维度的剪枝会传播到下一个GEMM的K维度，形成NK模式。
2.  **统一实现与基准测试**：为每个分类（Static M, Static K, Static NK, Dynamic M, Dynamic NK）设计并实现标准化的加速内核。这些内核作为“基线”，用于公平地比较不同分类下的代表性方法。

### 3.2 关键算法与数学推导

论文的核心数学推导在于分析不同维度剪枝对GEMM操作的影响及其传播特性。

**GEMM定义**：一个标准的GEMM操作可以表示为：
$$O = A W^\top$$
其中 $A \in \mathbb{R}^{T \times D}$ 是输入张量，$W \in \mathbb{R}^{I \times D}$ 是权重矩阵，$O \in \mathbb{R}^{T \times I}$ 是输出张量。逻辑维度对应关系为：$M = T$ (token维度), $N = I$ (输出特征维度), $K = D$ (输入特征维度)。

**不同维度剪枝的数学表达**：
对于基于掩码的结构化剪枝，其效果可以表示为：

- **M维度剪枝**：对输入 $A$ 的行进行掩码。
  $$O = (A \odot M_M) W^\top$$
  其中 $M_M \in \{0,1\}^{T \times 1}$，通过广播机制作用于所有列。这等价于跳过某些token的计算。

- **N维度剪枝**：对权重 $W$ 的列（输出特征）进行掩码。
  $$O = A (W^\top \odot M_N)$$
  其中 $M_N \in \{0,1\}^{1 \times I}$。这相当于减少了输出特征的维度。

- **K维度剪枝**：对权重 $W$ 的行（输入特征）进行掩码。
  $$O = (A \odot M_K) W^\top = A (W \odot M_K)^\top$$
  其中 $M_K \in \{0,1\}^{1 \times D}$。这相当于减少了输入特征的维度。

**传播特性**：
考虑一个后续的GEMM操作 $O' = O \hat{W}^\top$，其中 $\hat{W} \in \mathbb{R}^{D \times I}$。

- **M维度传播**：$M$ 维度上的掩码 $M_M$ 会直接传递到后续所有GEMM操作的 $M$ 维度上，因为输出 $O$ 的行数被减少了。
- **N维度传播**：$N$ 维度上的掩码 $M_N$ 会传播到下一个GEMM的 $K$ 维度上。因为 $O$ 的列数（即 $N$ 维度）被减少了，而 $O$ 作为下一个GEMM的输入时，其列数对应着下一个GEMM的 $K$ 维度。这形成了 **NK模式**。
- **K维度传播**：$K$ 维度上的掩码不会传播到后续GEMM，因为它只影响当前GEMM的规约维度，不影响输出张量的形状。

### 3.3 模型、训练与数据集

- **模型**：主要实验使用 **Llama3.1-8B** 作为基线模型。在附录中，使用 **Qwen3-14B** 进行额外验证。
- **训练**：所有剪枝方法在剪枝后，都使用统一的 **LoRA (Low-Rank Adaptation)** 微调策略进行恢复训练。训练配置为：5,000步（Dynamic M为10,000步），批次大小16，序列长度2,048，LoRA秩为16。训练数据来自 **RedPajama** 数据集的一个子集。
- **数据集**：
    - **质量评估**：使用 **WikiText2** 的困惑度（perplexity）来衡量语言建模能力。同时，使用多个常识推理基准测试（如ARC, BoolQ, PIQA, HellaSwag, WinoGrande）的平均准确率来衡量模型性能。
    - **加速评估**：使用自定义的基准测试框架，在 **RTX Pro 6000 Blackwell** GPU上进行延迟和吞吐量测量。

### 3.4 实验设置与结果

- **实验设置**：硬件为RTX Pro 6000 Blackwell。所有性能分析实验都集成了CUDA图以减少CPU开销。对于每种剪枝策略，通过随机采样其原子单元来模拟目标稀疏度。吞吐量计算为 `Token/s = (B * T_q) / TTFT 或 TPOT`。
- **主要结果**：
    1.  **Pareto前沿**：如图1所示，在吞吐量-质量权衡中，**静态M（Static M）** 在低性能损失（0-4%）下表现最佳；**动态M（Dynamic M）** 在中等损失（5-16%）下领先；**静态NK（Static NK）** 在高损失（17-26%）下才具有竞争力。静态K和动态NK等方法不在前沿上。
    2.  **Prefill与Decode速度**：如图5所示，静态方法（尤其是静态M）提供的加速比最稳定。在50%稀疏度下，其他方法的加速比仅相当于静态M在更低稀疏度（如34%-48%）下的表现。宽度剪枝的加速效果普遍落后于深度剪枝。
    3.  **延迟分解**：如图6所示，理论加速与实际加速之间的差距主要来自非GEMM开销（如内核预处理、额外投影操作）以及剪枝后GEMM本身的低效。动态M和静态K的非GEMM开销显著增加，抵消了部分计算减少带来的收益。

### 3.5 技术细节

- **静态K实现**：对于半结构化稀疏（如2:4），使用JIT接口直接调用cuSPARSELt库，将初始化延迟降低了94.4%（从726µs降至40µs）。对于低秩近似，将单个GEMM分解为两个较小的GEMM。
- **静态M和NK实现**：关键在于维护依赖操作符之间的维度一致性。通过一个统一的包装器，将相互依赖的模块组（如Attention和MLP中的多个线性层）作为一个整体进行重连和剪枝。对于跨层静态NK（如SliceGPT），需要处理嵌入层、残差连接和LM头的特殊传播路径（算法1）。
- **动态M实现**：核心挑战是将在线稀疏性转化为实际加速。采用“掩码重排序 + 基于Tile的内核跳过”策略（图4）。首先对token的活跃/非活跃掩码进行重排序，将活跃token聚集在一起，然后以tile为单位执行GEMM，只计算包含活跃token的tile，从而跳过非活跃token的计算。

## 4. 意义与影响（Significance and Impact）

- **解决的问题与贡献**：
    1.  **统一了评估标准**：解决了LLM剪枝领域长期存在的评估碎片化问题，为公平比较不同剪枝方法提供了理论基础和实用工具。
    2.  **揭示了真实加速瓶颈**：明确指出名义稀疏度是实际加速的弱预测因子，实际加速取决于剪枝分类法、结构传播和系统开销。这纠正了领域内对FLOPs等理论指标的过度依赖。
    3.  **绘制了Pareto前沿**：首次系统性地绘制了当前剪枝加速的性能边界，为研究者提供了清晰的“地图”，指导他们根据性能损失预算选择合适的剪枝方向。
- **对未来研究/应用的影响**：
    - **研究方向**：论文结论指出，未来的Pareto前沿很可能来自**混合设计**，即结合质量感知的动态分支（如Dynamic M）和结构化、静态的主干（如Static M或Static NK）。这为下一代剪枝算法指明了方向。
    - **应用实践**：为工程师在选择LLM剪枝加速方案时提供了数据驱动的决策依据。例如，在追求极小性能损失时，静态深度剪枝是首选；在允许一定性能损失以换取更高加速时，可以考虑动态深度剪枝。
- **局限与开放问题**：
    1.  **硬件平台有限**：实验主要在RTX Pro 6000 (sm120)上进行，未涵盖Hopper (sm90)或数据中心Blackwell (sm100)等平台，这些平台可能改变动态方法的相对优势。
    2.  **实现非最优**：使用的Triton/TileLang实现是便携式基线，可能不如手写CUDA内核或利用特定硬件特性（如TMA）的实现高效。
    3.  **方法覆盖不全**：只评估了每个分类下的代表性方法，而非所有变体。因此，报告的Pareto前沿是当前选定方法下的前沿，而非所有可能剪枝方法的绝对上限。
    4.  **系统级评估缺失**：测量聚焦于单步延迟和内核行为，未在生产级推理框架（如SGLang）中进行完整的服务负载测试。

## 5. 澄清与简化（Clarifications and Simplifications）

- **GEMM维度类比**：可以把GEMM想象成一个工厂生产线。
    - **M（Token维度）**：代表有多少个产品（token）需要同时处理。M剪枝就是减少同时处理的产品数量，比如只处理一半的产品。
    - **N（输出特征维度）**：代表每个产品需要经过多少道工序（输出特征）。N剪枝就是减少工序数量，比如原本需要100道工序，现在只做50道。
    - **K（输入特征/规约维度）**：代表每道工序需要消耗多少原材料（输入特征）。K剪枝就是减少原材料的种类，比如原本需要100种原材料，现在只用50种。
- **传播行为类比**：
    - **M剪枝**：就像你决定只处理第1、3、5号产品，那么后续所有工序都只需要处理这3个产品。这是“贯穿始终”的减少。
    - **N剪枝**：就像你决定第一道工序只做50种（而不是100种），那么这50种结果会成为第二道工序的“原材料”，第二道工序的原材料种类自然就减少了。这就是“N传播为K”。
    - **K剪枝**：就像你决定第一道工序只用50种原材料，但生产出来的产品（输出）仍然是100种。所以第二道工序的原材料种类并没有减少，影响仅限于第一道工序本身。
- **为什么静态M加速最稳定**：因为M剪枝（如移除整个层）直接减少了需要执行的GEMM操作数量，这是一种“粗粒度”的、结构化的减少，对硬件执行流影响最小，也最容易实现加速。而其他剪枝方法（如宽度剪枝）虽然减少了计算量，但引入了不规则的内存访问模式或额外的内核启动开销，导致实际加速比低于理论值。
