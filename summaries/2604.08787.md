## 0. 概览（Concise Summary）

- **主题（Topic）**：协作机器人实时底层运动规划；机器人中间件；异构硬件平台上的统一控制接口。
- **问题（Problem）**：现有协作机器人运动规划方案与特定硬件紧密耦合，缺乏标准化、可移植的软件接口，导致跨平台部署困难、学习成本高、实时性和可中断性不足。
- **方法（Method）**：提出一个名为 `rt-move-cartesian` 的统一实时底层运动规划API，集成在WinGs Operating Studio (WOS) 中间件中。该方法通过将笛卡尔空间路径点逆解到关节空间，然后使用 n 次多项式插值与二次规划（QP）求解器相结合，生成平滑、连续可微的关节空间轨迹。
- **创新（Innovation）**：
    1.  **统一接口**：提供了一个与硬件无关的、基于时间参数化笛卡尔路径点的运动规划接口，可控制WOS支持的任何协作臂。
    2.  **实时优化**：将n次多项式插值与QP优化结合，在保证轨迹平滑（最小化jerk）的同时，满足严格的实时计算约束。
    3.  **多模式支持**：同一API支持离线规划、可中断的低频重规划和实时遥操作三种典型应用场景。
- **意义（Significance）**：该工作显著降低了异构机器人应用开发的复杂性和门槛，提高了软件的可移植性和复用性。它为构建跨平台、可扩展的机器人系统提供了关键基础设施，对工业自动化、服务机器人和远程操作等领域具有重要价值。
- **一句话总结**：本文提出并实现了一个基于WOS中间件的统一实时运动规划接口，通过多项式插值与二次规划优化，为异构协作机器人提供了平滑、可中断、跨平台的笛卡尔空间运动控制能力。

## 1. 研究动机（Motivation）

- **解决的问题**：当前协作机器人应用开发面临三大障碍：
    1.  **硬件异构性与软件碎片化**：不同厂商的机器人（如Franka, Lite6）拥有专有的控制接口和编程方式，导致为特定机器人开发的运动规划代码难以移植到其他平台上。
    2.  **缺乏标准化、可移植的底层接口**：现有主流框架（如ROS MoveIt）虽然功能强大，但更侧重于离线路径规划，在需要**实时响应**、**在线重规划**和**可中断性**的动态环境中表现不佳。其依赖的采样规划器（如OMPL）在严格实时约束下预测性较差。
    3.  **开发门槛高**：多学科系统设计的复杂性、对底层硬件的依赖以及缺乏统一的软件抽象，导致学习曲线陡峭，集成困难。
- **为什么重要**：解决这些问题可以：
    1.  **提升开发效率**：开发者可以编写一次运动控制逻辑，即可在不同机器人上运行，无需为每种硬件重复开发。
    2.  **增强系统鲁棒性**：统一的接口使得在动态环境中进行实时、可中断的轨迹重规划成为可能，这对于人机协作、动态抓取等任务至关重要。
    3.  **促进技术复用**：高级应用逻辑（如AI决策）可以与底层硬件解耦，使得算法和功能模块可以更容易地在不同平台间迁移和复用。
- **挑战在哪里**：核心挑战在于设计一个**轻量级、实时性强、且与硬件无关**的底层运动规划接口。它需要：
    1.  能够处理来自不同硬件（如仅提供力矩接口的机器人）的异构性。
    2.  在计算资源有限的嵌入式平台上也能满足实时控制周期（如100Hz）的要求。
    3.  生成的轨迹必须平滑、连续可微（位置、速度、加速度），并满足关节的物理约束（速度、加速度极限）。
    4.  支持多种操作模式，从预编程的离线任务到需要频繁中断的在线重规划，再到高频率的遥操作。

## 2. 创新点（Innovation）

- **关键创新**：提出并实现了一个**统一的、实时的、低层运动规划API**，作为WOS中间件的核心功能。
- **与现有方案的不同**：
    - **vs. 专有接口**：传统方案（如[13, 14]）与特定硬件绑定，不可移植。本文的API通过WOS中间件抽象了硬件差异，实现了“一次编写，到处运行”。
    - **vs. ROS MoveIt**：MoveIt擅长离线规划，但在实时性、可中断性和动态环境适应性上存在局限。本文的API专门为**实时在线重规划**设计，通过QP求解器在毫秒级内生成新轨迹，并支持随时中断当前轨迹。
    - **vs. 纯多项式插值**：单纯的高次多项式插值可能导致轨迹在路径点间剧烈振荡或超出关节极限。本文通过**QP优化**，在保证轨迹平滑（最小化jerk）的同时，显式地施加了关节速度、加速度的**不等式约束**，确保了轨迹的动态可行性。
- **为什么能奏效**：
    1.  **架构解耦**：WOS中间件将硬件驱动、运动学求解、运动规划等模块抽象为独立的“节点”（Nodes），通过标准网络协议通信。这使得运动规划API可以独立于底层硬件。
    2.  **算法高效**：n次多项式插值保证了轨迹的解析连续性，而QP问题（公式10）的规模可控，可以使用高效的求解器（如OSQP）在毫秒级内完成求解，满足实时性要求。
    3.  **设计灵活**：API基于时间参数化的笛卡尔路径点，用户只需指定目标位姿和到达时间，速度和加速度由规划器自动生成。这种设计天然支持三种不同的使用模式（离线、可中断、遥操作），覆盖了广泛的应用场景。

## 3. 主要内容（Main Content）【重点】

### 3.1 设计架构与方法

- **整体架构**：基于WOS中间件，构建了一个统一的实时末端执行器控制管线（图2）。
    1.  **硬件抽象层**：WOS将不同机器人（A, B, C）抽象为统一的“组件”（Component），通过硬件驱动（Driver）处理底层通信。对于仅提供力矩接口的机器人，WOS集成了PID控制器，将其标准化为关节位置可控模型。
    2.  **功能节点**：逆运动学（IK）求解器、运动规划模块、PID控制器等被封装为内部系统节点。
    3.  **统一API**：`rt-move-cartesian` 命令是核心接口。它接收一个包含机器人ID、请求类型和结构化数据（笛卡尔路径点列表）的请求。
    4.  **执行管线**：包含两个线程：
        - **监听线程**：持续监听新的运动命令，调用IK求解器和轨迹插值模块，实时更新运动计划。
        - **控制线程**：以固定控制频率（如100Hz）运行，将最新的参考状态（位置、速度、加速度）发送给硬件驱动。
- **API设计**：
    - **输入**：用户只需提供一个笛卡尔路径点列表 $\mathcal{C} = \{\boldsymbol{\xi}_1, \boldsymbol{\xi}_2, \dots, \boldsymbol{\xi}_N\}$，其中每个路径点 $\boldsymbol{\xi}_i = (\mathbf{x}_i, D_i)$ 包含末端执行器的目标位姿 $\mathbf{x}_i \in \mathbb{R}^6$ 和到达该点所需的时间 $D_i$。
    - **输出**：生成一个平滑、连续可微的关节空间轨迹函数 $\hat{\mathbf{s}}(t) = (\hat{\mathbf{q}}(t), \hat{\dot{\mathbf{q}}}(t), \hat{\ddot{\mathbf{q}}}(t))$。
    - **特性**：支持异步、可中断请求，对非均匀传输速率具有鲁棒性。初始位姿从机器人当前状态推断，最终路径点强制速度和加速度为零，确保平滑停止。

### 3.2 关键算法与数学推导

1.  **问题形式化**：
    - 输入：笛卡尔路径点列表 $\mathcal{C}$。
    - 通过逆运动学（IK）求解器 $\mathcal{K}^{-1}$ 将笛卡尔路径点转换为关节空间路径点 $\mathcal{J} = \{\mathbf{j}_1, \dots, \mathbf{j}_N\}$，其中 $\mathbf{j}_i = (\mathbf{d}_i, D_i)$，$\mathbf{d}_i$ 是关节位置向量。
      $$\mathbf{d}_i = \mathcal{K}^{-1}(\mathbf{x}_i; \mathbf{d}_{i-1}) \quad (1)$$
    - 目标：生成一个关节空间轨迹函数 $\hat{\mathbf{s}}(t)$，使其在指定时间 $T_i$ 精确通过目标关节位置 $\mathbf{d}_i$，并在终点 $T_N$ 处速度和加速度为零，同时最小化整个运动过程中的jerk。

2.  **n次多项式函数**：
    - 对每个关节独立进行轨迹插值。对于第 $i$ 个时间段 $t \in (T_{i-1}, T_i)$，使用一个 $L$ 次多项式拟合：
      $$\hat{q}_i(t) = \mathbf{b}_L(t) \mathbf{p}_i^T \quad (3)$$
      其中 $\mathbf{b}_L(t) = [1, t, t^2, \dots, t^L]$ 是单项式基向量，$\mathbf{p}_i = [p_{i,0}, p_{i,1}, \dots, p_{i,L}]$ 是第 $i$ 段的系数向量。
    - 轨迹及其导数可以表示为：
      $$\hat{q}_i^{(k)}(t) = \mathbf{b}_L^{(k)}(t) \mathbf{p}_i^T, \quad k = 0,1,2,3 \quad (4)$$
    - 要求 $L \ge 4$ 以保证有足够的自由度来满足位置、速度和加速度的边界条件。

3.  **轨迹优化（QP问题）**：
    - 为了最小化整体jerk并保证段间连续性，将轨迹生成问题建模为一个二次规划（QP）问题。
    - **目标函数**：最小化所有段的总jerk。
      $$\min_{\mathbf{p}_i} \sum_{i=1}^N \mathbf{p}_i^T \mathbf{Q}_i \mathbf{p}_i \quad (6)$$
      其中 $\mathbf{Q}_i$ 是第 $i$ 段的jerk代价矩阵，通过对jerk的平方在离散时间点上求和得到：
      $$\mathbf{Q}_i = \sum_{t \in \tau_i} \dddot{\mathbf{b}}_L(t) \dddot{\mathbf{b}}_L(t)^T \quad (7)$$
      $\tau_i$ 是第 $i$ 段时间内的离散采样点集合。
    - **等式约束** $\mathbf{A}_{eq} \mathbf{p} = \mathbf{b}_{eq}$：
        - 初始状态约束：$\hat{q}_1(0) = q_0, \hat{\dot{q}}_1(0) = \dot{q}_0, \hat{\ddot{q}}_1(0) = \ddot{q}_0$
        - 终端状态约束：$\hat{q}_N(T_N) = d_N, \hat{\dot{q}}_N(T_N) = 0, \hat{\ddot{q}}_N(T_N) = 0$
        - 段间连续性约束：$\hat{q}_i(T_i) = \hat{q}_{i+1}(T_i), \hat{\dot{q}}_i(T_i) = \hat{\dot{q}}_{i+1}(T_i), \hat{\ddot{q}}_i(T_i) = \hat{\ddot{q}}_{i+1}(T_i)$
    - **不等式约束** $\mathbf{A}_{ineq} \mathbf{p} \le \mathbf{b}_{ineq}$：确保物理可行性。
        - 关节速度极限：$|\hat{\dot{q}}_i(t_{i,j})| \le v_{max}$
        - 关节加速度极限：$|\hat{\ddot{q}}_i(t_{i,j})| \le a_{max}$
    - **标准QP形式**：
      $$\min_{\mathbf{p}} \mathbf{p}^T \mathbf{Q} \mathbf{p} \quad \text{s.t.} \quad \mathbf{A}_{eq} \mathbf{p} = \mathbf{b}_{eq}, \quad \mathbf{A}_{ineq} \mathbf{p} \le \mathbf{b}_{ineq} \quad (10)$$
      其中 $\mathbf{Q} = \text{blockdiag}(\mathbf{Q}_1, \dots, \mathbf{Q}_N)$。
    - **实时执行**：求解QP得到系数向量 $\mathbf{p}$ 后，在实时执行过程中，根据当前时间 $t$ 直接代入公式(5)计算参考状态，避免了预计算整个轨迹带来的精度损失和计算开销。

### 3.3 模型、训练与数据集

- **模型**：本文不涉及机器学习模型的训练。核心模型是**n次多项式插值模型**和**二次规划（QP）优化模型**。这些是经典的数学优化模型，没有训练过程。
- **数据集**：没有使用标准数据集。实验中的轨迹（如直线、圆形、动态抓取路径）是由用户或感知系统实时生成的笛卡尔路径点序列。

### 3.4 实验设置与结果

论文通过三个实验验证了API的三种使用模式。

1.  **计算性能评估**：
    - **设置**：在三种不同计算平台（Jetson Nano, Raspberry Pi 5, 桌面级CPU）上，使用OSQP求解器，对6自由度机器人进行6个并行QP（每关节一个）求解，多项式次数 $L=5$，采集400次求解时间样本。
    - **结果**（图4）：所有平台均达到毫秒级求解时间。桌面级CPU最快（~0.5ms），Raspberry Pi 5平均约0.7ms，Jetson Nano平均约2.1ms。结论：算法能在主流嵌入式平台上满足实时控制需求。

2.  **离线运动规划：几何绘图任务**：
    - **设置**：Franka FR3机器人末端安装笔，执行绘制直线（7个路径点）和圆形（18个路径点）任务，控制频率100Hz。
    - **结果**（图5, 图6）：机器人能准确跟踪规划的笛卡尔轨迹，关节轨迹平滑连续。验证了即使在关节空间进行多项式插值，也能在笛卡尔空间获得满意的几何精度，足以满足焊接、喷涂等典型应用。

3.  **可中断低频重规划：动态物体抓取**：
    - **设置**：一个移动机器人携带一个橙色物体运动，固定FR3机械臂通过双摄像头（SOLOv2实例分割）以约1Hz频率跟踪物体位置。机械臂持续向物体上方位置发送单一路径点命令，每次新命令都会中断前一个未完成的轨迹。
    - **结果**（图7, 图8）：机械臂成功跟踪并抓取了移动中的物体。图8展示了末端执行器x坐标随时间的变化，清晰地显示了轨迹被反复中断和重规划的过程，验证了系统在动态环境下的鲁棒性和实时适应性。

4.  **高频实时控制：双臂遥操作**：
    - **设置**：操作员通过力传感器（Delta6）控制Lite6机械臂，Lite6的运动通过`rt-move-cartesian` API（50Hz）实时映射到FR3机械臂。FR3接收包含5个路径点缓冲区的请求（25Hz），执行频率100Hz。任务包括开抽屉、取印章、盖章等复杂操作。
    - **结果**（图9）：FR3平滑地跟随Lite6的运动，完成了整套复杂任务。关节轨迹（图9B）展示了在不同运动强度下，位置、速度、加速度曲线均保持连续平滑，且每个规划段在终点自然收敛到零速度和加速度，确保了安全性和流畅性。

### 3.5 技术细节

- **逆运动学（IK）**：使用通用的IK求解器，如TRAC-IK，支持解析解和数值解，以适应不同构型的机器人。
- **QP求解器**：使用OSQP（Operator Splitting Quadratic Program）求解器，专为嵌入式系统和实时应用设计，效率高。
- **多项式次数选择**：实验中 $L=5$（五次多项式），这是一个常见选择，在平滑性、计算复杂度和约束满足能力之间取得了良好平衡。
- **控制频率**：机器人控制器以固定频率 $f_c = 100\text{Hz}$ 运行，这是协作机器人控制的典型频率。
- **遥操作缓冲**：在遥操作模式中，引入一个固定大小的缓冲区（$N=5$个路径点），以平滑主端设备的低频、抖动和通信延迟，虽然引入了约200ms的固定延迟，但显著提升了运动的连续性和平滑性。

- **解决了什么问题**：
    - 解决了异构协作机器人平台间**底层运动规划接口不统一、不可移植**的核心问题。
    - 解决了在动态环境中**实时、可中断的轨迹重规划**的挑战。
    - 降低了机器人应用开发的**复杂性和门槛**。

- **贡献**：
    1.  提出了一个**轻量级、通用的实时运动规划接口**，作为WOS中间件的核心功能。
    2.  提出了一种**结合n次多项式插值和QP优化的轨迹生成方法**，在保证平滑性的同时满足实时性和物理约束。
    3.  通过**三个代表性实验**，全面验证了该接口在离线、可中断和遥操作三种典型场景下的有效性、鲁棒性和易用性。

- **对未来研究/应用的影响**：
    - **工业自动化**：为焊接、喷涂、码垛等任务提供了一种可快速部署、跨平台的运动控制方案。
    - **服务机器人**：使得机器人（如咖啡师、按摩机器人）的控制逻辑可以更容易地在不同硬件平台上复用。
    - **人机协作**：可中断的规划能力是实现安全、自然的人机协作的关键。
    - **远程操作**：为在危险环境（核废料处理、水下作业）中的遥操作提供了稳定、平滑的控制基础。
    - **AI
