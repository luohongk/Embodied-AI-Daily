## 0. 概览（Concise Summary）

- **主题（Topic）**：多智能体协作、软件工程、大语言模型（LLM）应用。
- **问题（Problem）**：现有的多智能体软件开发系统通常假设所有智能体共享相同的运行环境、中央对话服务器或同一API家族，导致来自不同供应商（如Anthropic和OpenAI）的LLM智能体难以在各自的执行环境中直接通信并协作开发同一代码库。
- **方法（Method）**：提出了一种名为`tap`的基于文件的协作协议。该协议采用“文件优先”设计，将Markdown文件作为原始消息的持久化存储，并结合了文件通信（Tier 1）和实时通知（Tier 2）两条路径，同时通过独立的Git工作树隔离工作空间。
- **创新（Innovation）**：核心创新在于利用普遍存在的本地文件系统作为异构LLM智能体之间的通用接口，而非依赖共享内存或统一的运行时环境。这解决了不同模型、不同执行环境智能体间直接通信的难题。
- **意义（Significance）**：`tap`协议证明了异构LLM智能体可以在没有中央协调服务器的情况下，在同一个生产仓库中协作完成开发与审查任务。实验表明，异构模型配对（如Claude审查Codex的代码）比同构模型配对能发现更多缺陷，为构建更具鲁棒性和多样性的多智能体系统提供了新思路。
- **一句话总结**：本文提出并验证了一个名为`tap`的基于文件的轻量级协议，该协议通过将消息持久化到文件系统，成功实现了来自不同供应商的LLM智能体（Claude和Codex）在异构环境下的直接协作与代码审查，并观察到异构模型配对能带来更广泛的审查视角。

## 1. 研究动机（Motivation）

当前，基于LLM的编码智能体（如ChatDev、MetaGPT、AutoGen）在软件开发中展现出巨大潜力。然而，这些系统普遍存在一个关键假设：所有参与的智能体必须运行在**相同的模型家族、共同的运行时环境、中央对话服务器或同一API家族**下。

这种假设带来了两个主要问题：
1.  **同质化偏见**：当作者和审查者来自同一模型家族时，他们可能共享相似的错误倾向。基于共识的策略（如多数投票）可能无法有效过滤这些共有的错误，陷入“流行陷阱”（popularity trap）。
2.  **协作壁垒**：实际开发中，开发者可能希望使用不同供应商的模型（如Claude和Codex）来利用各自的优势。但现有的架构无法让这些来自不同执行环境的智能体直接交换消息并协作。

因此，本文的动机是**设计一种不依赖于共同运行时或中央服务器的、轻量级的协作协议，使来自不同供应商、运行在不同环境下的LLM智能体能够直接通信，并在同一代码库上分工协作**。这旨在打破模型和环境的壁垒，引入多样性，从而提升协作质量和缺陷检测能力。

## 2. 创新点（Innovation）

本文的核心创新在于提出了**基于文件的协作协议（`tap`）**，其关键创新点如下：

1.  **文件优先设计（File-First Design）**：与传统的基于内存或网络消息的通信不同，`tap`将**文件系统作为消息的权威存储和原始记录**。每条消息首先被写入一个Markdown文件（包含YAML元数据），这个文件就是消息本身。这确保了消息的持久性、可审计性和可重放性。
2.  **双通道通信架构（Two-Tier Communication）**：`tap`结合了两种通信路径：
    - **Tier 1: 文件通信（File Communication）**：通过MCP（Model Context Protocol）工具直接读取`inbox/`目录下的消息文件。这是一种轮询或文件监控路径，作为基础保障。
    - **Tier 2: 实时通知（Real-time Communication）**：针对不同执行环境（如Claude的MCP通道、Codex的WebSocket）提供实时通知，降低通信延迟。
    - **创新之处**：Tier 2负责速度，Tier 1负责可靠性。即使实时通知失败或接收方重启，消息文件依然存在，可以通过Tier 1重新获取，实现了“至少一次”的消息传递语义。
3.  **环境感知配置（Per-Environment Configuration）**：`tap`不强制所有智能体使用统一的API。相反，它通过标准化的`tap add`命令，能够自动识别并修改不同智能体（Claude、Codex、Gemini）的配置文件（如`.mcp.json`、`config.toml`），实现了对异构执行环境的原生适配。
4.  **工作空间隔离（Workspace Isolation）**：通过为每个智能体创建独立的Git工作树（worktree），避免了文件冲突。同时，通过`instanceId`隔离PID文件、日志和状态文件，允许多个同类型智能体并行运行。

## 3. 主要内容（Main Content）【重点】

### 3.1 设计架构与方法

`tap`的架构围绕“文件优先”和“双通道通信”展开，如图1所示。

- **核心组件**：
    - **`inbox/`目录**：所有消息的原始存储位置。当一个智能体调用`tap_reply`时，一个包含YAML元数据的Markdown消息文件首先被写入此目录。
    - **`reviews/`, `findings/`, `retros/`, `handoff/`等目录**：用于存储不同类型的操作工件（artifacts），如审查记录、发现的问题、回顾总结和交接文档。
    - **Git工作树（worktrees）**：每个智能体拥有独立的Git工作树，用于隔离代码修改，避免冲突。
- **通信流程**：
    1.  发送方智能体调用`tap_reply`，将消息写入`inbox/`。
    2.  文件写入完成后，发送方尝试通过Tier 2（实时通知）通知接收方。
    3.  接收方收到通知后，**不依赖通知内容本身**，而是重新读取`inbox/`中的原始文件进行处理。
    4.  如果Tier 2失败，接收方可以通过Tier 1（文件轮询或MCP工具）在下次检查时发现并处理未读消息。

### 3.2 关键算法与数学推导

论文本身没有复杂的数学推导，其核心逻辑是协议设计。关键算法可以理解为`tap_reply`函数的伪代码：

```
Function tap_reply(sender, receiver, message_content):
    # 1. 创建消息文件
    message_file = create_markdown_file(inbox/, sender, receiver, message_content)
    write_metadata(message_file, sender, receiver, timestamp)

    # 2. 尝试实时通知 (Tier 2)
    if receiver.environment == "Claude":
        send_mcp_notification(receiver, message_file.path)
    elif receiver.environment == "Codex":
        send_websocket_message(receiver, message_file.path)
    # ... 其他环境

    # 3. 返回成功（无论通知是否成功）
    return success
```

接收方处理逻辑的伪代码：

```
Function tap_receive(receiver):
    # 1. 检查实时通知 (Tier 2) 或轮询 (Tier 1)
    if received_notification:
        message_path = notification.payload # 仅作为提示
    else:
        message_path = find_unprocessed_file(inbox/, receiver)

    # 2. 读取原始文件
    if message_path:
        message_content = read_file(message_path)
        process_message(receiver, message_content)
        mark_as_processed(message_path)
```

### 3.3 模型、训练与数据集

- **模型**：论文使用了三个不同供应商的LLM模型：
    - **Claude** (Anthropic)
    - **Codex** (OpenAI)
    - **Gemini** (Google) - 仅通过文件通信（Tier 1）进行实验性参与。
- **训练**：论文不涉及模型训练。`tap`是一个协议，用于协调这些**预训练**的LLM智能体。
- **数据集**：论文没有使用外部公开数据集。其数据来源于一次**27天、37代（generations）的自我应用（self-applied）操作**。在这个过程中，`tap`协议被用于开发、审查和维护`tap`协议本身。
    - **“代”（Generation）**：指一个智能体团队因上下文限制或任务完成而被替换，下一团队继承上一代操作记录的操作单元。
    - **数据规模**：
        - 209个与`tap`相关的Pull Requests (PRs)
        - 717个操作工件（artifacts），包括回顾、发现、审查和交接文档。
        - 375个审查工件（review artifacts）被用于分析。

### 3.4 实验设置与结果

- **实验设置**：
    - **环境**：一个包含多个应用和包的真实生产仓库。
    - **角色**：Claude和Codex分别扮演开发者和审查者角色。后期，一个Codex审查者在独立的Linux服务器上以无头模式（headless mode）持续运行。
    - **操作**：人类操作员负责协调工作流和最终审批，但消息传递和进度检查逐渐由智能体自主完成。
- **实验结果**：
    1.  **操作可行性**：`tap`成功支持了Claude和Codex在异构环境下的协作，完成了27天的自我开发与审查任务。
    2.  **缺陷检测率**：对375个审查工件的分析显示（表3）：
        - **异构模型配对**（如Codex审查Claude的代码）：缺陷或变更请求的检出率为 **69.8%** (183/262)。
        - **同构模型配对**（如Claude审查Claude的代码）：检出率为 **53.1%** (60/113)。
        - **结论**：在该实验场景下，异构模型配对展现出更高的缺陷检出率，表明结合不同模型和执行环境可以拓宽审查视角。
    3.  **故障恢复**：`tap`的文件优先设计成功处理了多种故障（表2），如路径问题、日志文件为空、名称冲突等，并将这些故障转化为协议改进项。
    4.  **跨代连续性**：717个操作工件作为外部记忆，使得后续代的智能体能够理解工作上下文和决策依据，弥补了LLM智能体缺乏持久内部状态的不足。

### 3.5 技术细节

- **消息格式**：消息以Markdown文件存储，包含YAML格式的元数据（如发送者、接收者、时间戳）。
- **实时通知**：
    - **Claude**：通过MCP（Model Context Protocol）通道进行通知。
    - **Codex**：通过一个“交付程序”（delivery program）使用WebSocket进行通知。
- **工作空间隔离**：使用`git worktree`命令为每个智能体创建独立的代码工作目录。
- **配置管理**：`tap add`命令通过检查当前配置、创建变更计划、应用并验证结果的标准流程，来修改不同智能体的配置文件。
- **包发布**：`tap`作为开源npm包`@hua-labs/tap` (v0.5.2) 发布。

## 4. 意义与影响（Significance and Impact）

- **解决的问题与贡献**：
    1.  **解决了异构LLM智能体协作的通信壁垒问题**，提出了一种不依赖共同运行时的轻量级解决方案。
    2.  **提供了长期、真实的生产环境操作案例**，证明了基于文件的协作协议在实践中的可行性。
    3.  **提供了异构模型协作能提升缺陷检测率的实证证据**，为多智能体系统的设计提供了重要参考，即引入模型多样性可能比追求模型一致性更有价值。

- **对未来研究/应用的影响**：
    - **多智能体系统设计**：`tap`的“文件优先”思想可能成为一种新的范式，特别是在需要审计、容错和跨代记忆的场景下。
    - **软件工程实践**：可以启发新的CI/CD工具和代码审查流程，允许开发者轻松地让不同AI助手协作。
    - **模型评估**：`tap`提供了一种新的评估视角，即通过模型间的相互审查来评估模型能力，而不仅仅是依赖标准基准。

- **局限性与开放问题**（作者明确提到）：
    1.  **观察性研究**：数据是观察性的，非受控实验，审查者分配非随机，存在混淆因素（如时间效应、模型类型、角色、操作系统差异）。
    2.  **分析单元**：分析单元是审查工件，而非独立缺陷，可能存在重复计数。
    3.  **泛化性**：实验仅在一个组织的一个仓库中进行，结论的泛化性需要验证。
    4.  **模型覆盖**：实时协作仅限于Claude和Codex，Gemini参与有限。
    5.  **协议演化**：`tap`协议本身在37代中不断演化，早期和晚期条件不同。
    6.  **性能变化**：LLM智能体性能会随基础模型更新而变化。
    7.  **因果推断**：无法从数据中分离出模型差异、角色分配、任务特性等因素对缺陷检测率的独立影响。

## 5. 澄清与简化（Clarifications and Simplifications）

- **“文件优先”类比**：可以把`tap`想象成一个**共享的实体公告板**。每个智能体（Claude、Codex）都像是一个团队成员，他们不直接打电话或发即时消息，而是把写好的便签（Markdown文件）贴在公告板（`inbox/`目录）上。为了加快沟通，公告板旁边还有一个**电子提示屏**（Tier 2实时通知），当有新便签时，提示屏会闪烁提醒特定成员。但如果提示屏坏了，成员们只要定期去公告板前看一眼，就依然能获取所有信息。这个公告板就是“文件系统”，它是最可靠、最持久的信息源。
- **“代”（Generation）类比**：可以理解为**项目团队的轮换**。由于AI智能体有上下文窗口限制，无法处理无限长的对话历史。所以，当当前团队（第N代）的“工作记忆”快满了，或者完成了一个里程碑任务后，就会进行团队轮换。新的团队（第N+1代）会继承上一团队留下的所有工作笔记（操作工件），从而了解项目进展和决策原因，继续工作。这就像软件开发中，新入职的开发者通过阅读项目文档和代码注释来接手工作一样。
- **异构 vs 同构审查**：想象一个项目，如果让两个都用同样教材、同样思维模式的学生（同构模型）互相检查作业，他们可能都发现不了某些特定的错误。但如果让一个擅长逻辑推理的学生和一个擅长代码风格的学生（异构模型）互相检查，他们更有可能从不同角度发现问题。`tap`的实验结果就体现了这种“多样性红利”。
