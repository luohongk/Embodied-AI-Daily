## 0. 概览（Concise Summary）

- **主题（Topic）**：面向大语言模型多智能体系统（LLM-MAS）的工作负载感知跨集群调度。
- **问题（Problem）**：在资源受限的云环境中，LLM-MAS 工作流面临三大挑战：1）非确定性的执行成本（输出长度不可预测）；2）长尾模型使用模式导致的内存碎片化和过度配置；3）跨集群调度中延迟与资源就绪度的权衡。
- **方法（Method）**：提出 Maestro 系统，通过一个**智能体感知的成本预测器**（预测输出长度、KV 缓存需求）驱动一个**三级层次化调度器**：节点级（动态多模型共置与弹性内存管理）、集群级（基于适应度的路由）、全局级（工作流感知的优先级队列）。
- **创新（Innovation）**：1）首次将智能体的语义信息（角色、工具调用意图）用于细粒度的阶段级成本预测；2）提出结合层次化权重驻留和预测驱动弹性 KV 缓存的时空复用技术；3）设计了一个统一网络延迟、模型就绪度和 KV 可行性的适应度分数，用于跨集群调度。
- **意义（Significance）**：显著提升了 LLM-MAS 在严格 GPU 预算下的资源利用率和服务质量（SLO），为大规模部署复杂智能体工作流提供了可行的系统解决方案。
- **一句话总结**：Maestro 通过智能体感知的预测和三级协同调度，有效解决了 LLM 多智能体系统在资源受限集群中的资源争用和延迟问题。

## 1. 研究动机（Motivation）

论文的研究动机源于在资源受限的云环境中大规模部署 LLM 多智能体系统（LLM-MAS）时遇到的独特且严峻的系统挑战。与传统的单轮 LLM 查询不同，LLM-MAS 将每个用户请求分解为多个智能体协作的迭代工作流，这极大地放大了资源消耗。具体挑战如下：

1.  **非确定性与输入相关的执行成本**：LLM 推理是自回归的，输出长度高度可变且难以预测。在多智能体工作流中，每个阶段的延迟和内存占用取决于其输出长度，而这是未知的。没有准确的成本估计，调度器无法有效进行优先级排序或内存分配，导致**队头阻塞（HoL blocking）**，即长推理步骤阻塞了短步骤。这种成本还会因智能体的角色、工具调用和推理策略（如 CoT）而异。
2.  **长尾使用模式与内存争用**：多智能体流水线中，模型调用呈现长尾分布：少数模型被频繁调用，多数模型很少使用。为每个模型分配专用 GPU 会导致资源利用率低下，而按需加载模型会引入数十秒的冷启动延迟，违反交互式延迟 SLO。将多个模型共置于一个 GPU 上可以缓解利用率问题，但会加剧内存争用。除了静态模型参数，KV 缓存的大小随输出长度动态增长，给有限的 GPU 内存带来巨大压力。
3.  **跨集群路由的权衡（延迟 vs. 资源就绪度）**：在分布式环境中，不同 GPU 集群的网络延迟和资源可用性各异。路由到拓扑上最近的集群可以最小化网络延迟，但该集群可能缺少缓存的模型权重或足够的 GPU 内存来立即服务请求。而一个更远的集群可能具有高资源就绪度（模型已加载、内存充足），但更高的往返时间（RTT）会增加端到端延迟，特别是首 token 时间（TTFT）。低效的路由会延迟整个流水线。

现有解决方案（如 FASTSERVE、MuxServe、SkyServe）仅能部分解决这些问题，它们要么针对单轮查询，要么采用静态路由，没有一个系统能将低开销的阶段级成本估计与面向智能体工作流的跨集群分配结合起来。因此，需要一个全新的、统一的工作负载感知调度系统。

## 2. 创新点（Innovation）

Maestro 的核心创新在于其**智能体感知（Agent-Aware）** 的设计哲学，它利用智能体的语义和角色信息来驱动调度决策，具体体现在以下四点：

1.  **智能体感知的成本预测（Agent-Aware Cost Prediction）**：这是 Maestro 的基石。不同于将 LLM 请求视为黑盒，Maestro 显式地利用智能体的**角色**、**工作流位置**和**工具调用意图**。它设计了一个两阶段预测器：首先通过一个轻量级分类器预测当前阶段是否会调用工具（工具调用通常产生短输出），然后结合结构化特征和语义特征（通过 MiniLM 提取）来预测输出长度。这显著降低了对输出长度和 KV 缓存需求的预测误差。
2.  **多智能体时空复用（Multi-Agent Spatio-Temporal Multiplexing）**：在节点层面，Maestro 提出了一种结合**层次化权重驻留**（GPU/CPU/磁盘）和**预测驱动的弹性 KV 缓存分配**的内存管理方案。通过 LRU 策略在 GPU、主机内存和磁盘之间动态迁移模型权重，并利用 CUDA 虚拟内存管理（VMM）实现 KV 缓存的按需分配和回收，从而在严格的 GPU 内存预算下实现多个专用模型的高效共置。
3.  **可行性感知的跨集群调度（Feasibility-Aware Cross-Cluster Scheduling）**：Maestro 设计了一个统一的**适应度分数（Fitness Score）**，将网络延迟、每个集群的模型就绪度和预测的 KV 缓存需求整合在一起，用于驱动跨集群的任务分发。这解决了传统路由策略中“就近”与“就绪”的矛盾，能够智能地选择能最快开始执行任务的集群。
4.  **工作流感知的优先级队列（Workflow-Aware Prioritization）**：在全局层面，Maestro 采用**边界抢占式最短剩余时间优先（Boundary-Preemptive Shortest-Remaining-Time-First, SRTF）** 队列。它根据预测的每个阶段执行时间和工作流模板的历史数据，估算整个作业的剩余完成时间，并据此进行调度。这有效缓解了依赖关系引起的队头阻塞，保护了延迟敏感型交互式任务，同时避免了长分析型作业的饥饿。

## 3. 主要内容（Main Content）【重点】

### 3.1 设计架构与方法

Maestro 采用一个阶段驱动的闭环控制流水线，其系统架构如图 5 所示。每个工作流阶段 `T` 作为独立的调度单元，经历以下五个阶段：

1.  **智能体上下文观察（Agent-Context Observation）**：当一个阶段被创建时，Maestro 拦截调用并提取一个紧凑的描述符，包含智能体角色、工作流位置、工具可用性以及输入上下文的语义嵌入。
2.  **成本预测（Cost Prediction）**：一个轻量级预测器在调度网关处估计该阶段的预期输出长度 $\hat{L}(T)$、KV 缓存需求 $\hat{R}_{kv}(T)$ 和工具调用概率 $\hat{p}_{tool}(T)$。
3.  **调度决策（Scheduling Decision）**：全局调度器首先根据策略和内存约束过滤不可行的节点，然后根据一个平衡了网络延迟、启动时间和 KV 缓存可行性的适应度分数，将阶段分派到最优节点。阶段按工作流感知的 SRTF 策略排序，并允许对延迟关键型任务进行抢占。
4.  **节点级执行与内存管理（Node-Level Execution and Memory Management）**：运行时管理器协调模型加载、KV 缓存分配，并在内存压力下执行最小影响的内存回收，同时将中断成本报告给调度器。
5.  **执行后分析（Post-Execution Profiling）**：记录实际的输出长度、延迟和内存使用情况，用于增量校准预测器并重新调整剩余阶段的优先级。

### 3.2 关键算法与数学推导

**1. 智能体感知成本预测：**

- **工具调用意图分类**：使用 LightGBM 分类器预测阶段 `T` 触发工具调用的概率：
  $$\hat{p}_{tool}(T) = \Pr[tool(T) = 1 | x(T)]$$
  其中 $x(T)$ 是输入特征（结构化+语义）。之后使用保序回归（Isotonic Regression）校准分类器输出。

- **输出长度预测**：使用回归模型预测输出长度 $\hat{L}(T)$。为了处理重尾分布，对 $\log(1 + L(T))$ 进行回归，推理时再逆变换。

- **执行时间估计**：将预测长度转换为执行时间估计：
  $$\hat{T}_{exec}(T) = t_{pre}(P(T), M(T)) + t_{dec}(M(T)) \cdot \hat{L}(T)$$
  其中 $t_{pre}$ 是预填充延迟，$t_{dec}$ 是每 token 平均解码延迟，$P(T)$ 是提示长度，$M(T)$ 是模型。

- **KV 缓存需求估计**：
  $$\hat{R}_{kv}(T) = \alpha(M(T)) \cdot \left( P(T) + \hat{L}(T) \right)$$
  其中 $\alpha(M)$ 是模型每 token 的内存占用。

**2. 节点级内存管理：**

- **层次化权重驻留与驱逐（算法 1）**：当需要加载模型 `m` 时，从最快可用层级（GPU > CPU > 磁盘）激活。如果 GPU 内存不足，使用 LRU 策略将最近最少使用的模型卸载到主机内存。如果主机内存也不足，则进一步驱逐到磁盘。

- **最小影响内存协调（算法 2）**：当 KV 准入失败时，运行时执行此算法以生成一个最小影响的降级计划。它定义了五个降级级别（从空闲模型休眠到中止活跃执行），并采用贪心策略，遍历驻留引擎以释放所需内存 $R$。总中断成本 $C_{deg}(N, T)$ 定义为执行该计划所需的**聚合恢复延迟**：
  $$C_{deg}(N, T) = \sum_{(e,a) \in P} c(e, a) + \mathbb{1}[I_{active}] \cdot c_{int}$$
  其中 $c(e, a)$ 是模型重载或 KV 重算的成本，$c_{int}$ 是 SLO 违规阈值。

**3. 工作负载感知的跨集群调度（算法 3）：**

- **跨集群适应度分数**：对于到达的阶段 `T`，调度器首先过滤候选节点，然后选择最小化预期阶段完成延迟的节点。由于 $\hat{T}_{exec}(T)$ 在节点间不变，排名基于紧凑分数：
  $$S(N, T) = A(N, T) - \lambda T_{ready}(N, T) - \mu C_{deg}(N, T)$$
  其中：
  - $A(N, T)$ 是亲和力项，结合了网络邻近度（RTT 的递减变换）和 KV 缓存适配度（最佳适配打包）。
  - $T_{ready}(N, T)$ 是预期启动时间，分解为节点本地排队延迟 $T_q(N, T)$ 和模型激活延迟 $T_{act}(N, M(T))$：
    $$T_{ready}(N, T) = T_q(N, T) + T_{act}(N, M(T))$$
  - $C_{deg}(N, T)$ 是运行时报告的中断惩罚。

- **剩余时间队列**：全局队列根据作业的估计剩余工作流执行时间 $\hat{T}_{rem}(J, k)$ 进行排序：
  $$\hat{T}_{rem}(J, k) = \hat{T}_{exec}(T_k) + \hat{T}_{future}(J, k)$$
  其中 $\hat{T}_{future}(J, k)$ 使用工作流模板的条件中位数历史执行时间来估计。

### 3.3 模型、训练与数据集

- **模型**：预测器使用 LightGBM（用于分类和回归）和 MiniLM（用于语义嵌入）。推理后端使用 vLLM。
- **训练**：预测器在记录的阶段级轨迹上进行训练。训练/测试集采用分层时间保留法：在每个智能体、工具使用和思维模式组内，最新的 20% 记录作为测试集，较早的记录用于训练。
- **数据集**：工作负载轨迹来自 9 个代表性的 LLM-MAS 应用，包含 46,769 个作业和 144,524 个 LLM 调用阶段（见表 I）。这些应用覆盖了串行、并行、循环和监督-工作者等模式。数据来源包括合成数据和公开数据集（如 MATH、GPQA、SQuAD、IBM HR Analytics 等）。

### 3.4 实验设置与结果

- **测试平台**：32 台 GPU 服务器（每台 2 块 NVIDIA A100 GPU），使用 Kubernetes 和 vLLM。
- **基线**：
    - **预测**：与 Linear、BERT-MLP、Magnus 比较。
    - **调度**：与 FCFS、EDF、Oracle-SRTF 比较。
    - **多模型服务**：与 No-Colocation、QLM-style Switching 比较。
- **主要结果**：
    1.  **整体性能（图 7）**：在高负载和高批处理比率下，Maestro 的 SLO 达标率比 EDF 提高了 **23.6 个百分点**（73.6% vs 50.0%），并将交互式排队延迟降低了 **84.8%**。
    2.  **资源效率（表 IV, V, 图 8）**：
        - 在 Travel Assistant 工作流中，使用 2 个 GPU 时，Maestro 比 QLM 快 **38.9%**；使用 1 个 GPU 时，快 **70.0%**。
        - 在 5 个模型共置于一个 40GB GPU 的场景下，Maestro 实现了 **3.05 倍**的有效内存占用（205% 的过度承诺），即 KV 预留 HBM 减少了 **67.2%**。
    3.  **组件分析（表 III, VI, VII, VIII）**：
        - **预测精度**：Maestro-Pred 的 MAE 为 165.43 tokens，比 Magnus 低 **19.2%**，$R^2$ 达到 0.7774。
        - **工具意图分类**：Maestro-Pred 的 AUC 最高（0.9625），MSE 和 LogLoss 最低。
        - **消融实验**：移除 BERT 语义特征或工具意图分类器都会导致预测精度下降。
        - **跨集群调度**：Maestro 的完整适应度分数（Maestro-Aff）在不同负载下均实现了最低的交互式排队延迟。

### 3.5 技术细节

- **预测开销**：Maestro 的预测模块中位延迟仅为 **11.24ms**（图 9），相对于网络 RTT 和 LLM 推理时间可以忽略不计。
- **模型激活延迟**：Maestro 通过复用运行时上下文和避免进程重启，将模型激活延迟从 QLM 的数十秒降低到 **0.11 到 2.50 秒**（图 10）。
- **抢占机制**：Maestro 仅在阶段边界（LLM 调用之间）进行抢占，避免中断正在进行的解码。它使用迟滞（hysteresis）来防止因估计噪声导致的振荡，并通过老化（aging）机制防止长等待的批处理作业发生饥饿。
- **内存安全**：通过显式内存记账和准入时的可行性检查，确保 $M_{kv} + M_{res} \leq M_{total}$，防止 OOM 错误。

## 4. 意义与影响（Significance and Impact）

- **解决的问题与贡献**：Maestro 解决了在资源受限的云环境中大规模部署 LLM 多智能体系统所面临的独特系统挑战。其核心贡献在于证明了**利用智能体语义信息进行工作负载预测**可以显著提升调度效率和资源利用率。它提供了一个完整的、可工作的系统原型，并通过详尽的实验验证了其有效性。
- **对未来研究/应用的影响**：
    - **系统设计范式**：Maestro 的工作表明，未来的 LLM 服务系统不应再将请求视为独立、同质的黑盒。相反，应该深入理解应用层语义（如智能体角色、工作流结构）来指导底层资源管理。这为构建更智能、更高效的 AI 基础设施指明了方向。
    - **预测驱动的调度**：Maestro 的成功验证了在 LLM 服务中使用细粒度、在线预测的可行性。这可能会激发更多关于 LLM 行为预测（如输出长度、推理时间）的研究，并将其应用于更广泛的场景，如自动扩缩容、成本优化等。
    - **多模型共置**：Maestro 提出的层次化权重驻留和弹性 KV 缓存方案，为在单个 GPU 上高效服务多个模型提供了实用范例，这对于降低私有化部署成本至关重要。
- **局限与开放问题**：
    - **抢占粒度**：作者提到，当前仅在阶段边界进行抢占是保守的。更细粒度的 token 级或迭代级抢占可以进一步减少长解码阶段的阻塞，但这需要与推理引擎进行更紧密的集成。
    - **外部工具延迟**：当前运行时不为外部工具调用预留 CPU 或网络资源。未来工作可以探索将工具执行延迟纳入调度模型。
    - **预测器泛化性**：预测器需要针对特定工作流进行训练。如何设计一个能快速适应新任务或新智能体角色的零样本或少样本预测器是一个开放问题。
    - **KV 缓存迁移**：在抢占时，KV 缓存可能无法保留。探索高效的 KV 缓存跨节点迁移技术可以进一步降低恢复延迟。

## 5. 澄清与简化（Clarifications and Simplifications）

- **KV 缓存（KV Cache）**：可以把它想象成一个“**思维草稿本**”。当 LLM 生成文本时，它会不断地回顾之前生成的词（token）来确保上下文连贯。KV 缓存就是用来存储这些“已读过的词”的关键信息，这样模型就不需要每次生成新
