## 0. 概览（Concise Summary）

- **主题（Topic）**：UI-to-Code 生成，即从用户界面截图自动生成可执行的前端代码（HTML/CSS）。
- **问题（Problem）**：现有方法将 UI-to-Code 视为单次生成任务，这与现实世界中 UI 开发固有的迭代、反馈驱动的流程严重不符。单次生成无法处理渲染环境差异（如字体、DPI）导致的视觉偏差，且缺乏自我修正机制。
- **方法（Method）**：将 UI-to-Code 重新定义为**交互式视觉优化**问题，并提出了**相对视觉策略优化（RVPO）**，一种基于偏好的强化学习方法。该方法通过比较多个渲染候选代码的视觉质量来构建稳定的奖励信号，驱动模型进行迭代优化。
- **创新（Innovation）**：
    1.  首次将 UI-to-Code 形式化为一个闭环的交互式视觉优化过程，而非单次生成。
    2.  提出 RVPO 算法，通过相对排序而非绝对评分来优化非可微的视觉目标，解决了奖励信号不稳定和噪声大的问题。
    3.  构建了 UI2Code^N 模型，一个通过持续预训练、监督微调和 RVPO 强化学习训练的 9B 开源模型，在多项基准上达到 SOTA。
- **意义（Significance）**：该工作更贴近真实的 UI 开发流程，显著提升了生成代码的视觉保真度和实用性。其提出的交互式优化范式和 RVPO 算法具有通用性，可应用于其他具有黑盒反馈的可执行代码生成问题。
- **一句话总结**：本文提出了一种将 UI 截图生成代码视为交互式视觉优化过程的新范式，并通过相对视觉策略优化（RVPO）算法，训练出性能卓越的 9B 模型 UI2Code^N。

## 1. 研究动机（Motivation）

- **解决的问题**：现有 UI-to-Code 方法大多被建模为**单次生成**问题，即模型根据一张截图一次性生成代码。然而，现实中的前端开发是一个**迭代、反馈驱动**的过程：开发者先写出草稿代码，渲染后检查视觉差异，然后根据反馈进行修改，直到满意为止。
- **为什么重要**：单次生成方法存在根本性缺陷：
    1.  **渲染不确定性**：代码的最终渲染效果受浏览器默认样式、字体回退、DPI 缩放等运行时因素影响，无法仅从代码静态推断。
    2.  **视觉差异的复杂性**：间距、对齐、排版等视觉问题相互耦合，单次预测难以完美解决。
    3.  **缺乏自我验证**：单次生成无法利用渲染后的视觉反馈进行自我修正。
- **挑战**：优化目标（视觉保真度）是在渲染后的视觉空间中定义的，相对于生成的代码 token 是**非可微**的。此外，使用 VLM 作为评判器给出的绝对视觉分数往往噪声大、校准差，不适合直接作为强化学习的奖励信号。

## 2. 创新点（Innovation）

1.  **新范式：交互式视觉优化（Interactive Visual Optimization）**：将 UI-to-Code 从单次生成问题重新定义为闭环的交互式优化问题。模型生成代码 -> 渲染 -> 视觉检查 -> 根据反馈迭代优化。这个范式统一了 UI 草稿（Drafting）、UI 打磨（Polishing）和 UI 编辑（Editing）三个任务。
2.  **新算法：相对视觉策略优化（RVPO）**：针对非可微视觉目标和绝对评分不稳定的问题，提出了一种基于偏好的强化学习方法。RVPO 不依赖绝对分数，而是通过**锦标赛式**的成对比较，在多个渲染候选代码中建立**相对视觉偏好排序**，并以此构建更稳定、更有效的奖励信号。
3.  **新模型：UI2Code^N**：一个 9B 参数的开源 VLM，通过三阶段训练（持续预训练、监督微调、RVPO 强化学习）实现。尽管参数量小，但在 UI 草稿、打磨和编辑任务上均超越了包括 GPT-5、Gemini-2.5-Pro 在内的更大规模闭源模型。

## 3. 主要内容（Main Content）【重点】

### 3.1 设计架构与方法

论文的核心架构是一个**交互式视觉优化循环**，如图 1 和图 2 所示。

- **UI 草稿（UI Drafting）**：模型根据目标 UI 截图 $I$ 生成初始代码 $C^{(0)}$。这是优化的起点。
- **UI 打磨（UI Polishing）**：这是一个迭代过程。在第 $t$ 步，模型将当前代码 $C^{(t)}$ 渲染成图像 $R^{(t)}$，然后结合目标图像 $I$、当前代码 $C^{(t)}$ 和渲染结果 $R^{(t)}$，生成改进后的代码 $C^{(t+1)}$。这个过程可以重复 $N$ 次，即 UI2Code^N。
- **UI 编辑（UI Editing）**：这是 UI 打磨的一个变体，除了目标图像 $I$ 和当前代码 $C$，还接受一个修改指令 $E$，生成满足指令的代码 $C'$。

### 3.2 关键算法与数学推导

**1. 交互式视觉优化形式化：**

优化目标是找到代码 $C^*$，使其渲染后的输出与目标图像 $I$ 的视觉差异最小化：
$$C^* = \arg \min_C \mathcal{D}(I, \text{Render}(C))$$
其中 $\text{Render}(\cdot)$ 是一个黑盒、非可微的渲染过程，$\mathcal{D}$ 是隐式的视觉距离。

**2. 相对视觉策略优化（RVPO）：**

RVPO 的核心思想是使用相对偏好而非绝对分数来指导策略优化。

- **偏好概率**：假设有一个视觉评判器 $C_\psi$，对于两个候选代码 $(y, y')$，它能给出一个偏好概率：
  $$p_\psi(y \succ y' | x) := P[C_\psi(x, y, y') = 1]$$

- **基于排名的替代目标**：RVPO 优化以下目标函数，即一个样本相对于其他策略样本的期望偏好：
  $$\mathcal{L}_{\text{rank}}(\theta) = \mathbb{E}_{y \sim \pi_\theta(\cdot|x)} \left[ \mathbb{E}_{y' \sim \pi_\theta(\cdot|x)} \left[ p_\psi(y \succ y' | x) \right] \right]$$

- **锦标赛式奖励构建**：为了近似上述目标，算法从策略 $\pi_\theta$ 中采样 $N$ 个候选 $\{y_i\}_{i=1}^N$。然后，在组内对所有有序对 $(i, j)$ 进行成对比较，得到二进制结果 $o_{ij}$。每个候选 $y_i$ 的奖励 $W_i$ 是其获胜次数：
  $$W_i := \sum_{j \neq i} o_{ij}$$
  这个 $W_i$ 经过归一化等处理后作为最终的标量奖励。这种方法对单个比较的噪声不敏感，且不依赖绝对分数校准。具体流程见算法 1。

- **策略优化（GRPO）**：RVPO 使用**组相对策略优化（GRPO）** 来更新策略。给定组内奖励 $\{r_i\}_{i=1}^N$，GRPO 计算组归一化的优势函数：
  $$A_i = \frac{r_i - \bar{r}}{\sigma_r}$$
  然后使用裁剪的代理目标函数进行更新：
  $$J(\theta) = \mathbb{E} \left[ \frac{1}{N} \sum_{i=1}^N \min(\rho_i A_i, \text{clip}(\rho_i, 1-\epsilon, 1+\epsilon) A_i) \right]$$
  其中 $\rho_i = \pi_\theta(y_i|x) / \pi_{\theta_{\text{old}}}(y_i|x)$。

### 3.3 模型、训练与数据集

- **模型**：UI2Code^N 是一个 9B 参数的视觉语言模型（VLM）。
- **三阶段训练流程**：
    1.  **持续预训练（Continual Pre-training）**：在大量 UI 截图和对应 DOM 结构/代码对上进行训练，建立视觉-代码的底层对齐。使用了 GUI-REG 风格的目标函数，强化局部区域（如 bounding box）与代码片段的关联。
    2.  **监督微调（Supervised Fine-tuning, SFT）**：使用包含 UI 草稿、打磨和编辑指令的数据进行微调。模型输出被结构化为 `<think>诊断文本</think><answer>代码</answer>` 的格式，以鼓励模型进行显式的视觉差异推理。
    3.  **强化学习（Reinforcement Learning, RL）**：使用 RVPO 算法进行强化学习，直接优化渲染后的视觉保真度。所有 RL 数据和奖励信号与评估基准不重叠。
- **数据集**：训练数据来自公开可访问的网页或现有数据集。评估基准包括 Design2Code、Flame-React-Eval、Web2Code，以及作者构建的更复杂的 **UI2Code-Real** 和用于 UI 打磨的 **UIPolish-bench**。

### 3.4 实验设置与结果

- **评估指标**：主要采用 **VLM 评分**（使用 VLM 作为评判器，评估视觉保真度），因为作者证明 VLM 评分比 CLIP 评分更符合人类偏好，且对结构差异更敏感。对于 UI 打磨，采用成对评估协议，计算打磨成功率。
- **主要结果（表 1）**：
    - **UI 草稿**：UI2Code^N-9B-RL 在所有基准（Design2Code, Flame, Web2Code, UI2Code-Real）上均取得了开源模型中的最佳成绩，并超越了大部分闭源模型，尤其是在复杂的 UI2Code-Real 上优势明显。
    - **UI 打磨**：UI2Code^N-9B-RL 在 UIPolish-Synthetic 和 UIPolish-Real 上分别达到 94.0% 和 80.0% 的准确率，远超其他开源模型，甚至超过了 GPT-5 和 Claude-4-5-Sonnet。
- **测试时扩展（表 3）**：随着打磨迭代次数 $N$ 的增加，UI2Code^N 的性能持续提升，验证了交互式优化范式的有效性。
- **消融实验**：
    - **奖励设计（图 3）**：RVPO（基于相对偏好）显著优于 SFT 和基于绝对评分的 RL 基线。
    - **训练阶段（表 6）**：持续预训练（+44.3%）、SFT（+26.0%）和 RL（+9.3%）三个阶段对最终性能都有显著贡献。
    - **真实数据影响（图 4）**：在 RL 阶段加入真实网页数据能显著提升模型在真实场景下的性能。
- **人类评估（图 5，表 7）**：在 Design2Code-HARD 和 UIEdit-Bench 上，UI2Code^N 在人类评估中胜出或与最强闭源模型持平，证明了其生成代码的实用性和高质量。

### 3.5 技术细节

- **RVPO 效率**：成对视觉比较并非计算瓶颈，仅占 RVPO 训练迭代总时间的 2.2%，主要开销来自自回归生成和渲染。
- **奖励黑客分析（表 5）**：RL 训练后，代码中绝对定位元素的比例反而从 0.7% 下降到 0.5%，表明模型没有通过走捷径（如滥用绝对定位）来欺骗奖励，而是真正提升了视觉质量。
- **评判器验证**：VLM 评判器与人类判断在模型排名上高度一致（Spearman 1.0），且行为保守，没有偏向作者模型。

## 4. 意义与影响（Significance and Impact）

- **解决的问题与贡献**：
    1.  解决了 UI-to-Code 领域“单次生成”与“迭代开发”现实之间的根本性错配问题。
    2.  提出了一个通用的、可迁移的“交互式视觉优化”范式，为其他需要黑盒反馈的代码生成任务（如图表生成、游戏关卡设计）提供了新思路。
    3.  提出的 RVPO 算法为在非可微空间中利用相对偏好进行策略优化提供了一种稳定有效的方法，解决了绝对奖励信号不稳定的痛点。
    4.  开源了性能强大的 9B 模型 UI2Code^N，为社区提供了一个强大的基线，降低了该领域的研究门槛。
- **对未来研究/应用的影响**：
    - 可能推动 UI 开发工具向更智能、更自动化的方向发展，减少重复劳动。
    - 其“思考-行动-观察”的闭环优化思想可以启发其他 AI 辅助编程工具的设计。
    - RVPO 算法有望被应用于更广泛的生成任务，如文生图、文生视频等，其中优化目标也是非可微的视觉质量。
- **局限与开放问题**：
    - 论文未明确提及的局限可能包括：对非常复杂或高度动态的 UI（如包含大量 JavaScript 交互）的处理能力有限。
    - 开放问题：如何更高效地进行多轮迭代？如何设计更细粒度的视觉评判器？如何将交互式优化扩展到更通用的软件工程任务？

## 5. 澄清与简化（Clarifications and Simplifications）

- **交互式视觉优化 vs. 单次生成**：可以类比为“写作文”和“做雕塑”。单次生成就像一次写一篇完美的作文，非常困难。交互式优化则像做雕塑，先搭一个大致的骨架（草稿），然后不断观察、打磨、修正细节，直到满意为止。UI2Code^N 就是学会了“做雕塑”的过程。
- **RVPO 为什么比绝对评分好**：想象一下，让一个老师给学生的作文打分。如果老师今天心情好，给分宽松；明天心情差，给分严格，那么绝对分数就不可靠。RVPO 的做法是，让老师比较两篇作文哪篇更好。即使老师标准有波动，但“A 比 B 好”这个相对判断通常比“A 得 85 分”这个绝对分数要稳定得多。RVPO 通过锦标赛式的两两比较，构建了这种更稳定的“相对偏好”信号。
- **锦标赛式奖励**：就像体育比赛中的小组赛。每个候选代码都是“选手”，它们之间两两比赛（视觉比较），赢的次数最多的选手获得最高奖励。这比只看一个“评委”给的“印象分”要公平和稳定得多。
